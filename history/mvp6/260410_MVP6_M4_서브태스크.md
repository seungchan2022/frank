요약/인사이트를 마크다운 형식으로 출력하고, 클라이언트에서 렌더링한다.

> M4 서버(`summarize.rs` + `exa.rs` + `openrouter.rs`)는 M1~M3와 독립적 — 서버 작업과 클라이언트 작업 병렬 진행 가능

## 서브태스크 목록

### S1: 서버 — exa highlights API 전환 + 스니펫 정제

**목적**: exa `text: true` 응답(HTML 혼입, 무제한 길이)을 highlights API로 교체해 snippet 품질 개선

**작업**:
- `server/src/infra/exa.rs` 수정
  - 요청 body에서 `text: true` 제거
  - `highlights: { numSentences: 3, highlightsPerUrl: 1 }` 추가
  - 응답 파싱: `result.highlights[0]` (없으면 빈 문자열)
  - 후처리 파이프라인 추가:
    1. `#` `##` `###` 로 시작하는 마크다운 헤더 행 제거
    2. 연속 공백/줄바꿈 → 단일 공백으로 collapse
    3. 300자 초과 시 truncate (단어 경계 무시, 단순 슬라이싱)

**기술 참고**:
- exa `highlights` API는 `text` 없이도 동작 — 두 필드 동시 사용 불필요
- 300자 제한: FeedItem.snippet은 카드 UI 미리보기 용도 → 짧을수록 좋음

**산출물**: 컴파일 + 기존 서버 테스트 유지

### S2: 서버 — openrouter 마크다운 프롬프트 + fake_llm 갱신

**목적**: LLM이 마크다운 형식(bold, italic, 목록)으로 요약/인사이트를 출력하도록 시스템 프롬프트 수정

**작업**:
- `server/src/infra/openrouter.rs` 수정
  - `SYSTEM_PROMPT` 상수에 마크다운 출력 지시 추가:
    - `**굵게**` — 핵심 키워드 강조
    - `*이탤릭*` — 용어, 개념 강조
    - `- 항목` — 나열 시 목록 사용
    - 문단 사이 빈 줄
    - 금지: 코드블록, 테이블, 헤딩(`#`)
  - `summary` 응답 길이 가이드: 3~4문장 또는 4~6개 목록 항목
  - `insight` 응답 길이 가이드: 2~3문장
- `server/src/infra/fake_llm.rs` 수정
  - `FakeLlmAdapter` 고정 응답에 마크다운 포함 (bold, italic, 목록)
  - 서버 테스트에서 마크다운 응답을 반환해야 실사용 검증 가능

**산출물**: 컴파일 + 서버 테스트 통과

### S3: 웹 — marked + typography 마크다운 렌더링

**목적**: 디테일 페이지의 summary/insight/snippet을 마크다운으로 렌더링

**기술 선택 경위**:
- `svelte-markdown` → Svelte 5 runes와 호환 불량 (제외)
- `marked` + `{@html}` → Svelte 5 호환, XSS 우려는 서버 응답 신뢰 범위 내 허용
- `@tailwindcss/typography` prose 클래스 → HTML 렌더링 시 기본 타이포그래피 제공

**작업**:
- `web/package.json`에 의존성 추가: `marked`, `@tailwindcss/typography`
- `web/tailwind.config.js` (또는 해당 설정)에 typography 플러그인 등록
- `web/src/routes/feed/article/+page.svelte` 수정
  - `import { marked } from 'marked';` 추가
  - summary/insight/snippet 표시 부분: `<p>{text}</p>` → `<div class="prose prose-sm">{@html marked(text ?? '')}</div>`
  - null/undefined 방어: `text ?? ''`

**기술 참고**:
- `{@html}` 사용 시 XSS 위험: 서버 LLM 응답이 신뢰 경계 내이므로 허용
- `prose-sm`: 디테일 페이지 폰트 크기에 맞는 타이포그래피 스케일

**산출물**: 웹 린트 + 타입체크 + 테스트 통과 (162개)

---

### S4: iOS — AttributedString 마크다운 렌더링

**목적**: 디테일 뷰의 summary/insight/snippet을 iOS 네이티브 마크다운으로 렌더링

**기술 선택 경위**:
- `MarkdownUI` (swift-cmark) → Tuist 빌드 이슈로 제외
- `AttributedString(markdown:)` → iOS 15+ 네이티브, 서드파티 불필요

**작업**:
- `ios/Frank/Frank/Sources/Features/Detail/ArticleDetailView.swift` 수정
  - `Text(item.summary ?? "")` →
    ```swift
    if let text = item.summary {
        Text(try! AttributedString(markdown: text))
    }
    ```
    실제 구현: `try?` fallback으로 파싱 실패 시 일반 `Text(text)` 표시
  - summary / insight / snippet 3곳 모두 동일 패턴 적용

**기술 참고**:
- `AttributedString(markdown:)` — iOS 15+ API, 타깃 iOS 26.0이므로 사용 가능
- `try?` fallback: 마크다운 파싱 실패(잘못된 형식) 시 일반 텍스트로 graceful degradation
- 지원 마크다운: bold, italic, 목록, 링크 — 코드블록/테이블은 지원 수준 낮으나 LLM 출력에 미포함이므로 무관

**산출물**: 빌드 통과 + iOS 테스트 유지

## 추가 버그픽스 (M4와 함께 진행)

### BF-1: 웹 — 토큰 만료 시 자동 갱신 + 재시도

**문제**: Supabase access_token은 1시간 후 만료. `currentAccessToken`은 `+layout.svelte` 초기 로드 시 1회만 주입 → 장기 세션에서 모든 API 호출이 401로 실패.

**원인**: httpOnly 쿠키 구조상 브라우저 JS에서 Supabase SDK의 `getSession()` 직접 호출 불가.

**해결**:
- `web/src/routes/api/auth/token/+server.ts` 신규 생성
  - `GET /api/auth/token` — SvelteKit 서버에서 `safeGetSession()`으로 새 access_token 반환
  - httpOnly 쿠키 접근 가능, Supabase SDK가 내부적으로 refresh 처리
- `web/src/lib/api/realClient.ts` 수정
  - `tryRefreshToken()`: `/api/auth/token` 호출 → `currentAccessToken` 갱신 → `true` 반환
  - `request()` 함수: 401 수신 시 `tryRefreshToken()` → 성공하면 원 요청 1회 재시도

**테스트**: `realClient.test.ts`에 토큰 갱신 시나리오 4종 추가 (162개)

### BF-2: iOS — checkSession 에러 전파 + 한국어 에러 메시지

**문제**: `AuthFeature.checkSession()` catch 블록에서 `self.error` 미할당 → 에러 발생 시 UI에 아무 메시지 미표시.

**해결**:
- `AuthFeature.swift`: catch 블록에 `self.error = error.localizedDescription` 추가
- `SupabaseAuthAdapter.swift`, `EmailSignInSheet.swift`, `LoginView.swift`: 에러 메시지 한국어 정렬

---

### BF-3: deploy.sh — iOS xcconfig 자동 생성 + DB 에러 경고

**문제**:
1. `ios/Frank/Config.xcconfig` git-ignore 처리됨 → 클론 후 iOS 빌드 시 매번 `host 없음` 에러
2. Supabase 대시보드로 생성한 계정은 `confirmation_token` 등 NULL → GoTrue `Database error querying schema` 에러

**해결**:
- `scripts/deploy.sh`에 `generate_ios_xcconfig()` 함수 추가
  - `server/.env`에서 `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SERVER_URL` 읽어 자동 생성
  - `$(SLASH)` 트릭: `printf`로 리터럴 `$(SLASH)$(SLASH)` 출력 → xcconfig에서 `://` 파싱 정상
- `warn_supabase_dashboard_accounts()` 함수 추가 — 대시보드 계정 문제 설명 + 수정 SQL 출력
