## 메인태스크 요약

태그 수에 관계없이 피드 응답시간을 일정하게 유지하고, 갱신 중에도 빈 화면 없이 이전 피드를 표시한다.

- **서버**: 태그별 `search()` 순차 호출 → `futures::join_all()` 병렬 전환

- **iOS**: `feedItems` 유지한 채 갱신 + refresh 완료 후 scroll to top

### S2: 서버 — join_all 병렬 검색 구현

**목적**: `for` 루프 순차 검색 → `futures::future::join_all` 병렬 전환

**작업**:
- `server/src/api/feed.rs`의 `get_feed()` 핸들러 수정
- 태그 루프 전에 `SearchJob { tag_id: Uuid, tag_name: String, query: String }` 소유 구조체 Vec 생성
- `futures::future::join_all`로 모든 태그 검색을 동시에 실행:
  ```rust
  let futures = jobs.into_iter().map(|job| {
      let chain = Arc::clone(&state.search_chain);
      async move {
          let result = chain.search(&job.query, 5).await;
          (job, result)
      }
  });
  let results = futures::future::join_all(futures).await;
  ```
- 실패한 태그는 `tracing::warn!` 후 skip (기존 동작 유지)
- `cargo clippy -- -D warnings`, `cargo fmt --check`, `cargo test` 통과 확인

**기술 참고**:
- `futures` crate: Cargo.toml에 이미 `futures = "0.3"` 있음
- lifetime 해결 핵심: `query: String` owned 필드로 futures 벡터에 캡처 — `&str` 직접 캡처 금지
- `state.search_chain`은 `Arc<dyn SearchChainPort>` — 각 future에서 `Arc::clone` 후 사용

**산출물**: `cargo test` 통과 (서버 기존 178개 + S0 확장 + 신규 2개 이상)

### S5: 웹 — Progress Bar UI + 스크롤 상단 이동

**목적**: refresh 중 카드 위 얇은 progress bar 표시, 새 결과 도착 시 스크롤 상단 이동

**작업**:
- `web/src/routes/feed/+page.svelte` 수정
  - `feedStore.isRefreshing` 상태를 사용해 카드 목록 위 progress bar 표시
  - progress bar: `<div class="h-1 bg-blue-500 animate-pulse w-full">` (Tailwind)
  - 새 피드 로드 완료 감지: 이전 값 추적 방식 (`prevRefreshing` 로컬 변수) — `isRefreshing true → false` 전환 + 에러 없음 + 아이템 존재 시만 스크롤
    ```svelte
    import { tick } from 'svelte';
    import { browser } from '$app/environment';
    
    let prevRefreshing = false;
    $effect(() => {
      if (prevRefreshing && !feedStore.isRefreshing && !feedStore.error && feedStore.feedItems.length > 0) {
        tick().then(() => {
          if (browser) window.scrollTo({ top: 0, behavior: 'smooth' });
        });
      }
      prevRefreshing = feedStore.isRefreshing;
    });
    ```
  - 버튼 disabled: `loading || isRefreshing` 기준으로 교체 (초기 로딩 중 버튼 비활성화 유지)
  - 기존 "갱신 중..." 텍스트/스피너는 `isRefreshing`으로 교체

**산출물**: 웹 피드 페이지 progress bar 표시, 갱신 성공 완료 후 자동 상단 스크롤

### S7: iOS — FeedView scroll to top

**목적**: refresh 완료 후 피드 목록 첫 항목으로 자동 스크롤

**작업**:
- `ios/Frank/Frank/Sources/Features/Feed/FeedView.swift` 수정
  - `scrollPosition(id:)` modifier 사용 (선언적, SwiftUI 철학 부합) — iOS 26.0 타깃이므로 제한 없음
  - `@State private var scrolledID: String?` 추가
  - `.scrollPosition(id: $scrolledID)` modifier 적용
  - `onChange(of: feature.phase)`에서 `.refreshing → .idle` 전환 + 성공 조건 확인 후 스크롤:
    ```swift
    .onChange(of: feature.phase) { oldPhase, newPhase in
        if oldPhase == .refreshing && newPhase == .idle {
            guard feature.errorMessage == nil,
                  let firstID = feature.articles.first?.id else { return }
            scrolledID = firstID
        }
    }
    ```
  - 실패 refresh(errorMessage 있음) 시 스크롤 미발동

**기술 참고**:
- `scrollPosition(id:)` — iOS 17+ API, 배포 타깃 iOS 26.0이므로 사용 가능
- 실패 refresh도 `.idle`로 복귀하므로 `feature.errorMessage == nil` 조건 필수
- `feature.articles.first?.id`가 nil인 경우 guard로 방어

**산출물**: refresh 성공 완료 시 FeedView 자동 상단 스크롤 (실패 시 미발동)

## 주의사항 및 리스크

- **join_all lifetime**: `SearchJob { tag_id, tag_name, query: String }` owned 구조체로 해결 — `&str` 직접 캡처 금지. future 내부에서 `chain.search(&job.query, 5)` 형태로 borrowed 사용
- **FakeSearchAdapter 쿼리 맵**: 태그별 다른 결과 표현을 위해 S0에서 먼저 개선. 동일 URL 반환 시 dedupe로 결과 감소 — 테스트 케이스마다 URL 분리 필수
- **웹 isRefreshing vs loading 분리**: 재진입 가드 `if (loading || isRefreshing) return;` 필수 — 두 상태 동시 true 방지
- **웹 scroll 타이밍**: `tick()` 후 `window.scrollTo` — DOM 반영 전 호출 방지. `browser` 가드로 SSR 안전
- **웹 scroll 조건**: `!feedStore.error && feedItems.length > 0` — 실패 refresh 시 스크롤 미발동
- **iOS scroll 조건**: `feature.errorMessage == nil && feature.articles.first?.id != nil` — 실패 refresh(`.idle` 복귀)에서 스크롤 방지
- **FakeSearchAdapter 병렬 검증 한계**: 순차/병렬 실행 시간 차이는 Fake로 측정 불가 — 호출 로그로 "두 쿼리 모두 발행됐다" 검증으로 대체
