MVP4 M5 완료 + MVP5 로드맵 확정 + M1 피드 아키텍처 전환 + M2 온디맨드 요약

하루에 MVP4 M5를 마무리하고, MVP5를 로드맵 작성부터 M1·M2 구현 완료까지 끝냈다. 핵심은 아키텍처 결정 하나였다 — "모두 수집·요약"에서 "피드는 실시간 검색, 요약은 요청할 때만"으로. DB 스키마를 통째로 다시 짰고, 온디맨드 요약 파이프라인(SummarizePort)도 서버·iOS·웹 모두 완성했다. M1 구현 중 두 번의 iteration이 있었는데, scheduler 모듈이 예상보다 범위가 컸다.
오늘의 결정 · 인사이트
DECISION 01

"모두 수집·요약" → "피드 = 실시간 검색, 요약은 온디맨드" — MVP5 핵심 아키텍처 전환

결정

articles 테이블을 제거하고, 피드는 Tavily 실시간 검색 결과로 바꿨다. 요약은 사용자가 기사를 선택할 때 POST /me/summarize를 호출하는 방식으로 전환. scheduler는 수집 주기만 관리하고, LLM 호출은 하지 않는다.

왜 중요한가

기존 방식은 스케줄러가 수집 시점에 모든 기사를 LLM으로 요약했다. 사용자가 보지도 않는 기사에 비용이 발생했고, 요약 품질도 일정하지 않았다. 실제 사용 패턴은 "목록 훑기 → 관심 기사만 요약 읽기"다. 이 흐름에 맞게 아키텍처를 바꾸면 비용·품질·UX 모두 좋아진다.

M1 구현 중 scheduler 모듈 범위가 예상보다 커서 커밋이 두 라운드로 나뉘었다. 처음 계획에서 scheduler를 과소평가했다. 하지만 방향은 맞았고, 재작업 없이 완료했다.
DB 스키마를 바꿀 때 "마이그레이션 파일 작성 → 로컬 테스트 → articles 테이블 제거" 순서를 지켜야 프로덕션 데이터가 안전하다. 이번엔 MVP 수준이라 괜찮았지만, 실서비스였다면 마이그레이션 계획이 더 엄밀해야 했다.
DECISION 02

SummarizePort 세션 캐시 — 같은 기사 재요약 방지

결정

세션 내에서 같은 URL의 기사를 재요약 요청하지 않도록 in-memory 캐시를 추가했다. iOS: SummarySessionCache: [URL: SummaryResult], 웹: summaryCache: Map<string, SummaryResult>. 앱 재시작 시 초기화 — 영속화 불필요.

왜

LLM 요약은 응답 시간 15~30초, 비용도 크다. 디테일 뷰에서 뒤로 갔다 다시 돌아오면 재요청하던 구조를 캐시로 막았다. 세션 단위 캐시는 구현이 단순하고, 메모리 부담도 없다.

캐시 설계의 두 축: 어디까지 기억할 것인가(scope)와 언제 버릴 것인가(invalidation). 세션 캐시는 scope = 앱 실행 중, invalidation = 앱 종료. 가장 단순한 형태다. 나중에 오프라인 지원이 필요해지면 영속화(Keychain/SQLite)로 확장하면 된다.
DECISION 03

인증·토큰·저장소 개념 정리 — 멘토 발표 후속 나선형 학습

배경

어제 멘토 발표에서 "흐름은 알지만 왜인지 모른다"는 문제가 드러났다. Supabase SDK 인증 흐름과 우리 서버 서비스 흐름이 완전히 다른 경로인데 하나의 도식에 묶어서 잘못 표현했다.

결정

인증·JWT·쿠키·Keychain 개념을 Mermaid 도식화 포함 HTML로 정리했다. 인증 흐름(Supabase SDK)과 서비스 흐름(우리 Rust 서버)을 두 개의 다이어그램으로 분리.

멘토의 말대로 "설명할 수 있을 정도"면 충분하다. 코드가 왜 그렇게 생겼는지 연결되면, 나중에 같은 개념을 다시 마주칠 때 더 깊게 팔 수 있다. 한 번에 다 이해하려는 욕심이 오히려 아무것도 제대로 모르는 상태를 만든다.
