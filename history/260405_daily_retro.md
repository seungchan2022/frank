## 핵심 의사결정과 '왜'

### [IMPACT] Claude Code 설정 체계 자동화

MVP1의 43개 커밋을 분석하니, main 직접 커밋 100%, 테스트 미실행 60%였다.
CLAUDE.md에 "금지"라고 적어도 지켜지지 않았다.
해결책: settings.json deny + pre-commit hook + commit-msg hook.
**규칙을 문서에서 엔진으로 옮기자 위반율이 0%가 됐다.**
이게 잘 동작한다는 걸 오늘 직접 체감했다 — main에 커밋하려 했더니 hook이 차단했고,
그게 "예상과 달랐던 점"이자 동시에 가장 만족스러운 순간이었다.

### [PATTERN] 서버/웹/iOS 동일 포트/어댑터 패턴

세 앱 모두 같은 패턴을 쓰니 컨텍스트 스위칭 비용이 줄어든다.
Mock Port 기반 TDD도 잘 맞았다. 다만 실제 어댑터 구현(M2~)을 해봐야
iOS에서 과한 추상화인지 판단할 수 있다. 판단 보류 — 체감 후 평가 예정.

### [SURPRISE] Tuist MCP: tools가 아예 없다

MCP 연결까지 했는데 tools capability가 비어있었다.
resources template(Xcode 그래프 읽기)만 지원.
결국 CLI로 프로젝트 생성 + MCP resource로 검증하는 하이브리드 전략으로 전환.
"MCP = 만능"이 아니라 "MCP = 각 서버가 노출하는 만큼만"이라는 교훈.

## 인사이트

- **규칙 강제의 법칙:** "문서에 적힌 규칙은 강제가 아니라 권고다." settings.json deny, Git hook, Claude hook — 세 층이 겹쳐야 비로소 강제. 이 구조는 AI 에이전트뿐 아니라 사람 팀에도 적용되는 보편적 원리다.

- **os.Logger의 함정:** info 레벨은 simctl log stream에 안 나온다. notice 이상이어야 확인 가능. 디버깅용 로그는 notice, 프로덕션 노이즈 줄이기는 info — 이 구분을 기억할 것.

- **의존성 추가 타이밍:** Supabase Swift SDK를 M1에서 안 넣고 M2로 미뤘다. Port(protocol)만 정의하면 SDK 없이 M1은 완결된다. "필요할 때 추가"가 포트/어댑터 패턴의 자연스러운 리듬.
