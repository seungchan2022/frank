MVP4 M3 웹 timeout 완료 + 로드맵 재편 + 멘토 발표

MVP4 5개 마일스톤을 정의하고, 웹 요약 timeout을 구현했다. 그런데 멘토에게 발표하면서 드러난 건 흐름만 알고 "왜"를 모른다는 것이었다. 쿠키가 뭔지, PKCE를 왜 쓰는지, JWT와 idToken의 차이가 뭔지 — 말로 설명하려니 막혔다. 개념과 구현을 동시에 빠르게 하려다 둘 다 어중간해진 패턴을 오늘 직면했다.
오늘의 결정 · 인사이트
DECISION 01

MVP4 로드맵을 5개 마일스톤으로 재편 — 기획은 "무엇을 만들지" 이지 "어떻게 만들지"가 아니다

결정

MVP4 작업 목록을 M1~M5로 재편했다. 각 마일스톤은 구현 범위(무엇을 만들지)를 정의하는 것이고, 코드 구현 방법은 여기 담지 않는다. 9단계 workflow 적합성을 기준으로 아이템을 분류했다.

왜 이게 중요한가

마일스톤 정의를 하면서 "구현 방법"까지 같이 생각하려 했는데, 그러면 기획이 아니라 설계가 돼버린다. 마일스톤은 범위를 잡는 것. 어떻게 만들지는 workflow 안에서 결정한다. 이 구분이 명확해지면 기획 단계에서 헤매는 시간이 줄어든다.

마일스톤 = "무엇을" / workflow = "어떻게". 이 구분이 흐리면 기획이 설계가 되고, 설계가 구현이 되면서 범위가 계속 커진다. M3 범위 축소가 좋은 사례 — ErrorBanner 공통화는 "어떻게"의 문제였고, 새 화면 추가 시점(MVP5)에 같이 하는 게 맞다.
DECISION 02

웹 요약 timeout — iOS M2와 같은 패턴으로 동기화

결정

웹 요약 API 호출에 30초 timeout + 재시도 버튼을 구현했다. iOS M2에서 쓴 AbortController 패턴을 웹에 그대로 적용해 크로스 플랫폼 UX를 맞췄다.

왜

iOS에서 요약이 느릴 때 사용자가 무한 로딩에 갇히는 문제를 M2에서 해결했고, 웹도 같은 상황이 발생한다. 패턴을 재사용하면 UX 일관성도 확보하고 구현도 빠르다.

같은 문제를 두 플랫폼에서 같은 방식으로 풀 수 있다면, 첫 번째 구현이 레퍼런스가 된다. iOS M2 → 웹 M3 순서가 그래서 맞았다.
DECISION 03

멘토 발표에서 드러난 것 — 흐름은 알지만 "왜"를 모른다

어떤 상황이었나

지금까지 구현한 것의 흐름·용어·개념을 정리해 Reveal.js 프레젠테이션으로 만들었고, 멘토에게 발표했다. 흐름은 대충 알고 있었는데, 설명하려니 계속 막혔다. 쿠키를 왜 쓰는지, 도식의 인증 흐름이 실제와 다르다는 것, PKCE가 뭔지 — 공부 목록은 따로 정리했다. (progress/notes.md)

도식화 문제

인증과 서비스를 하나의 도식으로 묶어 표현했는데, 실제로는 인증은 Supabase SDK를 타고 우리 서버를 거치지 않는다. 하나로 합쳐놓으니 잘못된 정보가 도식에 나왔고 보기도 복잡했다. 인증 흐름 / 서비스 흐름을 두 개로 분리해야 한다.

근본 원인

개념 공부와 구현을 동시에 빠르게 하려다 둘 다 어중간해졌다. 하나에 너무 깊게 파려는 성향 때문에 오히려 넘어가버렸고, 그 결과 지금 뭘 만들고 있는지, 어떻게 동작하는지를 말로 설명하지 못하는 상태가 쌓였다.

이건 기술 문제가 아니라 학습 방식의 문제다. 완전히 이해하고 넘어가려는 욕심이 오히려 아무것도 제대로 모르는 상태를 만들었다. 발표에서 드러난 게 오히려 다행이다.
개념은 "설명할 수 있을 정도"로만 이해하고 구현을 진행해라. 나중에 같은 개념을 다시 마주칠 때 그때 더 깊게 파면 된다. 빠르게 자주 보는 나선형 방식으로. 처음부터 완벽하게 이해하려 하지 마라.
