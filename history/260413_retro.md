## 의사결정 기록

### 1. iOS 퀴즈 "Swift 문제만 나오는" 버그

**결정**: AppDependencies에 `quiz` 포트 추가, `ArticleDetailView` 기본값 Mock 제거

**발견 경로**: 시뮬레이터에서 직접 앱을 써보다가 발견. 어떤 기사를 열어도 Swift 관련 퀴즈만 나왔고, 코드를 추적하니 `ArticleDetailView`의 `quiz` 파라미터 기본값이 `MockQuizAdapter()`였다. `AppDependencies`에 해당 포트가 없어서 View 생성 시 항상 Mock이 주입되는 구조적 결함.

**인사이트**: 단위 테스트는 통과해도 실제 의존성 연결 구조는 시뮬레이터 직접 사용에서만 잡힌다. AppDependencies 체인은 새 포트 추가 시 반드시 전체 연결을 확인해야 한다.

---

### 2. MVP8 스코프 — M2/M3 별도 마일스톤으로 분리

**결정**: UX 개선(M2)과 오답 아카이빙(M3)을 독립 마일스톤으로 구성

**이유**:
- M2는 신규 API 호출 없이 기존 화면 수정만 → M3와 코드 충돌 없이 독립 진행 가능
- M3는 M1에서 추가된 엔드포인트를 소비하므로, M1 → M2 → M3 의존성 방향이 자연스럽게 순서를 결정

**인사이트**: 마일스톤 분리 기준은 "작업량"이 아니라 "API 의존성 방향"이어야 한다. M2가 API 독립적이라는 사실이 병렬/직렬 설계의 근거가 됐다.

---

### 3. snippet 정제 — 외부 API 대신 코드 필터링

**결정**: Jina Reader 등 외부 크롤링 API 대신 `clean_snippet()` 함수로 HTML/URL/이메일 제거 + 단어 경계 절단. Known Issues로 한계 문서화.

**이유**:
- 외부 API 추가는 레이턴시·비용·의존성 리스크를 동시에 올림
- MVP 단계에서는 "완벽한 정제"보다 "빠른 개선 후 Known Issues 문서화"가 실용적
- 실제로 한국 뉴스 사이트의 네비게이션 텍스트 혼입은 외부 API 없이는 해결 어렵지만, 영어 기사 커버리지를 유지하는 현재 단계에서는 치명적이지 않음

**인사이트**: Known Issues 문서화는 "나중에 고치겠다"는 미루기가 아니라, 현재 결정의 한계를 명시하는 기술 부채 관리다. 다음 MVP에서 재검토 기준이 된다.

---

### 4. 퀴즈 UX — 확인 버튼 추가

**결정**: "보기 선택 즉시 정오 표시" → "확인 버튼 클릭 후 표시"로 변경

**발견 경로**: 시뮬레이터에서 실제로 퀴즈를 풀다가, 스크롤 중 실수로 보기를 탭하면 즉시 결과가 나와서 당황스러웠음.

**인사이트**: "즉각 피드백"이 항상 좋은 UX가 아니다. 모바일에서 터치는 실수가 많으므로, 선택과 확정 사이의 의도적 간격이 필요하다. 이 버그는 기획 단계가 아니라 직접 사용에서만 발견된다.

---

### 5. 연관 기사 제거

**결정**: MVP8 M2에서 연관 기사 섹션 전체 제거. 재설계는 다음 MVP로 이관.

**이유**: 현재 구현이 키워드 기반 검색 결과 → "취향 추천"이 아니라 "유사 검색". 원래 의도(사용자 취향 기반 연관)와 다른 기능이 계속 화면에 노출되는 것보다, 과감히 제거하고 제대로 설계하는 게 낫다고 판단.

**인사이트**: "있는 것보다 없는 게 나은 기능"이 존재한다. 잘못된 방향의 기능은 사용자 기대를 어긋나게 만들고, 나중에 올바른 기능을 붙일 때 혼란을 더한다. 제거도 의사결정이다.
