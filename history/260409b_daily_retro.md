## 결정 01 — "모두 수집·요약" → "피드 = 실시간 검색, 요약은 온디맨드"

**결정**: MVP5 핵심 아키텍처를 전환. articles 테이블 제거, 피드는 Tavily 실시간 검색 결과로, 요약은 사용자가 기사를 선택할 때만 호출.

**왜 중요한가**: 기존 방식은 스케줄러가 수집 시점에 모든 기사를 LLM으로 요약했다. 사용자가 보지도 않는 기사에 비용이 발생했고, 요약 품질도 일정하지 않았다. 실제 사용 패턴은 "목록 훑기 → 관심 기사만 요약 읽기"다. 이걸 모델에 반영하면 비용·품질·UX 모두 좋아진다.

구현 과정에서 scheduler 모듈이 별도로 필요했고(태그별 수집 주기 관리), M1 커밋이 두 라운드로 나뉘었다. 범위를 처음에 과소평가했지만, 방향은 맞았다.

---

## 결정 02 — SummarizePort 세션 캐시 (iOS + 웹)

**결정**: 같은 URL의 기사를 세션 내에서 재요약 요청하지 않도록 in-memory 캐시 추가. iOS는 `SummarySessionCache: [URL: SummaryResult]`, 웹은 `summaryCache: Map<string, SummaryResult>`.

**왜**: LLM 요약은 응답 시간 15~30초, 비용도 크다. 디테일 뷰에서 뒤로 갔다 다시 돌아오면 재요청하던 구조를 캐시로 막았다. 세션 단위이므로 앱 재시작 시 초기화 — 별도 영속화 불필요.

---

## 결정 03 — 인증·토큰·저장소 개념 정리 HTML

**결정**: 어제 멘토 발표에서 드러난 "흐름은 알지만 왜인지 모른다" 문제에 대한 후속 조치로, 인증·토큰·저장소 개념을 Mermaid 도식화 포함 HTML로 정리했다.

**왜**: 발표에서 인증과 서비스 흐름을 하나의 도식에 묶었더니 틀린 정보가 표현되었다. Supabase SDK가 처리하는 인증 흐름과 우리 서버가 처리하는 서비스 흐름은 완전히 분리된 경로다. 개념을 "설명할 수 있을 정도"로 정리하고 나서야 코드가 왜 그렇게 생겼는지 연결됐다.
