## TL;DR

M2가 좋아요 데이터를 모으는 단계였다면, M3는 그 데이터를 실제로 써먹기 시작한 첫 번째 단계다. 피드 쿼리 보강, 연관 기사 엔드포인트, LikesFeature 비관적 업데이트, LLM soft fail, 그리고 cross-tag 키워드 오염 버그까지 5개 커밋으로 M3를 닫았다. 기술보다 큰 과제는 따로 있었다 — 앱이 어디로 가야 하는지 문서로 먼저 명확히 잡는 것. 구현하고 나서야 "이 방향이 아닌 것 같다"는 느낌이 오는 루프가 오늘도 있었다.

## 결정 01 — M3의 진짜 의미: "수집"에서 "활용"으로의 전환

**결정**: M2는 좋아요 → 키워드 추출 → `user_keyword_weights` 누적이라는 데이터 수집 파이프라인이었다. M3에서 처음으로 그 데이터를 피드 검색 쿼리에 실제로 주입했다. `like_count >= 3` threshold를 넘으면 상위 3개 키워드가 Tavily 검색 쿼리에 추가된다. API 시그니처는 그대로, 서버 내부 로직만 바뀐다.

**왜 중요한가**: 사용자가 무엇을 좋아하는지 모으기만 하는 앱과, 그것을 반영해서 경험을 바꾸는 앱은 다르다. M3는 Frank가 단순 뉴스 피드에서 개인화 피드로 넘어가는 첫 번째 커밋이다. 기술적으로는 단순한 쿼리 보강이지만, 제품 방향에서는 큰 전환점이다.

인사이트: 개인화는 "추천 알고리즘"이 아니라 "사용자가 준 시그널을 어디에 쓰느냐"의 문제다. 좋아요 3회라는 단순한 threshold로도 피드 결과가 달라지기 시작한다.

## 결정 02 — tag_id 지정 피드에서 키워드 boost 제거

**문제**: 처음 구현에서 피드 개인화 boost가 전체 피드(`tag_id=None`)뿐 아니라 특정 태그를 선택한 피드(`tag_id=xxx`)에도 적용됐다. iOS 구독 A 태그를 보는데, B 태그를 좋아요하며 추출된 키워드가 쿼리에 섞이는 상황.

**결정**: `tag_id`가 지정된 요청에는 키워드 boost를 적용하지 않는다. 전체 피드 조회(tag_id = None)에만 boost를 적용해 다른 태그 키워드가 섞이지 않도록 경계를 명확히 했다. 다음 MVP에서 `user_keyword_weights`에 `tag_id` 컬럼을 추가하면 "키워드가 유래된 태그에만 boost"라는 더 정확한 버전으로 업그레이드 가능.

버그 원인: 피드 개인화 로직이 scope 없이 전역으로 동작했던 게 원인. `tag_id.is_some()` 체크 한 줄로 해결됐지만, "어떤 조건에서 어느 범위에 boost를 적용하는가"를 설계 단계에서 명확히 해두지 않았기 때문에 발생.

## 결정 03 — LikesFeature fire-and-forget → 비관적 업데이트 전환

**결정**: 좋아요 탭 즉시 `likedUrls`에 추가(낙관적)하던 방식을 API 성공 후에만 상태를 업데이트하는 비관적 업데이트로 바꿨다. 실패 시 `error`가 설정되고 `likedUrls`는 변경되지 않는다. TDD 테스트 명세가 비관적 업데이트를 전제하고 있어서 테스트 통과 요건이기도 했다.

**왜**: 좋아요는 서버에 키워드 가중치를 누적하는 작업이라, 실패했는데 하트가 채워지면 혼란스럽다. 비관적 업데이트는 응답 지연이 있지만 상태 일관성을 보장한다. 낙관적 vs 비관적은 "빠른 피드백"과 "상태 일관성" 사이의 트레이드오프 — 서버 사이드 효과가 중요한 경우엔 비관적이 맞다.

## 결정 04 — 앱 방향과 문서의 선후관계

**관찰**: 오늘 가장 힘들었던 건 코드가 아니라 방향 설정이었다. 문서를 먼저 명확히 잡아야 구현이 그대로 흘러가는데, 앱이 어디로 가야 하는지가 탐색 중인 상태라 문서를 먼저 확정하기가 어렵다. 구현하고 나서야 "연관 기사가 원래 의도와 다르게 됐다"는 걸 알게 되는 루프가 반복됐다.

**현재 상태**: M3의 연관 기사는 "기사 상세 하단 — 현재 기사 title+snippet+키워드 기반 유사 기사"로 구현됐지만, 원래 의도는 "사용자 취향 기반 추천 섹션(기사 상세와 무관)"이었다. 기능은 동작하지만 제품 개념이 다르다. 다음 MVP에서 재설계 대상으로 이미 문서에 적어뒀다.

인사이트: 탐색 단계에서 "문서 → 구현" 순서를 강제하기보다, "구현 → 불일치 발견 → 문서 갱신"을 빠르게 반복하는 게 현실적이다. 중요한 건 불일치를 발견했을 때 바로 문서에 남기는 것.
