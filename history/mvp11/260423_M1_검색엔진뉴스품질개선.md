## ST-1: Exa `category:"news"` + `startPublishedDate` 추가

> **[리뷰 수정 — 2026-04-23]** Exa 공식 문서 확인 결과, `type` 파라미터는 검색 알고리즘 선택용(`"neural"` / `"fast"` / `"auto"` 등)이며, 뉴스 콘텐츠 필터는 별도 `category` 파라미터로 처리한다. 원래 플랜의 `"type": "news"` 는 무효한 값으로 수정 효과가 없었을 것이므로 `"category": "news"` 로 수정한다.

### 현재 상태 (`exa.rs:109-118`)

```rust
let body = serde_json::json!({
    "query": query,
    "numResults": max_results,
    "contents": {
        "highlights": { "numSentences": 3, "highlightsPerUrl": 1 }
    }
});
```

### 목표 상태

```rust
let now = chrono::Utc::now();
let week_ago = now - chrono::Duration::days(7);
let body = serde_json::json!({
    "query": query,
    "numResults": max_results,
    "category": "news",
    "startPublishedDate": week_ago.to_rfc3339(),

        "highlights": { "numSentences": 3, "highlightsPerUrl": 1 }

});
```

## ST-3: URL 패턴 후처리 필터 추가 (listing 페이지 제거)

### 목적

ST-1·ST-2 파라미터 수정만으로 완전히 차단하지 못하는 listing 페이지를 URL 패턴 기반으로 제거한다. Firecrawl (3순위 폴백)은 뉴스 파라미터 자체가 없으므로 이 필터가 유일한 방어선이 된다.

### 필터링 대상 URL 패턴

```
/tag/, /tags/
/category/, /categories/
/topics/, /topic/
/section/
```

> 주의: `/news/`, `/articles/` 는 포함하지 않는다. 이 패턴은 정상 기사 URL (예: `/news/some-article`)을 오탐(false positive)할 수 있다.

### 위치 선택 — **A안 확정** (`feed.rs`)

`is_homepage_url()` 옆에 `is_listing_url()` 추가. 동종 URL 필터 응집, 현재 소비자 단일(feed.rs)로 기능 차이 없음.

### 추가할 함수 (`server/src/api/feed.rs`)

> **[리뷰 수정 — 2026-04-23]** `contains()` 단순 매칭은 `/category/tech/2026/article-slug` 같은 유효 기사 URL을 오탐한다. 패턴 다음에 의미 있는 슬러그가 없을 때(= 해당 세그먼트로 URL이 끝나거나 trailing slash만 남을 때)만 listing으로 판별한다.

```rust
/// listing URL 패턴 (태그·카테고리·섹션 인덱스 페이지) 판별
/// contains() 대신 path 마지막 2개 세그먼트 기준으로 판별하여 오탐 방지.
/// 예) /category/tech/article-slug → 통과, /category/tech → 차단
fn is_listing_url(url: &str) -> bool {
    // path 추출 (스킴+도메인 이후)
    let path = url
        .find("://")
        .and_then(|scheme_end| {
            let after_scheme = &url[scheme_end + 3..];
            after_scheme
                .find('/')
                .map(|slash_pos| &after_scheme[slash_pos..])
        })
        .unwrap_or("/");
    let path = path.to_lowercase();
    // query string 제거
    let path = path.split('?').next().unwrap_or(&path);
    let path = path.trim_end_matches('/');

    let segments: Vec<&str> = path.split('/').filter(|s| !s.is_empty()).collect();

    // listing 패턴 세그먼트
    let listing_segments = ["tag", "tags", "category", "categories", "topics", "topic", "section"];

    // 마지막 세그먼트가 listing 키워드 → listing 페이지
    // 마지막-1 세그먼트가 listing 키워드이고 마지막이 슬러그가 아닌 경우
    // (단순 /category/tech는 2개 세그먼트: ["category", "tech"] — 마지막이 listing 아니므로 통과)
    // 실제로 /category/ (세그먼트 1개: ["category"]) → 차단, /tag/ → 차단
    // /category/tech/article (3개 세그먼트) → 마지막 "article"은 listing 아님 → 통과
    match segments.last() {
        None => false,
        Some(&last) => listing_segments.contains(&last),
    }
}
```

> **구현 결정 근거**: 마지막 path 세그먼트가 listing 키워드인 경우만 차단한다. 이 방식은:
> - `/tag/` → 차단 (`["tag"]`)
> - `/category/tech` → **통과** (`["category", "tech"]`, last="tech")
> - `/topics/` → 차단 (`["topics"]`)
> - `/topics/ai` → **통과** (`["topics", "ai"]`, last="ai")
> - `/tags/` → 차단 (`["tags"]`)
>
> **트레이드오프**: `/tag/ai-news`(태그 listing)를 통과시킴. 하지만 이 경우는 `is_homepage_url`도 통과하며, 실제 기사 URL(`/news/article-title`)과의 오탐 비용이 더 낮으므로 현재 단계에서 수용 가능.

### 추가할 테스트

- `listing_url_tag_only_filtered` — `https://example.com/tag/` → 차단 확인
- `listing_url_category_only_filtered` — `https://example.com/category/` → 차단 확인

## 아키텍처 결정 기록 (2026-04-23)

URL 패턴 필터는 **두더지 잡기 방식**이며 근본 해결이 아님. 각 웹사이트의 URL 구조가 달라 완전한 차단 불가. `published_at` 유무 기반 필터도 Exa 전용 커스텀이 되어 다른 엔진에서 정상 기사가 탈락하는 문제 발생. 현재 URL 필터는 1차 방어선으로만 사용하고, 근본 해결은 후속 마일스톤에서 아래 방법 중 하나로 진행:

- 뉴스 전용 API 교체 (NewsAPI 등) — 기사만 반환 보장
- LLM 분류 후처리 — 엔진 독립적, 추가 비용 감수
