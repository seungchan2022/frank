# MVP5 로드맵 — 학습 기능 (피드 · 요약 · 즐겨찾기)

> 작성일: 2026-04-09  
> 브랜치 기반: `main` (MVP4 완료 이후)

---

## 목표

"보고 싶은 것만 요약, 저장하고 싶은 것만 즐겨찾기" 흐름으로 전환한다.  
피드는 매번 검색 API를 직접 호출해 결과를 표시한다. DB 저장은 즐겨찾기 시에만 발생한다.

## 아키텍처 원칙 (확정)

| 항목 | 결정 |
|------|------|
| 피드 탭 | 검색 API 직접 호출 → 결과 표시. **DB 저장 없음** |
| pull-to-refresh | 동일한 검색 API 재호출. DB 저장 없음 |
| DB 저장 | **즐겨찾기 시에만** favorites 테이블 |
| 즐겨찾기 탭 | DB에서 favorites만 읽어 표시. 피드와 완전히 독립 |
| articles 테이블 | **제거** (불필요) |
| 백그라운드 스케줄러 | **제거** (불필요) |
| 요약/인사이트 출력 | **한국어 고정** |
| 요약 캐시 | **세션 캐시** (클라이언트 메모리, 앱 종료 시 소멸) |
| 플랫폼 구현 순서 | **서버 먼저 → iOS + 웹 동시** |

## DB 스키마

### articles 테이블: 제거

articles 테이블은 불필요. 피드는 검색 API를 직접 호출해 ephemeral 결과를 사용한다.

### favorites 테이블 (신규)

```sql
favorites (
  id           uuid PK default gen_random_uuid(),
  user_id      uuid FK → profiles NOT NULL,
  title        text NOT NULL,
  url          text NOT NULL,
  snippet      text,
  source       text NOT NULL,
  published_at timestamptz,
  tag_id       uuid,
  summary      text,
  insight      text,
  liked_at     timestamptz,
  created_at   timestamptz default now() NOT NULL,
  UNIQUE (user_id, url)
)
```

> article_id FK 없음 — articles 테이블을 참조하지 않고 기사 메타 전체를 직접 저장.  
> UNIQUE (user_id, url) — 동일 URL 중복 즐겨찾기 방지.

## M1: 피드 아키텍처 전환

### 목표
피드를 DB 기반에서 검색 API 온디맨드 호출 방식으로 전환.  
articles 테이블 + 백그라운드 스케줄러 완전 제거.

### 서버

**마이그레이션**:
- articles 테이블 관련 기존 마이그레이션을 롤백/교체
- favorites 테이블 신규 생성 (위 스키마)

**신규 엔드포인트**:
```
GET /me/feed   → 검색 API 직접 호출, 결과 반환 (DB 저장 없음)
```

**GET /me/feed 동작**:
1. 사용자 태그 조회 (DB)
2. 각 태그별 검색 API 호출 (SearchChainPort)
3. 결과 취합 → `Vec<FeedItem>` 반환 (DB 저장 없음)

**GET /me/feed 응답**:
```json
[
  {
    "title": "...",
    "url": "...",
    "snippet": "...",
    "source": "...",
    "published_at": "...",
    "tag_id": "..."
  }
]
```

**제거**:
- `POST /me/collect` 엔드포인트
- `GET /me/articles` 엔드포인트
- `collect_service.rs`
- `scheduler/` 모듈
- `DbPort`의 articles 관련 메서드 (save_articles, get_user_articles, get_user_article_by_id, get_all_user_ids)

### iOS

- Pull-to-refresh: `GET /me/feed` 직접 호출 (POST /me/collect 제거)
- `CollectPort` → `FeedPort`로 전환 (또는 제거 후 ArticlePort 통합)
- `APICollectAdapter` → `APIFeedAdapter`로 전환
- `FeedFeature`: `collectAndRefresh` → `fetchFeed`

### 웹

- `GET /me/feed` 호출로 전환 (POST /me/collect + GET /me/articles 조합 제거)
- `collectArticles()` 제거 → `fetchFeed()` 추가
- 새로고침 = `GET /me/feed` 재호출

- [x] favorites 테이블 생성 (위 스키마)
- [x] GET /me/feed 엔드포인트 동작 확인
- [x] POST /me/collect 제거

## M2: 디테일 페이지 + 온디맨드 요약

### 목표
기사 클릭 시 디테일 페이지 진입. 요약/인사이트는 버튼 클릭 시 온디맨드 LLM 호출.

### 서버

**신규 엔드포인트**:

```
POST /me/summarize  → URL 크롤링 + LLM 요약/인사이트 생성 → 반환
```

**POST /me/summarize 요청 바디**:
```json
{
  "url": "https://...",
  "title": "..."
}
```

**POST /me/summarize 동작**:
1. URL 크롤링 (Firecrawl) → 본문 추출
2. LLM 호출: 요약 + 인사이트 한국어로 생성
3. 결과 반환 (DB 저장 안 함 — 클라이언트 세션 캐시)
4. favorites에 해당 url이 있으면 → summary, insight 컬럼 업데이트

**응답**:
```json
{
  "summary": "...",
  "insight": "..."
}
```

> `GET /me/articles/:id` 없음 — articles DB 없음. 디테일은 클라이언트가 피드 데이터 그대로 사용.

**타임아웃**: 30초

### iOS

**ArticleDetailView 구성**:
```
[ 제목 ]
[ source | published_at ]
[ 원문 리드 (snippet, 3~5문장) ]
[ "원문 보기" 버튼 ] → SFSafariViewController로 열기
─────────────────────────────
[ 요약하기 버튼 ]
  → 로딩 중: 30초 타임아웃 + 재시도 버튼
  → 완료 시:
    [ 요약 섹션 ]
    [ 인사이트 섹션 ]
─────────────────────────────
[ 즐겨찾기 버튼 ] (M3에서 구현, 여기서 레이아웃만 예약)
```

**세션 캐시**:
- `ArticleDetailFeature`에서 `[url: SummaryResult]` 딕셔너리로 관리 (article_id 아닌 url 키)
- 같은 기사 재진입 시 LLM 재호출 없이 캐시 반환

### 웹

**`/feed/[url]` 또는 `/feed?url=...` 페이지**:
- 제목, source, 날짜
- snippet (원문 리드)
- "원문 보기" 링크 → 새 탭
- "요약하기" 버튼 → 로딩 UX (spinner + 30초 타임아웃)
- 요약/인사이트 결과 표시 섹션

**세션 캐시**:
- Svelte store에 `Map<url, SummaryResult>` 보관

## M3: 즐겨찾기 + 스크랩 목록

### 목표
즐겨찾기 시 기사 메타 + 요약/인사이트(nullable)를 favorites DB에 저장.  
스크랩 목록은 하단 탭바 별도 탭으로 접근.

### 서버

**신규 엔드포인트**:

```
POST   /me/favorites           → 즐겨찾기 추가
DELETE /me/favorites           → 즐겨찾기 해제 (body에 url) 또는 DELETE /me/favorites/:id
GET    /me/favorites           → 즐겨찾기 목록 (최신순)
```

**POST /me/favorites 요청 바디**:
```json
{
  "url": "...",
  "title": "...",
  "snippet": "...",
  "source": "...",
  "published_at": "...",
  "tag_id": "...",
  "summary": null,
  "insight": null
}
```

**GET /me/favorites 응답**:
```json
[
  {
    "id": "...",
    "title": "...",
    "url": "...",
    "source": "...",
    "published_at": "...",
    "tag_id": "...",
    "summary": null,
    "insight": null,
    "created_at": "..."
  }
]
```

### iOS

**즐겨찾기 버튼** (ArticleDetailView):
- 하트/북마크 아이콘
- 누르면: 현재 세션의 summary/insight(있으면 포함) + 기사 메타데이터 → POST /me/favorites
- 이미 즐겨찾기된 상태면: 채워진 아이콘 표시 + 누르면 DELETE

**즐겨찾기 후 요약**:
- favorites에 이미 등록된 기사에서 요약하기 → POST /me/summarize
- 서버가 favorites 테이블 자동 업데이트

**스크랩 목록 탭**:
- 하단 탭바에 북마크 아이콘 탭 추가
- 즐겨찾기 목록 (제목 + source + 즐겨찾기 날짜, 최신순)
- 탭하면 ArticleDetailView로 진입 (요약 있으면 바로 표시)

### 웹

**즐겨찾기 버튼** (디테일 페이지):
- 하트/북마크 아이콘, 동일 동작

**스크랩 목록 탭**:
- `/favorites` 라우트
- 목록 (제목 + source + 즐겨찾기 날짜, 최신순)
