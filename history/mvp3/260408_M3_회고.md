## 실 서버 검증 중 발견 (ST-8 §방법 3 진행 중)

ST-8 §방법 3 진행하며 실 Supabase + Rust 서버에 시뮬레이터를 연결하던 중 다음을 발견. **여러 부채가 동시에 드러남**.

### iOS 측 발견 → M3 안에서 모두 fix 완료

#### 1. ST-5 절반만 fix되어 있던 결함 (진짜 fix)
- **증상**: 온보딩 → "시작하기" 누르면 화면 변화 없음 (다시 onboarding)
- **원인**: ST-5에서 `updateOnboardingCompleted`를 `PUT /api/me/profile`로 정정했지만, **`currentSession`/`mapToProfile`은 여전히 Supabase user_metadata에서 읽음**. 두 데이터 출처 분리 → server profiles 테이블 갱신해도 다음 checkSession이 user_metadata 보고 false 반환 → RootView가 Onboarding 분기 유지
- **fix**: `ProfileAPI.fetchProfile(token:serverURL:session:)` 추가, `SupabaseAuthAdapter`의 모든 profile 반환 메서드(signIn/signUp/signInWithApple/currentSession)가 server fetch 사용. `mapToProfile` 완전 제거. 404 fallback만 minimal profile 반환
- **테스트**: ProfileAPITests +3 (200/404/401)

#### 2. PostgREST microseconds 6자리 timestamp 디코딩 결함
- **증상**: "수집/요약에 실패했습니다" 에러 (수집 자체는 server에서 200으로 통과)
- **원인**: `JSONDecoder.dateDecodingStrategy = .iso8601`은 `ISO8601DateFormatter`를 내부 사용하는데, fractional seconds를 디폴트 미지원. `.withFractionalSeconds`도 millisecond 3자리까지만. **PostgREST가 반환하는 `2026-04-07T07:32:37.350714Z`(microseconds 6자리)는 항상 디코딩 실패** → fetchAndCache throw → catch
- **fix**: `APIArticleAdapter.makeDecoder()` custom strategy. 표준 ISO8601 + fractional 3자리 + microseconds(6자리) → millisecond truncation 후 재시도
- **테스트**: APIArticleAdapterTests +1 (`fetchArticlesMicrosecondTimestamp`)

#### 3. NSURLError -999 cancelled 경합 (미fix, 부채)
- **증상**: 수집/요약 진행 중 사용자가 기사 카드 → 상세 진입 → 뒤로 돌아오면 "수집/요약 실패" 에러 배너
- **원인**: SwiftUI navigation으로 view tree가 바뀌면서 진행 중이던 `Task { await feature.send(.collectAndSummarize) }`가 cancel됨. URLSession task가 cancel되며 throw → catch → "실패"
- **현재 상태**: 미fix. M3 ST-7 부채 흡수 범위 외. 후속 작업 필요
- **fix 방향**: collectAndSummarize 호출을 view-bound Task가 아닌 detached Task로, 또는 cancellation을 감지해 errorMessage 안 띄우고 silent skip

### server 측 발견 → server hotfix 필요 (M3 외)

#### 4. 요약 기능 미동작 → server hotfix 2건으로 해결 ✅
- **증상(당시)**: `POST /api/me/summarize` → `{"summarized":0}` 반환. 20개 기사 모두 `summary=null`, `insight=null`, `title_ko=null`. server는 200 OK 반환하지만 실제로 LLM 호출이 작동 안 함. **웹(M2)에서도 동일**
- **server hotfix 1 — `5d0adc7`**: OpenRouter reasoning 파라미터 `effort:"none"` → `exclude:true`. MiniMax M2.5 mandatory reasoning 모델이 `effort:none`을 거부해 400 응답 후 silent failure
- **server hotfix 2 — `8a9a778`**: Rust 서버 CORS 미들웨어 추가 (웹/iOS 클라이언트 직접 호출 지원)
- **본 worktree에 적용**: `git stash -u && git rebase main && git stash pop`으로 두 hotfix 흡수
- **검증**: 신규 user 12건 수집 → **요약 10/12 (83%)** 성공. web 17/20과 동등. DB 직접 조회로 확인
- **결정**: 해결됨. ST-9 진행 가능

#### 5. server `/api/me/summarize` 60s timeout (신규 부채 — MVP4로 이월) 🆕
- **증상**: 기존 user(누적 미요약 20건)에서 iOS의 `/api/me/summarize` 호출이 60s URLRequest timeout으로 status -1 실패. 신규 user(미요약 12건)는 23s 내 완료되어 성공
- **iOS 로그 증거**: `Task <28> ... transaction_duration_ms=60999, response_status=-1, NSErrorFailingURLStringKey=http://localhost:8081/api/me/summarize`
- **원인 추정**: server `/api/me/summarize` handler가 N개 LLM 호출을 sequential 처리. N≥10에서 60s 초과
- **임계값**: 12건 = 23s OK / 20건 = 60s+ NG. 8~12건 사이가 경계
- **iOS-only fix는 미봉책**: timeout 60→180s 1줄 변경 가능하지만, N≥40에서 또 깨짐. 진짜 fix는 server 쪽 (batch + async + limit)
- **결정**: M3에서 fix 안 함. **MVP4(성능·안정성·관측성)** 신규 사이클로 이월
  - 이유: ① SESSION_SCOPE(server/ read-only) ② 진짜 fix는 server 아키텍처 결정 필요 ③ 배포 일정 없음 → 임시 패치 명분 없음 ④ M3 미션(iOS API 전환) 이미 달성 ⑤ 부채 1건만으로 새 MVP를 잡진 않지만 부채 묶음으로 사이클 무게 충분

### ★ MVP4(성능·안정성·관측성) 신규 사이클 권장 항목

M3 검증 중 발견된 부채 5건은 단일 성능/안정성 테마로 묶여, 다음 사이클로 분리하는 게 적절. M3 머지 + M4(Apple Login) 후 MVP4로 신규 마일스톤 기획 권장.

| # | 항목 | 영역 | 심각도 |
|---|---|---|---|
| M4-1 | `/api/me/summarize` N개 sequential → 60s timeout | server | 🔴 사용자 차단 (10건 이상 누적 시) |
| M4-2 | NSURLError -999 cancelled (Detail 진입 시 Task cancel) | iOS | 🟡 산발적 |
| M4-3 | 태그 stale (구독 변경 후 "전체" 탭에 잔류) | server | 🟡 정합성 |
| M4-4 | LLM 안정성 (reasoning hotfix 후에도 3/20 실패) | server | 🟡 |
| M4-5 | observability (server logger 부분 기록, 클라이언트 metrics 부재) | server + iOS | 🟡 디버깅 비용 |

**근거 (왜 M3에서 안 잡고 MVP4로 이월하는가)**:
- 진짜 fix는 server 아키텍처 결정 필요 (batch / async / pagination 중 선택)
- 본 worktree는 SESSION_SCOPE로 server/ read-only
- iOS-only 미봉책(timeout 180s)은 N≥40에서 또 깨짐 → 진짜 fix 아님

## 의사결정 인사이트 (왜)

### 왜 Profile.email을 단순 제거했나
사전 grep으로 UI 사용처 0건 확인 (5분 작업). SPEC FROZEN을 깨면 M2 탭 hotfix가 필요하니 SPEC 그대로 가는 게 격리 원칙과 일치. ProfilePort 권장은 informational이고 M3 메인 문서가 binding.

### 왜 SupabaseAuthAdapter를 그대로 두고 ProfileAPI helper로 분리했나
ProfilePort 신설(Q2=B)은 M3 메인 문서 동결 상태에서 ST 분해 변경에 해당. 책임 섞임 부채는 인정하되, 단위 테스트 가능성을 확보하기 위해 helper만 분리. SupabaseClient 의존 분리로 Mock 없이 ProfileAPI 단독 테스트 가능.

### 왜 MockURLProtocol을 호스트 기반으로 고쳤나
3개 suite 병렬 충돌이 단순한 race가 아니라 process-global static handler 디자인의 본질적 문제. 호스트별 라우팅으로 격리하면 향후 ST 추가 어댑터 테스트도 동일 패턴 재사용 가능. 일회성 fix가 아닌 패턴 정립.

### 왜 LoadingPhase enum 도입에 derived property를 썼나
4 Bool을 한 번에 갈아엎으면 cascade가 큼 (뷰 + 테스트 모두 isLoading/isCollecting 직접 참조). derived property로 인터페이스 호환을 유지하면 117개 기존 테스트가 무수정 통과 → 회귀 위험 0. 점진적 전환이 가능. "큰 변경은 작은 단계로"(workflow.md §원칙).

### 왜 시뮬레이터 실 서버 검증을 미뤘나
Q4=C는 "M2 탭 진행 상태에 따라 결정"이 정해진 답이었음. 본 worktree에서 단독으로 결정하면 M2 탭 작업과 충돌 가능. 머지 시점에 사용자/메인 탭이 함께 결정하는 게 안전.
