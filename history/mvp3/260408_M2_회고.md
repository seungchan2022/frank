## 옵션 B 전환 사유 (ST-4 → ST-7 재작업)

### 발견 과정
1. ST-4 1차 구현 시 `@supabase/ssr` 표준 패턴 그대로 적용 (`createBrowserClient` + `cookies` 어댑터)
2. ST-7 검증 단계에서 `Playwright context.cookies()`로 `sb-*-auth-token` 메타정보 점검
3. **결과**: `httpOnly: false`, `sameSite: Lax` ✅, `secure: false` (localhost dev)

### 원인
- `createBrowserClient`는 client-side JS가 `document.cookie`로 cookies를 set
- 브라우저 API 한계: **JS가 set하는 cookie는 `HttpOnly` 플래그 적용 불가** (HttpOnly는 server response의 `Set-Cookie` 헤더에서만 가능)
- `@supabase/ssr` v0.10의 client-side 패턴은 구조적으로 httpOnly를 보장 못 함

### 보완 (옵션 B)
ST-4 deliverable인 "httpOnly 쿠키 세션"을 충족하기 위해 SvelteKit form action 기반 server-side auth로 재구성:

| 변경 | 파일 | 효과 |
|---|---|---|
| `setAll` 강제 옵션 | `hooks.server.ts` | `httpOnly: true, sameSite: 'lax'` 강제 |
| signin/signup form action | `routes/login/+page.server.ts` (신설) | server에서 supabase 호출 → Set-Cookie 헤더로 cookie set |
| signout form action | `routes/logout/+page.server.ts` (신설) | server에서 signOut → 만료 cookie 헤더 |
| Header 변환 | `lib/components/Header.svelte` | `<form method="POST" action="/logout">` |
| login svelte 변환 | `routes/login/+page.svelte` | client `signInWithEmail` → form + use:enhance |
| root server redirect | `routes/+page.server.ts` (신설) | session 유무로 /login or /feed 분기 |
| 전역 가드 | `routes/+layout.server.ts` | 미인증 + 비공개 경로 → /login |
| 토큰 주입 패턴 | `lib/api/realClient.ts` | client supabase 의존 제거 → `setRealClientToken(token)` |
| layout 동기화 | `routes/+layout.svelte` | `$effect`로 `setAuth(data) + setRealClientToken(...)` |
| auth store SSR화 | `lib/stores/auth.svelte.ts` | 80줄 → 35줄. signIn/signUp/signOut 함수 전부 제거 |
| supabase.ts **삭제** | `lib/supabase.ts` | 사용처 0건, 강등 → 제거 |

### Trade-off
`access_token`을 `+layout.server.ts`에서 `page.data.session`으로 client에 전달 (httpOnly cookie의 의미를 일부 약화). 이는 ST-5 SvelteKit endpoint proxy 제거와 양립하기 위한 절충이며, 진정한 zero-token-on-client 패턴은 endpoint proxy 부활이 필요하므로 별도 마일스톤(M4 또는 M5)으로 분리.

## 다음 마일스톤 (M4 Apple 로그인) 인계 사항

### 코드 자산
- **httpOnly 쿠키 세션 + server form action 패턴 확립** — Apple OAuth flow도 동일하게 `routes/login/apple/+page.server.ts` 같은 server action으로 처리하면 토큰이 client에 노출되지 않음
- **`hooks.server.ts`의 `safeGetSession()`** — `getSession()` + `getUser()` 이중 검증 패턴이 OAuth callback 후에도 동일하게 동작
- **`+layout.server.ts` 전역 가드** — 새 OAuth 콜백 경로(`/auth/callback` 등)는 `PUBLIC_PATHS` set에 추가
- **`realClient.ts`의 `setRealClientToken` 패턴** — Apple 로그인 후에도 `+layout.svelte`의 `$effect`가 자동으로 토큰 주입

### 미해결 부채 (옵션 B trade-off + M2 범위 외 후속)
1. **`access_token`의 page.data 노출** — 진정한 zero-token-on-client는 SvelteKit endpoint proxy 부활이 필요. M4에서 다룰지 별도 마일스톤으로 분리할지 결정 필요
2. **태그 stale article 잔류 (260408 검증 발견)** — `user_tags` 비활성 시 articles 행 잔류. server SQL 변경 필요 (옵션 B: INNER JOIN user_tags). M2 머지 후 server/ 마일스톤에서 처리 예정
3. **Apple Sign-In on Web** — Supabase Auth의 OAuth provider 설정 + Apple Service ID + return URL 등록 필요
4. **계정 통합** — Apple 로그인 시 기존 email 계정과 자동 link/conflict 처리 정책 결정 필요 (Supabase Auth `linkIdentity` API)
5. **iOS App과 토큰 공유** — iOS Apple Sign-In은 `ASAuthorizationAppleIDProvider` 사용. 웹 세션과 별도 흐름이지만 Supabase 사용자 ID는 동일
