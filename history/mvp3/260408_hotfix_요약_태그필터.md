## 발견된 두 버그

### 이슈 #1 — POST /api/me/summarize 가 0건만 처리 ✅ 해결

**진단 과정** (260408 16:48~17:00 메인 탭)

1. DB 사전 확인 — 가설 2(이미 요약됨) 배제: `summary IS NULL` 48/48
2. `RUST_LOG=debug PORT=8081 cargo run | tee /tmp/frank-server.log`로 재기동
3. 사용자 웹 클릭 → 로그 확인:
   ```
   OpenRouter API error (400 Bad Request):
   {"error":{"message":"Reasoning is mandatory for this endpoint and cannot be disabled."}}
   ```
4. 원인 확정: `openrouter.rs:89` `reasoning: { effort: "none" }`. MiniMax M2.5는 reasoning 강제 모델이라 disable 거부 → 15건 article 전부 동일 400으로 silent fail → `summarized: 0`

**fix**: `effort:"none"` → `exclude:true`. 모델은 추론 수행하되 응답에서 reasoning trace만 제외하는 방식. mandatory 모델에서도 안전.

**테스트**: `request_uses_reasoning_exclude` wiremock 회귀 추가 (요청 body에 `reasoning.exclude=true` 검증). 기존 11개 LLM 테스트 + 신규 1 = 12개 통과. 전체 158 cargo test 통과.

**실서버 검증**: fix 적용 후 사용자 재클릭 → DB 상태 `2aaf03aa` 계정 20건 중 17건 요약 성공. 이슈 #1 종료.

**참고**: 남은 3건 null은 `parse failed: invalid type: null` (LLM이 일부 article에 빈 content 반환). summary_service에서 이미 catch 후 다음 article 처리하는 구조이므로 사용자 영향 없음. content를 `Option<String>`으로 만드는 robustness 개선은 추후 별도 작업으로 이관.

### 이슈 #2 — 태그 변경 후 "전체" 탭에 이전 키워드 article 잔류 → **별도 server 마일스톤으로 분리**

**1차 결정 (260408 오후)**: hotfix 스코프에서 제외. M2(웹) 워크트리에서 server 변경 + 실 API 꽂아 검증하는 흐름이 자연스럽다고 판단.

**2차 결정 (260408 후반, 사용자 지시)**: M2 워크트리에서도 처리하지 않음. 사유:
- 본 작업은 server/DB 영역(SQL JOIN 또는 cascade delete)이고, M2 워크트리의 격리 원칙은 "web/만 수정"
- M2의 핵심 deliverable(API 통일 + httpOnly 쿠키 세션)은 이미 모두 충족됨. 본 이슈는 M2 deliverable에 포함되지 않은 별개 버그
- M2/M3가 각각 web ↔ iOS API 통일에 집중해야 PR 단위가 명확해짐
- "테이블 수정은 서버나 DB쪽 이슈니까 이후에 수정하자" — 사용자

→ M2/M3 둘 다 main 머지 완료 후, 별도 server/ 마일스톤(예: `260409_server_tag_cascade.md`)에서 옵션 B(domain port에 `restrict_to_user_tags` 옵션 추가 + postgres_db/fake_db에 INNER JOIN 시맨틱) 처리. iOS/web 양 클라이언트는 server 변경만 받으면 자동 정합화됨.

**M2 워크트리 처리 결과 (260408 Playwright 재검증)**:
- 이슈 #1 (hotfix `5d0adc7`): "요약 대기 중..." 2건 → 한국어 요약 정상 생성 ✅
- 이슈 #2: 재현됨 (user_tags={AI/ML}로 변경 후 "전체" 탭에 프로덕트 article 8건 잔류). 본 worktree에서는 처리하지 않고 후속 부채로 기록만 함 (`260406_MVP3_M2_웹전환.md`, `260408_M2_회고.md` 참고)

---

(이하 옵션 분석은 M2에서 참고용으로 보존)


- `saveMyTags`/`updateMyTags`는 `user_tags` 테이블만 교체
- `articles` 테이블은 cascade 없음
- `GET /api/me/articles` (필터 없음)는 사용자의 모든 articles 반환
- 결과: user_tags에 더 이상 없는 키워드(`데이터 사이언스` 등)의 article이 "전체"에 그대로 노출

DB 확인 (Tab1 보고): user_tags 2개(`AI/ML`, `프로덕트`)인데 articles 15건 중 3건이 `search_query="데이터 사이언스"`로 포함됨.

**해결 방향: 옵션 B (server INNER JOIN)**

3개 옵션 중:
- A (cascade delete) — destructive, 키워드 swap 시 아카이브 손실
- B (server에서 user_tags와 INNER JOIN으로 필터) ✅
- C (client filter) — web만 고치고 iOS에서 똑같이 재발견 → 싱크 원칙 위반

옵션 B는 양 클라이언트가 자동 정합화되며, 데이터 보존 + 키워드 재추가 시 자동 복구 + 단일 진실 원천(server) 유지.
