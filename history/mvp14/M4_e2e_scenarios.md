# M4: E2E 시나리오 완성 + 검증

> 프로젝트: Frank MVP14
> 상태: 대기
> 예상 기간: 1주
> 의존성: M2 (버그 수정), M3 (UX 개선)

## 목표

M1에서 구축한 E2E 인프라 위에, M2+M3에서 수정/개선한 항목을 기준으로 실제 시나리오를 작성하고 검증한다. 이후 MVP에서 수동 QA 없이 회귀를 잡을 수 있는 완성된 E2E 세트를 만든다.

> M1에서 환경 세팅만 완료한 상태. 시나리오 내용은 M2+M3 완료 후 올바른 동작 기준으로 작성한다.

## 성공 기준 (Definition of Done)

- [ ] 웹: Playwright W-01~W-03 시나리오 작성 + 실행 통과
- [ ] iOS: XCUITest I-01~I-05 시나리오 작성 (기존 파일 확장 + 신규) + 실행 통과
- [ ] M2+M3 수정 항목 ↔ 시나리오 커버리지 매핑 표 완성
- [ ] `/e2e` 스킬 파일 시나리오 내용으로 채우기 완료
- [ ] E2E 실행 결과 리포트 1건 생성 (`progress/kpi/YYMMDD_e2e_report.md`)
- [ ] 서버·웹·iOS 기존 단위/통합 테스트 전체 통과 유지

## TDD 선행 정의 (각 시나리오별 실패 케이스 + 경계 케이스)

| 시나리오 | 먼저 실패해야 할 케이스 | 경계 케이스 |
|---------|----------------------|------------|
| W-01 | 요약 카드가 노출되지 않으면 실패 / 재시도 후에도 실패 상태 유지 시 실패 | 네트워크 에러 후 재시도 버튼 클릭 → 성공 전환 |
| W-02 | 탭 전환 후 이전 태그 기사가 그대로면 실패 | 마지막 탭 → 첫 탭으로 순환 전환 |
| W-03 | 좋아요 탭 후 디테일 페이지로 이동 시 실패 / 좋아요 상태 변경 없으면 실패 | 이미 좋아요 상태에서 재탭 → 취소 |
| I-02 | 스와이프 후 태그 목록이 변경 안 되면 실패 | 첫 번째 태그에서 왼쪽 스와이프 (경계) |
| I-03 | 요약 후 버튼이 가려져서 탭 불가 시 실패 | 긴 요약 텍스트에서 버튼 가시성 확인 |
| I-04 | refresh 후 기사 목록이 동일하면 실패 | 빈 피드에서 pull-to-refresh |
| I-05 | 좋아요 탭 후 디테일 이동 시 실패 | 연속 빠른 탭 → 중복 요청 없음 |

## E2E 격리 전략

M1에서 확립한 전략을 시나리오 실행에 적용한다.

| 항목 | 전략 |
|------|------|
| 계정 상태 | 시나리오 실행 전 즐겨찾기·오답·요약 캐시 초기화 (API DELETE 또는 Supabase SQL) |
| 세션 | Playwright: 로그인 후 storage state 저장 → 재사용 |
| iOS | XCTest setUp()에서 API 클리어 엔드포인트 호출 또는 launchArguments `--reset-data` |
| 독립성 | 각 시나리오는 setup → 실행 → teardown 구조 |
