핵심 의사결정과 그 이유

챕터 A · 결정 1

DEBT-04 — NavigationLink 제거 → ZStack + onTapGesture

상황: iOS 피드 카드에서 좋아요 버튼을 탭하면 디테일 화면으로 같이 이동. NavigationLink가 카드 전체를 덮고 있어 좋아요 Button의 탭이 번져 들어갔다.

A. simultaneousGesture로 좋아요 탭 가로채기 — 코드 적음, SwiftUI 버전 의존, 안정성 의문
B. ✓ NavigationLink 제거 → ZStack + onTapGesture — 좋아요 Button이 최상위 레이어. 제스처 충돌 원천 봉쇄

왜: simultaneousGesture는 SwiftUI 버전마다 거동 차이가 크고, "버튼 단독 탭이 정말 단독"을 보장하기 어렵다. 제스처 우선순위에 의존하기보다 레이어 구조로 푸는 게 인지 부담이 적다.

UX 버그는 "기능이 안 됨"이 아니라 "사용자 의도가 안 통함"이다. 후자를 잡으려면 코드 변화량을 두려워하면 안 된다.
챕터 A · 결정 2

DEBT-05 — swipeActions / DragGesture 폐기 → ScrollView(.viewAligned) + contextMenu

상황: TabView 기반 태그 스와이프가 빠르게 밀면 2~3페이지를 건너뜀. swipeActions는 좋아요·삭제 버튼과 시각 충돌.

A. TabView 그대로 + 속도 제한 코드 — 임의 임계값, 디바이스마다 다름
B. ✓ ScrollView + viewAligned(limitBehavior: .always) — OS-level 1페이지 스냅 보장

왜: viewAligned의 .always는 페이지당 한 번 스냅을 OS 차원에서 보장한다. 직접 짜는 임계값 코드보다 신뢰도가 높다. swipeActions는 사용 빈도(삭제·해제) 낮은데 시각 면적은 컸다 — contextMenu가 더 맞다.

"직접 제스처 코드 vs OS 동작에 위임"이 둘 다 가능할 때, 후자가 늘 안전하다. 챕터 A 전체를 관통하는 위임 사고의 작은 사례.

챕터 B · 결정 4 ★

5↔9단계 모순 해소 — 강제 룰 SSOT 통일

상황: rules/0_CODEX_RULES.md §3은 5단계, §9는 9단계가 각각 "유일한 강제 워크플로우"라고 자칭. 같은 파일 안에 모순. LLM은 그때그때 한쪽 인용해 자기합리화 가능 — 강제력 0.

결정:

  • 5단계는 Codex 호출 시 자동 작동하는 내부 사고 모델로 위치 정정
  • 본 저장소 강제 워크플로우는 9단계(step-1~step-9)로 단일화
  • 강제 룰 SSOT를 .claude/skills/workflow/SKILL.md로 통일. 나머지 파일은 SSOT 링크만

왜: 강제 룰이 두 곳에 있으면 어느 쪽이 진짜인지 모호해진다. SSOT를 한 곳으로 묶고 나머지는 링크만 두면, 모순 자체가 발생할 수 없는 구조가 된다.

분산된 룰은 그 자체로 자기합리화의 빌미가 된다. SSOT는 "참조 편의"가 아니라 "모순 차단"이다.
챕터 B · 결정 5 ★

메모리/룰 → hook + state file 전환

상황: 인터뷰 망각의 본질은 LLM이 자기 컨텍스트 안에서 자기를 감시 못 한다는 것. "Q2/4 미완료" 같은 상태를 컨텍스트 텍스트로만 추적하면, 한 질문 깊이 토론하는 순간 그게 most recent로 남고 "남은 질문 스택"은 attention dilution으로 밀려난다.

A. 메모리/룰북 추가 보강 — 지금까지 4번 시도. 같은 패턴 반복
B. ✓ state file 외부화 + hook 자동 주입/차단 — LLM이 자기를 감시할 필요 없음

왜: A로 4번 이상 시도했지만 같은 패턴이 반복됐다. "LLM은 자기 컨텍스트에서 자기를 강제 못 한다"는 본질을 인정해야 했다.

같은 문제가 두 번 반복되면 문서가 아니라 기계로 막아야 한다 — 머리로는 알았지만 오늘에야 실행했다.
챕터 B · 결정 6 ★

hook + 안전장치 6개를 처음부터 동시에 설계

상황: PreToolUse hook은 잘못 만들면 작업 자체가 막혀서 진척이 0이 된다. 보통은 단순 시작 후 보완이지만, hook은 그 사이클이 안 도는 도구.

결정: 안전장치 6개 통합.

  • S1 matcher 좁히기 (Edit|Write만)
  • S2 활성 step 체크 (none/empty 시 통과)
  • S3 step 전환 시 자동 cleanup
  • S4 메타태스크 면제 (.claude/, scripts/, rules/, progress/meta/, history/, 메모류)
  • S5 bypass 환경변수 (WORKFLOW_BYPASS=1)
  • S6 자동 expire (24h 넘으면 stale 무시, BSD date 호환)

왜: "리뷰 반영 + 실제 사용 후 재수정" 사이클을 빠르게 돌리려면, 첫 배치에서 작업이 막히지 않을 수준의 안전망은 깔려있어야 한다.

보수와 공격은 반대가 아니다 — 보수적 안전망 위에서야 공격적 반복이 가능하다.

설계 시각 — 하네스 엔지니어링

오늘의 두 챕터는 표면적으로 다르다.

  • 챕터 A: 사용성 개선 + UITest/Playwright 자동화 추가 → Claude에게 더 위임
  • 챕터 B: hook + state file로 워크플로우 강제 → Claude를 강제로 묶음

챕터 A에서 UITest 자동화를 깔면서 "내가 매번 시뮬레이터 켜는 것보다 위임이 정확하다"는 인식이 굳어졌다. 단위 테스트는 "내가 정한 로직이 작동하나"까지지만, UITest는 "사용자가 실제로 쓸 때 동작하나"를 본다. 후자는 내가 직접 하면 엣지 케이스가 빈다 — Claude가 더 잘한다.

그런데 "더 위임"의 다른 면은 "방향이 어긋날 때 큰 비용". 그래서 챕터 B의 hook이 따라온 것이다. 위임 자유도가 커질수록 레일이 더 필요해진다.

이건 오늘 갑자기 생긴 시각이 아니다. 이전부터 enforce_workflow, branch_naming, deploy_script 같은 메모리들이 같은 방향으로 누적돼왔다. 오늘은 누적의 분기점이라기보다 선언된 날이다.

인사이트 & 피드백

1. 도구를 손보러 시작한 게 아니라, 일하다가 손보게 된 것

챕터 B는 미리 잡힌 메타태스크가 아니었다. 챕터 A 진행 중 또 같은 문제(단계 누락)가 발생했고, 그 자리에서 "더는 문서로 안 된다"가 굳어졌다. 자정 전 32분 만에 Phase 1~3 끝.

다음에 비슷한 상황: "또 같은 문제다"가 들어오면, 그날 안에 도구 변경까지 가는 걸 기본 옵션으로. 미루지 말 것.

2. SSOT는 편의가 아니라 모순 차단 장치

5↔9단계 모순은 같은 파일 안에 있었는데 한 번도 못 봤다. 분산된 룰은 자기합리화의 빌미가 된다. 강제 룰은 한 곳에만.

3. "LLM은 자기 컨텍스트 안에서 자기를 강제 못 한다"

가장 큰 깨달음. 메모리도 룰북도 같은 컨텍스트 안의 텍스트라 attention dilution을 못 막는다. 외부 state file + hook으로 매 turn 첫 줄에 자동 주입해야 한다. 컨텍스트 외부의 강제만이 진짜 강제다.

4. 보수와 공격은 반대가 아니다

PreToolUse hook을 만들 때 안전장치 6개를 같이 깐 건 보수적 결정. 하지만 이 보수 위에서 "리뷰 반영 → 실사용 → 재수정" 사이클이 빠르게 돈다. 안전망은 속도를 막지 않는다 — 오히려 가능하게 한다.

그림자 — 하네스 검증 부재

오늘의 의구심

Playwright와 UITest 자동화가 제대로 깔렸는지 본인이 검증을 못 했다.

단위 테스트는 "로직 작동 확인"까지. 오늘 한 사용성 테스트는 "실제 동작 확인"인데, 본인이 시뮬레이터를 직접 움직여서 매칭해보지 않았다. 앞으로 마일스톤마다 이 테스트 스킬을 돌릴 텐데 — 만약 시나리오 자체가 어긋나 있다면, 통과해도 통과한 게 아니다.

이게 단순한 "자동화 신뢰도" 문제는 아닌 것 같다. 한 단계 들어가면 "하네스를 깔았는데 하네스 자체를 누가 검증하지?" 라는 메타 위임의 빈자리다. Claude에게 위임은 했는데, 그 위임이 잘 됐는지 메타 검증은 빠져있다. 다음 MVP에서 한 번은 마주칠 질문이다.

배운 것

  • SwiftUI: NavigationLink 제거 + ZStack/onTapGesture로 제스처 충돌 봉쇄가 simultaneousGesture보다 안정적
  • SwiftUI: ScrollView + viewAligned(limitBehavior: .always)는 TabView 고속 스와이프 다중 스킵의 OS-level 해법
  • Playwright: MutationObserver는 클릭 전에 전역 설치해야 첫 mutation을 놓치지 않음
  • XCUITest: RefreshControl은 시뮬레이터 hit 확률 100% 아님 — retry는 현실 수용
  • Claude Code 하네스: PreToolUse hook은 PostToolUse·UserPromptSubmit과 다르게 물리적 차단(exit 2)이 가능
  • 프로세스: 같은 문제가 두 번 반복되면 기계(hook/deny/스킬)로 막는다 — 오늘 처음 실행

느낀 점

뿌듯한 점: 챕터 B를 32분에 끝낸 거. 미루지 않고 그날 안에 손본 게 컸다. 자정 직전 마지막 커밋 들어갈 때 후련함.

힘들었던 점: 챕터 A에서 UITest/Playwright 자동화를 깔고 나서 "이게 제대로 깔린 게 맞나?" 라는 의구심이 남았다. 위임을 더 했는데 위임이 잘 됐는지는 본인이 못 본 채로 끝났다.

재밌었던 점: 하네스 엔지니어링이라는 표현이 회고 도중에 입에서 나왔는데, 그게 오늘의 두 챕터를 한 큐에 묶었다. 이전부터 해온 일에 이름이 붙은 느낌.
