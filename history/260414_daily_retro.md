## 의사결정 기록

### 1. keyword_weights 태그별 분리 — tag_id 컬럼 추가 (안 2 선택)

**결정**: `user_keyword_weights` 테이블에 `tag_id` 컬럼을 추가해 태그별로 키워드를 분리 저장.

**배경**: 기존엔 좋아요 전체에서 키워드를 추출해 하나의 풀에 합쳤다. 사용자가 "AI" 기사에 좋아요를 많이 하면 "mobile", "swift" 태그 검색에도 AI 관련 키워드가 boost되는 cross-tag 오염이 발생했다.

**안 1 vs 안 2**:
- 안 1: 태그 변경 시 전체 키워드 초기화 — 단순하지만 기존 개인화 데이터 소실
- 안 2: tag_id 컬럼 추가 — DB 마이그레이션 필요하지만 이전 가중치 보존 + 태그 복귀 시 재활용

**인사이트**: "단순한 해결" vs "데이터 보존" 트레이드오프에서 사용자가 축적한 개인화 데이터를 날리는 건 명백한 손실이다. 마이그레이션 비용이 있어도 안 2가 맞다.

### 2. 요약 실패 에러 코드 분리 — 422 / 503 / 408 원인별 분리

**결정**: 기존 500으로 뭉뚱그리던 요약 실패를 원인별로 분리.
- 422: 크롤링 실패 (기사 본문 없음, 접근 불가)
- 503: LLM 장애 (OpenRouter/Groq API 오류)
- 408: 타임아웃

**이유**: 클라이언트가 500만 받으면 "서버 에러"라는 동일 메시지밖에 표시할 수 없다. 원인을 알면 "이 기사는 요약이 어렵습니다"(크롤링 실패)와 "AI 서비스에 일시적 문제가 있습니다"(LLM 장애)를 구분해 표시할 수 있다.

**인사이트**: HTTP 상태 코드는 클라이언트와의 계약이다. 500으로 모든 에러를 뭉개면 클라이언트는 맥락을 잃는다. 원인 분류가 가능한 에러는 코드 수준에서 분리해야 한다.

### 3. 좋아요 요청 tag_id 누락 — fire-and-forget silent 버그

**결정**: 웹/iOS 좋아요 요청에 `tag_id`를 포함하도록 수정.

**발견 경로**: M2 버그 수정 작업 중 서버 코드를 보니 `keyword_weights` 저장 시 `tag_id NOT NULL` 조건이 있었다. 클라이언트가 `tag_id`를 보내지 않으면 조용히 실패하고 있었던 것.

**왜 늦게 발견됐나**: 좋아요 동작 자체는 성공했다(hearts 테이블은 정상 저장). keyword_weights 저장만 실패하는 구조라서 UI에서 아무 에러도 보이지 않았다. fire-and-forget 패턴의 전형적 함정.

**인사이트**: 부수 효과(키워드 추출, 통계 업데이트 등)를 fire-and-forget으로 처리하면 실패가 무음으로 지나간다. 중요도에 따라 실패 로깅 또는 알림 정책을 명시적으로 정해야 한다.

### 4. FavoritesFeature.markQuizCompleted(url:) — QuizFeature 대신 FavoritesFeature에서 처리

**결정**: 퀴즈 완료 후 즐겨찾기 배지 갱신을 QuizFeature가 아닌 FavoritesFeature에서 담당.

**배경**: 퀴즈 완료 후 즐겨찾기 화면으로 돌아오면 배지가 표시되지 않는 문제. QuizFeature에서 완료 신호를 보내고 FavoritesFeature가 수신해 갱신하는 방식이 자연스러워 보였지만, iOS @Observable 불변 복사 갱신 특성상 QuizFeature가 보내는 신호가 stale 상태를 건드리는 타이밍 문제가 있었다.

**해결**: FavoritesFeature에 `markQuizCompleted(url:)` 메서드를 추가하고, 퀴즈 완료 시점에 직접 호출하는 방식으로 변경. 상태 소유자가 직접 갱신.

**인사이트**: iOS @Observable에서 상태 갱신은 "데이터를 소유한 Feature가 직접 처리"가 원칙이다. 다른 Feature에서 신호를 보내 갱신을 유발하는 방식은 타이밍 문제를 만든다.

### 5. 피드 TTL 캐시 구현 — Arc<Mutex<HashMap>> + Instant 기반 동시성 설계

**결정**: Redis 대신 Rust 인메모리 (`Arc<Mutex<HashMap>>` + `Instant`)로 TTL 캐시 구현. 5분 TTL, 최대 100 엔트리.

**설계 핵심**:
- 캐시 키: `"{user_id}:{sorted_tag_ids}"` — 사용자 격리 + 태그 조합 구분
- UUID prefix 기반으로 `invalidate_user()` 구현 가능
- 만료 엔트리 우선 제거 → 그래도 초과면 가장 오래된 것 제거 (oldest-first eviction)

**Redis 제외 이유**: 현재 사용자 규모에서 네트워크 왕복 비용이 캐시 이득보다 크다. 추후 다중 인스턴스 배포 시 Redis 전환 — 이는 FeedCachePort 교체만으로 가능하다.

**인사이트**: 포트/어댑터 패턴의 가치는 구현 교체 비용에 있다. Redis vs 인메모리 결정이 "지금은 인메모리, 나중에 Redis"로 미룰 수 있는 건 FeedCachePort가 경계를 만들어주기 때문이다.

### 6. NoopFeedCache — 기존 테스트 8곳 무수정 통과 보장

**결정**: `FeedCachePort` 구현체로 `NoopFeedCache`(항상 miss, 저장 무시)를 도입하고 테스트 픽스처에 주입.

**왜 필요했나**: `InMemoryFeedCache`를 기존 테스트에 주입하면 테스트 간 캐시 공유 문제가 발생한다. 캐시 없이 동작하는 Noop이 가장 안전하고 의미 있는 기본값.

**결과**: `make_test_state()`를 수정하지 않고 Noop 주입만으로 기존 테스트 8곳 전부 통과.

**인사이트**: 포트/어댑터의 실전 가치는 "기존 테스트를 건드리지 않고 새 구현체를 추가할 수 있다"는 데 있다. 이것이 TDD와 아키텍처가 교차하는 지점이다.

### 7. Cache-Control: no-cache 헤더 — 캐시 우회 전략

**결정**: 수동 새로고침(웹) / pull-to-refresh(iOS) 시 `Cache-Control: no-cache` 헤더를 붙여 서버 캐시를 우회.

**문제**: 피드 TTL 캐시 도입 후 "새로고침인데 캐시 HIT → 오래된 데이터 표시" 문제가 생겼다.

**해결 방식**: 서버가 `Cache-Control: no-cache` 헤더를 감지하면 캐시를 우회하고 신선한 데이터를 가져온다. 단, 그 결과는 다시 캐시에 저장 (no-cache 요청의 결과가 가장 최신 데이터이므로 다음 일반 요청에서 재활용).

**인사이트**: "캐시 우회"와 "캐시 무효화"는 다르다. 우회는 이번 요청만 신선하게 가져오는 것, 무효화는 캐시 전체를 버리는 것. 사용자 관점에서는 새로고침 = 우회로 충분하다.

## Keep / Problem / Surprise

### Keep
- **NoopFeedCache 패턴**: 포트/어댑터가 "기존 테스트 무수정 통과"를 실제로 증명한 날. 인터페이스 먼저 설계하는 습관이 이런 순간을 만든다.
- **에러 코드 원인별 분리**: 클라이언트가 맥락을 잃지 않게 하는 서버 설계 원칙. 앞으로 새 에러 케이스마다 적용할 기준이 됐다.

### Problem
- **fire-and-forget silent 실패**: tag_id 누락 버그가 오래 잠들어 있었던 건 부수 효과 실패를 모니터링하는 체계가 없었기 때문. 중요한 fire-and-forget은 실패 로깅이 필수다.
- **캐시 키 설계를 나중에 바꾸기 어렵다**: `{user_id}:{sorted_tag_ids}` 포맷은 지금은 맞지만, 태그 이외의 필터(날짜 범위 등)가 추가되면 키 구조를 전면 수정해야 한다. 지금 시점에 문서화해둬야 한다.

### Surprise
- **인메모리 캐시가 이렇게 단순할 수 있다**: `Arc<Mutex<HashMap<String, (Vec<Article>, Instant)>>>` 한 줄로 동시성 안전 TTL 캐시가 완성됐다. Rust의 타입 시스템이 복잡한 동시성 보일러플레이트를 대신한다.
- **MVP10이 정말 버그만 잡는 날이 됐다**: 신규 기능 0개. 실사용에서 발견된 버그 5건 + 성능 개선 1건. 이런 날이 필요하다는 걸 다시 확인했다.
