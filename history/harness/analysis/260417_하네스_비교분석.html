Analysis Note · 2026-04-17

하네스 엔지니어링 비교 분석 노트

Anthropic / OpenAI 하네스 가이드 × 프로젝트 frank
본인 학습용 + 나중에 꺼내 쓸 매뉴얼.

프로젝트는 하네스 원칙을 문서로는 잘 반영했다. 서브 룰북, 에이전트 역할 분리, /step-7 리팩토링 3R + 리뷰 3R 같은 장치가 이미 갖춰져 있다. 남은 과제는 문서화된 원칙을 자동 실행 가능한 기계 장치로 격상하는 방향이다. 즉 "해야 한다"고 쓴 것을 "어기면 막는다 / 사람 없이도 굴러간다"로 바꾸는 일이다.

1. 두 문서 핵심 원리

왜 이 두 문서인가

둘 다 2026년 초에 AI 에이전트가 혼자서 오래(수 시간~수 개월) 일하게 하는 법을 다룬 실전 보고서다. Anthropic은 Claude 연구를 통해 "왜 AI가 긴 작업에서 실패하는지"를 4가지 유형으로 정리했고, OpenAI는 "5개월 동안 코드 한 줄도 직접 쓰지 않고 AI로만 100만 줄 프로그램을 완성한" 실제 실험 결과를 공개했다. 두 문서가 강조하는 원리는 다르지만 공통 철학은 하나: "AI를 더 똑똑하게 만들기보다 AI가 일할 환경을 잘 만드는 것이 중요하다".

1.1 Anthropic — 실패 4유형 × 해결

AI가 긴 작업에서 실패하는 4가지 방식과 해결책.

실패해결
① 대충 끝내려 함체크리스트
② 다 했다고 거짓말체크리스트 + 다른 AI 검사
③ 원래 목표 잊음인수인계서
④ 자기 답안 자기 채점다른 AI 검사 + 5~15회 반복
원문 용어 매핑 (헷갈림 방지)
  • 체크리스트 = Feature List = 기능 목록 = 피쳐 리스트
  • 다른 AI 검사 = 평가자 분리 = 다중 에이전트
  • 인수인계서 = 컨텍스트 리셋

②와 ④는 원인이 같다 — 둘 다 "자기 평가 편향". 그래서 같은 약(다른 AI 검사)이 통함.

1.2 OpenAI — 4대 원리 (100만줄/5개월 프로젝트)

3~7명이 5개월간 AI로만 100만 줄 프로그램을 만들면서 쌓은 교훈.

원리무엇인가왜 중요한가
① AGENTS.md (지도+도서관)100줄짜리 짧은 안내서 + 상세 내용은 별도 docs/안내서가 너무 길면 작업 집중 못 함. 얇은 가이드 + 필요 시 찾는 도서관 구조
② 기계적 규칙 강제6-layer 아키텍처를 린터가 자동 검사. 위반 시 거부 + "이렇게 고치면 됨" 힌트까지AI는 상황 따라 다르게 판단함. "해야 한다"만으론 안 지켜짐. 기계가 막아야 함
③ 피드백 루프AI가 Chrome DevTools로 자기 화면 직접 보고 수정. 성능 기준치(예: "0.8초 기동") 통과할 때까지 자동 반복사람이 매번 확인하면 속도 안 남. AI가 자기 결과를 자기가 검증하는 자율 루프 필요
④ 가비지 컬렉션AI가 매일 낡은 코드·문서 청소 PR을 자동 올림. 1분 내 자동 반영쓰레기 쌓이면 시스템이 느려짐. 큰 리모델링 대신 매일 조금씩 청소
엔지니어의 새 역할: OpenAI가 말한 "요리사 → 주방 설계자". 직접 코드를 쓰지 않고 AI가 일할 환경을 설계하고, 실수가 나오면 환경에 박제하여 재발 방지한다.
2. 채점 루브릭

왜 루브릭이 필요한가

이 노트의 스코어(★, n/10)가 "왜 그 점수인지" 설명 없으면 주관적 느낌으로 읽힌다. 루브릭을 먼저 정해두면 같은 기준으로 재채점 가능하고, 나중에 봐도 "내가 왜 이렇게 매겼는지" 추적 가능하다.

2.1 정합도 (★ 기준)

등급정의언제 이 등급인가
★★★★★문서 + 기계 강제 양쪽원칙이 룰북에 있고 hook/lint/CI가 자동으로 막음
★★★★☆문서화 + 일부 기계 강제원칙은 문서에 있으나 강제 장치가 일부만 있거나 약함
★★★☆☆문서만, 강제 없음문서에 있지만 어겨도 알림 없음
★★☆☆☆부분 구현원칙 자체가 일부만 반영됨
★☆☆☆☆언급 수준이름만 있고 실질 구현 없음

2.2 스코어 (n/10)

점수의미해석
9~10업계 모범 사례이 영역에선 더 개선할 여지 적음
7~8실무 적용 충분핵심은 갖춰짐, 일부 자동화 여지
5~6기계 강제 부족문서·운영 수준은 있으나 어기면 막는 장치 약함
3~4실천 산발적원칙은 인지하나 일관성 부족
1~2부재해당 장치 없음
한계: 같은 항목을 다시 채점하면 1~2점 변동 가능. 특히 7과 6의 경계가 주관적이다. 따라서 절대 등급이 아닌 상대적 진단으로만 사용. "X는 Y보다 부족하다"가 핵심이지 "X는 객관적으로 6점이다"가 아니다.

4. Strong Fit (정합된 부분)

이 섹션의 의미

두 문서가 강조하는 원칙 중 프로젝트가 이미 잘 반영하고 있는 부분을 정리한다. ★5가 아닌 항목엔 "감점 사유"를 ⚠로 표시하되, 수정할 필요가 있는지를 명확히 한다 — 감점 사유가 있다고 다 고쳐야 하는 건 아니다.

⚠ 행동 권고 3종
  • [넘어가기] 감점 사유이지만 지금 수정 불필요 (이유 명시)
  • [선택적] 여유 있을 때 개선하면 ★5 가능
  • [Gap 이관] 별도 Gap 항목에서 관리. Strong Fit 자체는 유지
4.1 AGENTS.md = 지도+도서관 (OpenAI ①)
★★★★☆

왜 강점인가: CLAUDE.md(173줄)는 "지도" 역할, rules/sub/*.md 9개는 상세 "도서관" 역할을 맡는다. 언어 스코프 룰은 rust/swift/svelte/python-ml 작업 시 자동 로드되어 컨텍스트 효율이 높다. OpenAI가 100만줄 프로젝트에서 도입한 구조와 거의 동일하다.

⚠ 감점 사유: CLAUDE.md가 OpenAI 권장 "100줄"을 초과(173줄). [넘어가기] — OpenAI 기준은 100만줄 대형 프로젝트에서 컨텍스트 과부하를 막기 위한 수치. 현재는 MVP 규모라 173줄이 오히려 적절한 밀도다. 세션 시작 지연이나 가독성 문제가 체감될 때 분할.
4.2 기계적 규칙 강제 (OpenAI ②)
★★★★★

왜 강점인가: 프로젝트는 두 계층의 hook과 deny 규칙으로 위험 명령·형식 위반·테스트 미통과를 기계적으로 차단한다.

계층어디역할
Git pre-commit hook.git/hooks/pre-commitmain 직접 커밋 차단, 테스트 미통과 차단
Git commit-msg hook.git/hooks/commit-msg커밋 메시지 형식 강제

PreCompact hooksettings.json컴팩션 전 대화 로그 자동 백업

"해야 한다"가 아니라 "어기면 막는" 수준이다. OpenAI가 강조한 "기계적 규칙 강제"의 상당 부분이 이미 실현됨.

참고: settings.json의 Stop hook은 이모지 출력만 하는 placeholder. 하지만 실질 검증은 Git hook이 담당하므로 공백 없음. §9에서 정리 예정.

4.3 다중 에이전트 · 평가자 분리 (Anthropic ③)
★★★★★

왜 강점인가: /debate 스킬이 Claude(Anthropic) + Codex(OpenAI) + Serena(MCP 심볼 분석) 3자 토론을 진행한다. 서로 다른 회사·다른 방법이라 편향이 부분적으로 독립되어 있다. 역할 에이전트 7종도 작성자와 평가자를 분리한다. 특히 /step-7은 Claude 작성 / Codex 리뷰 구조로 "자기 답안을 자기가 채점"하는 실패 유형 ④를 방지한다. MVP 단계 코드 리뷰 영역에서는 이 조합만으로 충분한 독립성이 확보된다.

4.4 반복적 개선 (Anthropic ④)
★★★★☆

왜 강점인가: /step-7 스킬이 리팩토링 3R(R1/R2/R3) + 리뷰 3R(보안/아키텍처/최종) 구조를 갖는다. Anthropic이 프론트엔드 디자인 작업에서 5~15회 반복 적용해 품질을 끌어올린 방식과 동일한 아이디어. 한 번에 완벽한 결과 대신 만들고→평가하고→다시 만들기 반복.

4.5 Discovery-First 탐색 60% (공통)
★★★★★

왜 강점인가: /milestone 스킬과 rules/sub/milestone.md에 "탐색과 브레인스토밍이 전체 시간의 60%"가 명시. "무엇을 만들지 함께 발견하는" 절차가 룰로 박혀 있다. 구현 바로 뛰어들지 않는 원칙은 Anthropic ① One-shotting 방지, OpenAI "의도 명확화" 철학과 연결된다.

4.6 TDD 문화 (OpenAI 검증 루프)
★★★★★

왜 강점인가: rules/0_CODEX_RULES.md §2.2에 "실패 테스트 먼저 → 구현 → 통과 확인" 순서가 강제. 90% 커버리지 기준도 명시. "구현 후 테스트 추가 금지"까지 포함되어 OpenAI의 "성능 기준치 통과해야 완료" 철학을 만족한다.

4.7 실수 → 룰 영속화 (OpenAI 철학)

4.9 컨텍스트 압축 보존 (Anthropic ①)
★★★★☆

왜 강점인가: settings.json의 compactPrompt에 보존할 5항목(CLAUDE.md 규칙 / 태스크 진행 / 수정 파일 / 피드백 / 실패 접근법)이 명시. PreCompact hook이 컴팩션 전 대화 로그를 백업한다. Anthropic이 "근무 교대 시 인수인계서" 비유로 설명한 장치의 상당 부분이 구현됨.

5. Gap 4개 (부족한 부분)

이 섹션의 의미

문서 원리 중 아직 부족한 부분을 5개로 정리한다.

중요한 원칙: 이 리스트는 "해야 할 일 목록"이 아니라 "실수가 실제로 나왔을 때 꺼내 쓸 매뉴얼"이다. 5개 중 몇 개는 영영 안 고쳐도 프로젝트가 잘 굴러갈 수 있다. OpenAI의 "실수가 나오면 박제"의 반대는 "실수 나오기 전에 예방적으로 다 설치"이고, 이건 과투자다.

7.1/10

해석

평균 7.1은 "실무 적용은 충분하지만 자동화 여지 있음" 구간의 상단이다. 특히 컨텍스트 관리(9)·문서 구조(8)·기계적 강제(8)·평가자 분리(8)가 강점이고, 가비지 컬렉션(5)·체크리스트(6)·피드백 루프(6)·실수→룰 승격(6)이 상대적 약점이다. 약점 영역은 모두 "문서화된 원칙을 자동 실행으로 격상"하면 1~2점 상승 가능하다.

7. 개선 추천 순위

이 순위의 원칙

  • Gap 리스트 = 매뉴얼이지 "해야 할 일 리스트"가 아니다
  • 실수가 실제로 나왔을 때 해당 Gap을 꺼내 쓰기 (OpenAI "실수 박제" 철학)
  • 4개 중 몇 개는 영영 안 고쳐도 됨. 프로젝트 맥락에 맞춰 선택적 도입
  • 단 G1은 학습 가치와 임팩트 모두 높아서 권장

최우선 1개 (지금 가치 있는 것)

#Gap이유
1G1 Feature List 자동 생성난이도 하, 임팩트 높음. 반나절 작업. 학습·체감·저위험 교집합. 하네스 원리 체득 최적 입문. Anthropic 실패 ①②③ 동시 대응해 학습 효율 최고

앞으로 더 배울 것 (학습 체크리스트)

이 노트를 작성하면서 아직 확신이 없는 지점 2가지. 나중에 멘토·동료·자료로 보완할 주제.

  • 하네스의 경계선 정의 — 어디까지가 하네스이고 어디부터가 그냥 프로젝트 인프라인가? 예: scripts/deploy.sh는 하네스? Git hook은? CI 파이프라인은? 실무 기준이 필요
  • MVP 단계별 하네스 투자 우선순위 — "이론적 중요도"와 "상황별 긴급도" 충돌 시 판단 기준. "유저 100명 전엔 이 수준까지만" 같은 rule of thumb
8. 실제 수정 TODO (분석 중 발견)

이 섹션의 의미

§5의 Gap 4개는 "이론적 개선 지점"(언젠가 필요해지면 꺼내 쓰는 매뉴얼)이다. 반면 이 §8은 "분석 과정에서 실제로 발견된 즉시 수정 가능한 항목"을 누적한다. Gap과 달리 지금 당장 처리하면 더 깔끔해지는 작업들이다.
