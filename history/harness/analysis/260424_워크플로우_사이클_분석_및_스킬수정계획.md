## 1. 사용자 원래 진행 방식 (기준)

### 채팅 단위 원칙
**모든 단계는 새로운 채팅으로 분리.** 각 채팅은 독립된 컨텍스트로 운영.

### 전체 사이클

```
[Phase 0] 새 채팅
  /milestone-review → MVP 방향 확정 + 로드맵 초안 작성

[Phase 1] 마일스톤 기획 (마일스톤 N개만큼 채팅 N개)
  새 채팅 → /milestone → M1 기획문서 작성 → 문서 업데이트 → 채팅 종료
  새 채팅 → /milestone → M2 기획문서 작성 → 문서 업데이트 → 채팅 종료
  ...

[Phase 2] 구현 (마일스톤 N개만큼 채팅 N개)
  새 채팅 → /workflow M1 → step 1~9 강제 → 커밋 + 문서 업데이트 → 채팅 종료
  새 채팅 → /workflow M2 → step 1~9 강제 → 커밋 + 문서 업데이트 → 채팅 종료
  ...

[Phase 3] MVP 완료
  전체 마일스톤 done → MVP 완료 처리 + 아카이빙
```

### 스텝 강제 원칙

- 기획 문서(M{X}_*.md)나 메인태스크 문서가 이미 있어도 **무조건 step 1부터 시작**
- 해당 없는 스텝도 건너뛰지 않음 — **"이 스텝은 해당 없어 패스합니다"를 명시** 후 다음으로
- 각 스텝을 통과하면서 기획에서 빠진 부분을 보완하는 것이 목적
- 넘어가도 된다고 판단 시 사용자에게 명시적으로 알리고 넘어감

### 이 방식의 의도

- `/milestone`이 기획 집중 → `/workflow`가 구현 집중으로 역할 분리
- 기획이 잘 돼 있어도 워크플로우 스텝을 통해 빠진 부분 보완
- 채팅 분리로 컨텍스트 오염 방지

## 3. 하네스 튜닝이 만든 구조적 문제

### 튜닝 내용 (2026-04-20)

- **G1**: `/workflow` step-4에 Feature List 초안 생성 추가
- **G6**: 스킬별 allowed-tools 보수 원칙 정비
- **§8.2**: `active_subtask.txt` → `feature-list-check.sh` 커밋 차단 체인

### 문제가 되는 의존성 체인

```
/milestone → M{X}_*.md 생성
              ↓ (연결 없음)
/workflow step-4 → 메인태스크 문서 + Feature List 생성
                   ↑ M{X}_*.md가 이미 있으면 AI가 이 단계 건너뜀
                   → active_subtask.txt = "none" (기록 안 됨)
                   → feature-list-check.sh 조용히 통과
                   → 커밋 차단 방어망 무력화
```

### 근본 원인

하네스 튜닝이 `/workflow` 안에 Feature List 체인을 심었는데:

1. `/milestone`에서 만든 M{X}_*.md와 `/workflow` step-4에서 만드는 메인태스크 문서가 **연결되지 않음**
2. AI가 M{X}_*.md를 보고 "기획 완료"로 판단 → step-4 건너뜀
3. `active_subtask.txt`가 `/milestone`에서 설정되지 않아 `/workflow`에서 체인 시작 불가
4. 스텝 강제 원칙이 스킬에 명시되지 않아 AI가 임의로 스텝 압축

## 4. Option B: 스킬 수정 계획

### 핵심 원칙
- M{X}_*.md(기획 문서)를 Feature List의 **원본**으로 사용
- `/milestone` 완료 시 `active_subtask.txt` 자동 설정
- `/workflow`에서 문서 있어도 **모든 스텝 강제**, 건너뛸 때 명시 선언 의무화

### 수정 대상

#### `/milestone` 스킬
```
변경: M{X}_*.md 생성 마지막 단계에 추가
→ active_subtask.txt = "progress/mvp{N}/M{X}_*.md 경로" 자동 기록
→ 이후 /workflow가 이 파일을 읽어 Feature List 원본 참조 가능
```

#### `/workflow` step-4
```
변경: active_subtask.txt 읽기 → M{X}_*.md의 아이템 표를 Feature List로 사용
→ 별도 메인태스크 문서 생성 제거 (문서 중복 제거)
→ 아이템 표 없으면 직접 Feature List 생성 (fallback)
```

#### `feature-list-check.sh`
```
변경: M{X}_*.md의 아이템 표 (`| # | ... | 상태 |` 포맷) 파싱
→ 미체크 항목 있으면 커밋 차단 (기존 동작 유지)
→ active_subtask.txt = "none"이면 경고만 (차단 제거 — 기획 채팅에서는 subtask 없음)
```

#### `/workflow` 스킬 (스텝 강제 원칙)
```
추가: 스킬 상단에 원칙 명시
→ "기획 문서가 있어도 step 1부터 무조건 시작"
→ "해당 없는 스텝은 '이 스텝은 해당 없어 패스합니다' 명시 후 다음으로"
→ "사용자 확인 없이 스텝 압축 금지"
```

### 기대 효과

| | 수정 전 | 수정 후 |
|--|---------|---------|
| 마일스톤별 채팅 분리 | ❌ 흐려짐 | ✅ 복원 |
| /workflow = 구현 집중 | ❌ 기획 재생성 압박 | ✅ 복원 |
| Feature List 커밋 차단 | ⚠️ 형식만, 실작동 안함 | ✅ 실작동 |
| 스텝 강제 | ❌ AI가 임의 압축 | ✅ 명시 강제 |
| 문서 중복 | ✅ M{X}_*.md + 메인태스크 둘 다 | ✅ M{X}_*.md 하나로 통합 |
