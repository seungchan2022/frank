- Feature List 초안 생성 로직 (01번)

### 사용자 일괄 보고 허용

사용자가 여러 항목을 한 번에 보고할 경우 일괄 갱신 허용:

```
사용자: "F-01, F-02, E-01 통과. E-02는 N/A (백엔드라 UI 무관). T-01 연기 (다음 마일스톤)."
```

→ Claude는 4건 한 번에 Write 갱신. 각 항목 확인 프롬프트 생략. **Frank 규모(15~25개) 기준 실용적**.

### 항목 추가/수정/삭제 규칙 (실측 중 발견)

실측 중 초안에 없던 중요 검증 포인트 발견 또는 항목 부정확 판단 시:

| 동작 | 트리거 | 처리 |
|------|--------|------|
| 항목 추가 | 사용자 "이거 빠졌어. 추가해: ..." | 해당 카테고리 마지막 순번 다음으로 ID 자동 부여 (예: `F-18`), 초기 상태 `[ ]` |
| 항목 내용 수정 | 사용자 "F-03 문구 바꿔: ..." | ID 유지, 내용만 교체 |
| 항목 삭제 | 사용자 "F-05 삭제" | 해당 라인 제거. 뒤 항목 ID는 재번호 안 함 (안정성 유지) |

**ID는 한 번 부여되면 삭제돼도 재활용 안 함**. 뒤 항목 ID 재번호 안 함(파싱 안정성). 순번에 간격이 생겨도 무방.

## 리뷰 결과

### Claude 리뷰 (문서 일치성)
- **도구 권한 불일치**: 기존 step-8 스킬 `allowed-tools`는 Bash·Read만. 자동 체크 갱신은 Write 필요 → **재작업으로 해소**: In-Scope에 "allowed-tools에 Write 추가" 명시.
- **"미체크 = 미완료" 전제 붕괴**: AI가 초기 생성한 항목이 구현 중 불필요로 판명되면 사용자 수정 경로 없음 → **재작업으로 해소**: 4상태 모델(`[~]`/`[-]`) + 항목 추가/수정/삭제 규칙으로 실측 중 대응.
- **1:N 시나리오-체크박스 매핑 신뢰도**: **재작업으로 해소**: 시나리오 개념 자체 제거. 체크박스 순회로 단순화.

### Codex 리뷰 (기술적 타당성)
- ❌ **반려** → 재설계로 승인 조건 충족
- 자동 갱신 도구 권한 불일치 → **해소**: allowed-tools Write 포함
- `§카테고리 #번호` 규약 01 미정의 → **해소**: 시나리오 개념 제거 + 01 안정 ID 규약 사용 통일
- `skip` 메타 01과 불일치 → **해소**: 01의 단일 메타 1줄(`<!-- size | count | skip -->`) SSOT 준수
