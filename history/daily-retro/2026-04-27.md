## 오늘 무엇을 결정했나

### 1. BUG-F 오답 태그 필터 — 근본 수정 defer 확정

오답 태그 칩이 비는 근본 원인은 `wrong_answers` 테이블에 `tag_id` 컬럼이 없는 서버 스키마 문제다. 해결하려면 DB 스키마 + 서버 API + 웹 + iOS 4개 레이어를 동시 변경해야 한다.

문제를 발견한 시점이 너무 늦었다. 서버·DB 작업을 완료하고 웹 구현까지 다 끝낸 후에야 이 이슈를 발견했다. 즉 근본 수정을 하려면 이미 완료된 단계들을 전부 다시 되돌아가 수정해야 하는 상황이었다. 범위와 비용이 MVP12에서 감당하기 어려운 수준이라 다음 MVP로 넘겼다.

MVP12 범위 내에서 할 수 있는 건 "즐겨찾기 미등록 오답은 태그 필터에서 제외한다"는 정책을 웹·iOS에서 통일하는 것까지였다.

**결정 기준**: 발견 시점이 늦어 이전 완료 단계 전체 재수정 필요 → 다음 MVP 단독 기획.

### 2. 피드 탭 필터 방식 — 클라이언트 필터링 유지, 재검토는 다음 MVP

**배경**: MVP6부터의 이전 방식은 탭을 선택하면 서버에 `?tag_id=xxx` 파라미터를 붙여 별도 API 요청을 보내고, 서버가 해당 태그로 필터링된 기사를 반환하는 방식이었다.

MVP12에서 "전체만 한 번 API 요청하고, 다른 탭은 가져온 데이터에서 클라이언트가 필터링하면 되지 않을까"라는 생각으로 방식을 바꿨다. 서버 요청 횟수를 줄이고 탭 전환을 빠르게 하려는 의도였다.

그런데 **예상 못한 사이드 이펙트가 생겼다**: 전체 데이터 로딩이 완료되기 전에 다른 탭을 선택하면 기사가 제대로 표시되지 않는다. 클라이언트가 필터링할 원본 데이터 자체가 아직 없는 타이밍이기 때문이다.

이전 방식(탭별 서버 요청)으로 되돌리는 게 맞는지 다시 검토할 필요가 있지만, 두 방식의 장단점을 분석하고 결정까지 하면 작업 흐름이 늦어질 것 같아 일단 현재 방식을 유지하고 다음 MVP에서 재검토하기로 했다.

### 3. BUG-F 최종 방식 — 클라이언트 사이드 B안 확정

세 개 개선안(A: 서버 스키마, B: 클라이언트 필터, C: 혼합)을 분석 문서로 비교했다. 결론은 B-1(태그 필터 대상을 즐겨찾기 등록 기사로 한정) + B-2(orphan selectedTagId 자동 초기화).

A안은 맞는 방향이지만 MVP12 범위를 벗어난다. 아키텍처 원칙 우선 — 지금 고칠 수 없는 건 명확히 defer하고 할 수 있는 것만 깔끔하게 구현.
