01
BUG-F 오답 태그 필터 — 근본 수정 defer 확정
defer → MVP13 발견 시점 문제
근본 원인은 wrong_answers 테이블에 tag_id가 없는 서버 스키마 문제. 문제를 발견한 시점이 너무 늦었다 — 서버·DB 작업 완료 + 웹 구현까지 다 끝낸 후에야 이슈를 발견했고, 근본 수정을 하려면 이미 완료된 단계들을 전부 되돌아가 재수정해야 하는 상황이었다. 범위와 비용이 MVP12에서 감당하기 어려운 수준이라 클라이언트 필터 정책 통일(B안)까지만 처리하고 스키마 변경은 다음 MVP로 넘겼다.
02
피드 탭 필터 방식 — 클라이언트 필터링 유지, 재검토는 다음 MVP
재검토 → MVP13 사이드 이펙트 발견
이전 방식(MVP6~): 탭 선택 시 서버에 ?tag_id=xxx 요청 → 서버가 필터링해서 반환.

MVP12 변경: 전체만 한 번 API 요청 + 클라이언트 filter()로 다른 탭 처리. 서버 요청 횟수를 줄이고 탭 전환을 빠르게 하려는 의도였다.

사이드 이펙트: 전체 데이터 로딩 완료 전에 탭을 선택하면 기사가 표시되지 않음. 클라이언트가 필터링할 원본 데이터 자체가 아직 없는 타이밍 문제.

이전 방식으로 되돌리는 게 맞는지 재검토가 필요하지만, 지금 분석하면 작업 흐름이 늦어지므로 현재 방식 유지 + 다음 MVP에서 재결정.
03
BUG-F 최종 방식 — 클라이언트 B안 확정
아키텍처 원칙 우선
분석 문서에서 A(서버 스키마) / B(클라이언트 필터) / C(혼합) 비교. A안은 방향은 맞지만 MVP12 범위 초과. B-1(즐겨찾기 등록 기사로 태그 필터 한정) + B-2(orphan selectedTagId 자동 초기화)로 확정. "지금 고칠 수 없는 건 명확히 defer, 할 수 있는 건 깔끔하게" 원칙 적용.
