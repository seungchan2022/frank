🧠 핵심 의사결정
결정 1

OpenRouter → Groq 교체

요약·퀴즈 생성이 30초 이상 걸렸다. "모델이 느린 게 아니라 무료 대기열이 느린 것"임을 파악한 게 핵심이었다.

옵션 레이턴시 비용
OpenRouter 유료 전환 5~10초 월 청구 시작
다른 모델로 교체 불명확 동일
✅ Groq 직접 연결 3~5초 무료 1,000 RPD

    A[Frank 서버] -->|Before: 30초+| B[OpenRouter\n무료 대기열]
    B --> C[모델 회사]

    D[Frank 서버] -->|After: 3~5초| E[Groq\n전용 칩 LPU]

    style B fill:#ffcccc,color:#333
    style E fill:#ccffcc,color:#333
느린 이유를 "모델 성능" 탓으로 넘기지 않고, 실제 병목(중계 레이어의 대기열)을 파악한 게 결정적이었다. 같은 모델을 어디서 실행하느냐에 따라 UX가 천지 차이가 난다.
결정 2 — 헷갈렸던 부분 🤔

Tavily time_range: "week" 추가

피드 새로 고침을 해도 같은 기사만 계속 나왔다. 날짜 필터가 없어서 Tavily가 항상 동일한 "최상위 기사"를 반환했다.

❌ days: 7 topic: "news" 설정 시에만 동작
✅ time_range: "week" 일반 검색에서 최근 7일 필터
// 변경 전 — 날짜 필터 없음 let body = json!({ "query": query, "max_results": max_results, "search_depth": "advanced", "include_answer": false, }); // 변경 후 — 최근 7일 필터 let body = json!({ "query": query, "max_results": max_results, "search_depth": "advanced", "include_answer": false, "time_range": "week", });
API 파라미터는 문서를 꼭 확인해야 한다. days와 time_range처럼 비슷해 보이지만 동작 조건이 다른 파라미터는 어림짐작 금지. wiremock body_partial_json 매처로 요청 본문까지 검증하는 테스트가 안전망이 됐다.
결정 3

clean_snippet 복합 방식 — 패턴 필터 + 문장 경계 절단

snippet에 arrow_forward - #Android Studio, TechCrunch –:–:–:–, 00:00 01:00 02:00 같은 노이즈가 섞여 있었다. MVP8 clean_snippet은 HTML/URL/이메일만 제거했다.

접근 장점 단점
노이즈 패턴 정규식 정확히 제거 오탐 가능
HTML 제거 + 절단 단순, 범용 정보 손실 가능
LLM 정제 가장 정확 비용 + 레이턴시
✅ 복합 방식 레이어로 보완 -
텍스트 정제는 "완벽하게 깨끗하게"가 목표가 아니라 "충분히 읽을 수 있게"가 목표다. 비용 없이 regex만으로도 체감 품질이 확 올라간다.
결정 4 — 헷갈렸던 부분 🤔

SessionWrongAnswerAccumulator — 독립 struct 분리

퀴즈 진행 중 오답을 로컬에 누적해서 완료 화면에 표시해야 했다. 처음엔 배열을 View 안에 바로 추가하면 되지 않을까 생각했다.

// 방법 A: QuizView 안에 배열 직접 @State private var sessionWrongAnswers: [SessionWrongAnswerItem] = [] // 오답 시 View에서 조건 체크 if selected != question.answerIndex { sessionWrongAnswers.append(...) } // 방법 B: Accumulator struct 분리 (실제 구현) @State private var wrongAccumulator = SessionWrongAnswerAccumulator() // 오답 판별 로직이 Accumulator 안에 캡슐화 wrongAccumulator.record(question: question, userIndex: selected)

차이는 단 하나: 조건 체크 로직이 어디 있냐. 방법 B는 Accumulator 안에 guard userIndex != question.answerIndex가 있어서, QuizView 없이 독립적으로 테스트할 수 있다.

"이걸 왜 따로 struct로 만들지?" 라는 의구심이 들 때, 그 이유가 "테스트 가능성"이라는 걸 체감으로 이해하게 됐다. TDD를 따르면 자연스럽게 설계가 좋아진다.
결정 5

✅ 클라이언트만 수정 M1과 완전 독립 기존 엔드포인트 재사용
새 기능을 만들 때 "서버 변경이 반드시 필요한가?"를 먼저 확인하는 습관. 기존 데이터를 클라이언트에서 다르게 조합하는 것만으로 UX가 크게 달라질 수 있다.

💡 인사이트
1
병목 진단이 먼저다. "LLM이 느리다" → "OpenRouter 무료 대기열이 느리다" → "Groq에서 실행하면 6배 빠르다". 증상이 아니라 원인을 찾는 게 해결책의 품질을 결정한다.
2
API 파라미터는 반드시 문서 확인. days와 time_range처럼 비슷해 보이지만 동작 조건이 다른 파라미터는 어림짐작 금지.
3
테스트 가능성이 설계를 이끈다. SessionWrongAnswerAccumulator를 별도 struct로 분리한 이유가 "QuizView 없이도 단위 테스트 가능"이라는 것. TDD를 따르면 자연스럽게 설계가 좋아진다.
4
"서버 변경이 필요한가?"를 먼저 확인. M2 전체가 클라이언트만으로 해결됐다. 기존 데이터를 다르게 조합하는 것만으로 UX가 크게 달라질 수 있다.
5
하루에 MVP 하나가 가능한 조건. 문제가 명확하고, 의존성이 없고, 병렬 구조가 잡혀 있을 때. 오늘이 딱 그 조건이었다.
📚 배운 것
Groq vs OpenRouter 구조 차이: 중계 레이어가 없는 전용 인프라는 레이턴시 차원이 다름. 무료 플랜 = 대기열 = 느림.
Tavily time_range vs days: topic 설정 여부에 따라 쓰는 파라미터가 다름. 일반 검색 = time_range: "week".
SessionWrongAnswerAccumulator 패턴: View에서 로직을 빼서 별도 struct에 캡슐화 → 독립 테스트 가능. 조건 체크 위치가 설계를 바꾼다.
wiremock body_partial_json 매처: 요청 본문의 일부만 검증. 파라미터가 실제로 전달되는지 안전망 역할.
lazy fetch 패턴: "이미 로드됐으면 재사용, 없으면 그때 fetch" — 불필요한 API 호출 없이 기존 데이터를 재활용.
😌 느낀 점

하루 만에 MVP 하나를 기획부터 아카이빙까지 완주한 게 오늘의 하이라이트다. 처음엔 M1이 끝나야 M2를 시작해야 하는 줄 알았는데, 의존성을 확인해보니 완전히 독립적이었다. 병렬로 동시에 진행했고, 하루 안에 전부 끝났다.

SessionWrongAnswerAccumulator를 처음 봤을 때 "이걸 왜 따로 struct로 만들었지?"가 솔직한 반응이었다. 막상 이유를 파악하고 나니 "테스트를 먼저 생각하면 설계가 이렇게 된다"는 게 납득됐다. TDD가 단순히 테스트를 먼저 쓰는 게 아니라 설계 방식 자체를 바꾼다는 걸 오늘 실감했다.

Tavily time_range는 문서 안 보고 days: 7로 바로 했다면 효과가 없었을 텐데, 마일스톤 문서에 주의 사항이 적혀 있어서 잡아냈다. 문서가 제대로 작성되어 있으면 이런 함정을 방지할 수 있다는 걸 다시 확인했다.
