## 핵심 결정 3개

### 1. MVP11 기획 순서 — 버그 우선 · 서버 먼저 · 웹·iOS 병렬

**왜 이 순서인가**  
M2·M3·M4는 서버 API가 이미 태그를 반환하고 있어 의존성이 없다.
M1(BUG-004)만 먼저 잡으면 나머지는 병렬로 돌릴 수 있는 구조였다.
버그를 안고 기능을 올리면 디버깅 범위가 넓어지기 때문에 수정 우선 원칙을 택했다.

**결정**  
M1 완료 후 M2(iOS 초기화 버그) · M3(웹 즐겨찾기/오답 태그 필터) · M4(iOS 즐겨찾기/오답 태그 필터) 병렬 진행 구조 확정.

> 의존성을 먼저 그리면 병렬화 가능한 곳이 보인다. "서버 먼저"는 속도가 아니라 의존 관계의 문제다.

### 2. `category:"news"` — API 팩트 정정이 플랜보다 먼저였어야 했다

**무슨 일이 있었나**  
M1 플랜에는 Exa 파라미터로 `type:"news"`가 명시돼 있었다.
실제로 적용 전 공식 문서를 확인했더니 `type`은 검색 알고리즘 선택용(neural/keyword)이고,
콘텐츠를 뉴스로 필터하는 건 `category:"news"`였다.

**어떻게 처리했나**  
문서 확인 즉시 플랜 수정 후 `category:"news"`로 구현.

> 플랜은 가설이다. 외부 API 파라미터는 구현 전에 공식 문서에서 한 번 더 확인하는 루틴이 필요하다.

### 3. 기술 부채 명시 — "두더지 잡기"임을 인정하고 기록

**왜 이 결정이 중요했나**  
URL 패턴 필터(is_listing_url)는 각 검색엔진의 URL 구조가 제각각이라
완전한 차단이 불가능하다. 오늘 가장 어려웠던 부분도 이 지점이었다 — 엔진마다 패턴이 달라
어떻게 처리해야 할지 아직 확정이 안 됐고, 이후로 미뤘다.

**기록한 내용**  
후속 과제 2가지를 로드맵에 명시:
- 뉴스 전용 API(NewsAPI / Bing News) 교체 → 기사만 반환 보장
- LLM 기반 스니펫 분류 → 엔진 독립적 후처리

> 완성처럼 보이지만 완성이 아닌 것을 완성처럼 두면 미래의 나가 속는다.
> 한계를 기록하는 것이 부채를 갚는 첫 번째 단계다.
