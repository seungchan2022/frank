## 핵심 의사결정과 그 이유

### 결정 1: MVP 1 스코프 — "스크랩 중심"으로 좁히기

- **상황**: "AI 기반 스터디앱"이라는 넓은 개념에서 MVP 1의 핵심 가치를 정해야 했다.
- **선택지들**:
  - A) 스크랩 중심 — URL/키워드로 콘텐츠 수집 + LLM 요약
  - B) 학습 경로 중심 — 개인화 학습 플랜 생성
  - C) 인사이트 대화 중심 — 스크랩을 컨텍스트로 LLM과 채팅
- **결정**: A) 스크랩 중심
- **왜**: 나 자신이 AI 정보 과잉으로 힘들어하는 게 출발점이었다. "일단 좋은 콘텐츠를 자동으로 모아주는 것"이 가장 먼저 해결해야 할 문제. 학습이나 채팅은 콘텐츠가 쌓인 후에야 의미가 있다.
- **인사이트**: MVP에서 가장 중요한 건 "하나의 핵심 루프"를 완성하는 것. 여러 기능을 얇게 만드는 것보다 하나를 제대로 만드는 게 낫다. Artifact가 실패한 이유도 참고했다 — 뉴스 소비만으로는 앱 유지가 어렵다는 교훈.

### 결정 2: 에코 서버 + 포트/어댑터 패턴 채택

- **상황**: Rust API 서버의 DB 접근 방식을 정해야 했다. Supabase를 쓰는데, sqlx(직접 SQL)를 쓸지, REST API를 쓸지가 문제였다.
- **선택지들**:
  - A) 혼용 전략 — DB는 sqlx, Auth는 Supabase REST (Claude 제안)
  - B) 에코 서버 — 모든 외부 호출을 포트로 통일, Supabase REST만 사용 (내가 제안)
  - C) sqlx만 — Supabase를 순수 PostgreSQL로만 사용
  - D) 자체 구축 — Supabase 없이 전부 직접
- **결정**: B) 에코 서버 패턴
- **왜**: TDD가 프로젝트의 핵심 원칙인데, 에코 서버 패턴이 TDD와 궁합이 압도적으로 좋았다. Fake 어댑터 하나면 외부 의존 없이 전체 테스트가 가능하다. 혼용 전략은 sqlx mock + REST mock을 각각 관리해야 해서 인지 부하가 높았다. 그리고 나는 Rust 초보라서, sqlx의 매크로/컴파일 타임 검증과 씨름하는 시간은 MVP에 도움이 안 된다고 판단했다.
- **인사이트**: 이건 내가 직접 제안한 패턴이다. Claude가 제안한 혼용 전략을 심판해달라고 했더니, 내 제안이 25.5 vs 34.5로 이겼다. "모든 외부 호출 = HTTP"라는 단일 패턴의 힘이 크다. 그리고 이 패턴은 서버뿐 아니라 웹, iOS 앱에도 동일하게 적용할 수 있다. 포트 추상화가 되어 있으니, 나중에 성능이 문제되면 sqlx 어댑터를 추가하면 된다. 지금 결정이 영구적이지 않다는 게 핵심.

### 결정 3: 콘텐츠 수집 폴백 체인 설계

- **상황**: 사용자가 키워드를 선택하면 자동으로 웹 검색해서 콘텐츠를 가져와야 하는데, 어떤 소스를 쓸지 정해야 했다.
- **선택지들**:
  - 단일 소스 (Tavily만)
  - 다중 소스 동시 호출
  - 폴백 체인 (우선순위 기반)
- **결정**: Tavily → Exa → Firecrawl → arXiv 폴백 체인
- **왜**: 경쟁 서비스 조사를 해보니, 기존 서비스 중 Tavily + Exa + Firecrawl + arXiv를 하나의 파이프라인으로 연결한 곳이 없었다. 각 API의 무료 티어(Tavily 1,000회/월, Exa 1,000회/월, arXiv 무제한)를 합치면 MVP에 충분한 월 2,000+ 검색이 가능하다. 폴백 방식을 선택한 이유는 한 API가 실패하거나 한도가 소진되었을 때 자동으로 다음 소스로 전환하기 위해서다.
- **인사이트**: 무료 티어들을 조합하면 MVP에서는 비용 걱정 없이 시작할 수 있다. "먼저 무료로 검증하고, 트래픽이 생기면 유료로 전환"이 스타트업 사고방식이라는 걸 배웠다.

### 결정 4: SvelteKit 서버 프록시 방식

- **상황**: 웹 프론트에서 Rust API 서버를 호출하는 방식을 정해야 했다.
- **선택지들**:
  - A) SvelteKit 서버에서 프록시 — 브라우저 → SvelteKit → Rust API
  - B) 브라우저에서 직접 호출 — CORS 설정 필요
- **결정**: A) 서버 프록시
- **왜**: 브라우저에서 직접 API 서버를 호출하면 API 키가 노출될 위험이 있고, CORS 설정도 번거롭다. SvelteKit의 +server.ts에서 프록시하면 API 키를 서버 사이드에 숨길 수 있고, 클라이언트는 같은 도메인으로만 통신하면 된다.
- **인사이트**: 보안은 설계 단계에서 결정하는 것이지, 나중에 추가하는 게 아니다.

## 기획/설계 과정

오늘의 기획 과정은 독특했다. 코드를 한 줄도 안 짜고, 하루 종일 "무엇을 만들 것인가"만 논의했다.

### 인터뷰 기반 기획

멘토가 `/workflow` 스킬의 인터뷰 방식으로 하나씩 질문을 던져줬다. 처음에는 "AI 기반 스터디앱"이라는 막연한 아이디어만 있었는데, 질문에 답하다 보니 점점 구체화됐다.

특히 인상 깊었던 순간은 **"수동 스크랩이 아니라 자동 수집"**으로 방향이 바뀐 때다. 원래는 사용자가 URL을 직접 저장하는 방식을 생각했는데, "키워드를 선택하면 앱이 알아서 찾아주는 게 더 낫지 않나?"라고 말하니 기획이 완전히 달라졌다. 핵심 가치가 "저장 도구"에서 "자동 큐레이션"으로 바뀌었다.

### 경쟁 서비스 리서치

기획 중간에 "나만 이런 생각을 한 게 아닐 텐데"라는 의문이 들어서 경쟁 서비스 조사를 요청했다. Artifact(실패), Feedly(유료), Readwise(수동), Perplexity Discover(개인화 약함) 등을 분석하고 나니, Frank의 포지셔닝이 명확해졌다:

> "키워드 기반 자동 수집 + LLM 인사이트를 하나로 묶은 서비스는 아직 없다"

이 리서치가 없었으면 그냥 "또 하나의 뉴스 앱"을 만들 뻔했다.

### 아키텍처 논쟁

가장 뜨거웠던 건 DB 접근 방식 논의였다. Claude는 sqlx + Supabase REST 혼용을 제안했고, 나는 에코 서버 패턴을 제안했다. "심판해달라"고 요청했더니 8개 기준으로 채점해서 비교해줬는데, 내 제안이 이겼다.

이 경험에서 배운 건: **AI의 첫 번째 제안을 그대로 받아들이지 말 것**. 나만의 관점으로 대안을 제시하고 비교해보면 더 나은 결정이 나온다.

## 인사이트 & 피드백

### 1. "다 모르면 오히려 유리하다"

iOS 개발자가 Rust + Svelte + Supabase를 처음 접하는 건 무섭을 수 있다. 근데 실제로 해보니, 기존 지식에 얽매이지 않아서 더 깔끔한 결정을 내릴 수 있었다. "Swift에서는 이렇게 하는데..."라는 편견이 없으니 각 기술의 장점을 있는 그대로 받아들일 수 있었다.

### 2. "바이브 코딩은 의사결정의 연속이다"

코드를 한 줄도 안 짰지만, 오늘 내린 결정들이 앞으로 수천 줄의 코드 방향을 결정한다. 바이브 코딩은 "코드를 빨리 짜는 것"이 아니라 "올바른 방향을 빠르게 찾는 것"이었다.

### 3. "AI에게 심판을 맡기면 객관적 비교가 된다"

에코 서버 vs 혼용 전략을 비교할 때, 직접 판단하려면 편향이 생길 수 있다. "8개 기준으로 채점해달라"고 요청하니 정량적 비교가 나왔고, 근거가 명확해졌다. 다음에 비슷한 상황이 오면 이 방법을 또 쓸 것 같다.

### 4. "문서 분리의 중요성"

처음에 CLAUDE.md에 기획 내용을 전부 넣었다가, "이건 작업 가이드와 기획 문서의 역할이 다르다"는 걸 깨달았다. CLAUDE.md는 간결한 가이드, progress/는 상세 기획. 문서의 목적에 맞게 분리하니 둘 다 읽기 좋아졌다.

### 5. "포트/어댑터 패턴은 모든 플랫폼에 적용 가능하다"

에코 서버 패턴을 Rust에만 쓰는 게 아니라, iOS(Protocol), Svelte(Interface)에도 동일하게 적용하기로 했다. "한 번 배운 패턴을 모든 곳에 재사용"이라는 게 진짜 아키텍처의 가치라는 걸 느꼈다.

## 배운 것

### 기술적으로

- **Rust 웹 프레임워크 생태계**: Axum이 2026년 사실상 표준이라는 것, Tokio/Tower 생태계와 네이티브 통합
- **Supabase REST API**: PostgREST 기반으로 SQL 없이 DB 접근 가능, RLS 자동 적용
- **웹 검색 API 생태계**: Tavily(AI 랭킹), Exa(시맨틱), Firecrawl(크롤링) 각각의 강점
- **OpenRouter**: 다양한 LLM을 하나의 API로 접근할 수 있는 라우터

### 프로세스에서

- 인터뷰 기반 기획은 막연한 아이디어를 구체화하는 데 매우 효과적
- 경쟁 서비스 리서치는 기획 중간에 해야 함 (처음이 아니라)
- AI와 "토론"하는 것과 AI의 "제안을 수용"하는 것은 완전히 다른 경험

## 느낀 점

솔직히 처음에는 "iOS 개발자가 Rust 서버를 만든다고?"라는 생각이 있었다. 근데 하루 종일 멘토와 함께 기획을 하고 나니, 코드를 짜기 전에 방향을 제대로 잡는 게 얼마나 중요한지 느꼈다.

바이브 코딩 1일차 치고는 꽤 많은 걸 결정했다. 에코 서버 패턴을 직접 제안하고 그게 채택된 건 좀 뿌듯했다. "모르니까 질문하고, 질문에서 답이 나오고, 답에서 또 질문이 나오는" 이 사이클이 재밌었다.

내일부터는 진짜 코드를 짜야 하니까 좀 긴장되긴 한다. 근데 오늘 기획이 탄탄하니까 방향을 잃을 일은 없을 것 같다.
