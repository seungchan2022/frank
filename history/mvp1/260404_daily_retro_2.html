🔑 핵심 의사결정

SCOPE
결정 1: MVP 범위를 어디서 끊을까

기능 욕심이 계속 났지만, "이게 핵심 루프에 필요한가?"라는 질문으로 걸렀다. 태그 필터와 레이지 로딩은 넣었지만(기본 UX), 스터디/채팅은 MVP2로 미루었다.

MVP 범위는 한 번 정하고 끝이 아니라, 만들면서 계속 재조정하는 것이다.
ARCHITECTURE
결정 2: Supabase REST → sqlx 전환

아침에 "REST만 쓴다!"고 했는데, 실제로 써보니 복잡한 쿼리에서 마찰이 컸다. 포트/어댑터 패턴 덕분에 어댑터만 교체 — 서비스 레이어는 한 줄도 안 고쳤다.

좋은 아키텍처는 "결정을 바꿔도 충격이 적은 구조"를 만드는 것이다.
QUALITY
결정 3: 버그를 다 고치고 나서야 "완료"

기능만 만들고 "MVP 완료!"라고 할 수 있었다. 근데 작동 안 하는 소프트웨어는 MVP가 아니다. 커밋 11개를 전부 버그 수정에 썼다.

기능 구현보다 버그 수정에 시간이 더 걸렸다. 이건 소프트웨어 개발의 본질이다.

🔧 가장 많이 삽질한 곳

서버-클라이언트 연결 체인

브라우저 → SvelteKit 프록시 → Rust API → Supabase DB

각각은 이해할 수 있는데, 체인으로 엮이면 디버깅이 지옥이었다. JWT, CORS, RLS 정책, 프록시 라우팅이 동시에 문제를 일으키면 어디서 터진 건지 찾기가 너무 힘들었다.

프로세스에서 배운 3가지

1️⃣ 기획과 구현은 선형이 아니라 반복적 — 만들면서 기획이 바뀐다

2️⃣ MVP 범위는 "한 번 정하고 끝"이 아니라 계속 재조정하는 것

3️⃣ 바이브 코딩에서 내 역할은 "의사결정자" — 코드는 AI가, 방향은 내가

💭 느낀 점

솔직히 오늘은 "만족하는 결정"보다는 "하나씩 흐름을 찾아가는" 느낌이었다.

기획할 때는 자신 있었는데, 코드가 나오기 시작하니까 모르는 게 쏟아져 나왔다. 특히 서버 연결 부분은 iOS 개발에서는 경험할 수 없었던 영역이라 많이 헤맸다.

그래도 "전부 다 어려운데 전부 다 해냈다"는 게 뿌듯하다. 커밋 28개가 그 증거다.

바이브 코딩이 "쉬운 코딩"은 아니었다. 코드를 직접 안 짜는 것뿐이지, 모든 결정은 내가 내려야 한다. 오히려 결정의 빈도가 높아서 더 집중력이 필요했다.
