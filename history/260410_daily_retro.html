결정 · 인사이트
DECISION 01

이미지 소스: Tavily images[] → og:image 직접 크롤링

결정

썸네일 이미지를 Tavily가 반환하는 images[] 배열 대신, 기사 URL에서 og:image를 직접 크롤링해서 가져오도록 변경했다. 병렬 크롤링 + 5초 타임아웃. URL 정규화(http/https·www·trailing slash 통일) 후 HashSet dedup으로 중복 기사도 함께 제거.

왜 중요한가

Tavily의 images[]는 기사 페이지에 있는 모든 이미지를 반환한다. 광고, 아이콘, 버튼 이미지가 섞여서 대표 이미지로 쓸 수 없었다. og:image는 기사 발행자가 직접 지정한 대표 이미지라 품질이 일정하다. 크롤링 실패 시 플레이스홀더로 대체하고, 5초 타임아웃으로 느린 사이트를 대응했다.

서드파티 API가 반환하는 데이터가 항상 우리가 원하는 형태는 아니다. Tavily가 편의를 위해 제공하는 images[]보다, 원문 사이트에서 직접 가져오는 게 품질이 더 높았다. "API가 주는 것"과 "실제 필요한 것"을 구분하는 습관.
DECISION 02

Rust join_all lifetime — SearchJob owned 구조체로 해결

결정

병렬 검색(tokio::join_all) 구현 시 &str 대신 String 필드를 가지는 SearchJob owned 구조체로 job을 묶었다. SearchJob { tag_name: String, query: String } 형태로, &str 참조는 구조체 어디에도 허용하지 않는다.

왜

join_all은 여러 Future를 동시에 실행한다. 각 Future가 외부 참조(&str)를 들고 있으면, 컴파일러가 lifetime을 보장하지 못해 컴파일 에러가 난다. owned 타입(String)으로 바꾸면 Future가 자기 데이터를 직접 소유해 lifetime 문제가 사라진다.

Rust에서 비동기 병렬 처리에 참조를 넘기면 이 문제가 반드시 나온다. 해결책은 단순하다 — owned 타입으로 바꿔서 Future가 독립적으로 실행되게 하면 된다. Arc로 공유하는 방법도 있지만, 복사 비용이 작은 경우엔 owned가 더 깔끔하다.
DECISION 03

iOS scroll to top 구현 → UX 판단으로 롤백

상황

M2 stale-while-revalidate 구현 시 새 데이터 도착 후 자동 scroll to top을 추가했다. 커밋까지 했지만, 실제로 써보니 어색했다. 다음 커밋에서 제거했다.

왜 제거했나

pull-to-refresh는 완료되면 손이 화면 상단에서 떼어지면서 자연스럽게 위로 돌아온다. 여기에 추가로 scroll to top을 하면 이중 동작이 돼서 어색하다. 웹은 progress bar + 백그라운드 갱신이라 scroll to top이 유효하지만, iOS pull-to-refresh는 동작 방식이 다르다.

구현하고 나서 쓰기 전까지 몰랐다. 짧게 써보고 바로 빼는 게 맞는 흐름이었다. "일단 구현하고 써보면서 결정"이 특히 UX 판단에서는 유효하다.
DECISION 04

tagCache 전략 — 탭 전환은 캐시 히트, pull-to-refresh만 무효화

결정

탭 전환 시 캐시가 있으면 재요청 없이 즉시 표시. pull-to-refresh만 현재 탭 캐시를 삭제하고 서버에 재요청. 캐시 키: 웹 'all' | uuid.string, iOS "all" | uuid.uuidString.

왜

탭 전환마다 서버에 요청하면 네트워크 딜레이로 탭이 끊기는 느낌이 난다. 반면 캐시를 영구 유지하면 실제 구독 태그가 바뀌어도 모른다. "수동 새로고침만 갱신"이 중간 지점이다. 초기 로드 시 전체 태그 프리패치로 탭 전환을 항상 즉시 응답으로 만들었다.

Settings에서 태그 변경 후 저장하면, 이전 캐시가 남아서 이전 태그 결과를 보여줬다. handleSave 후 feedStore.reset()으로 전체 캐시를 무효화해서 해결. 캐시 설계 시 "무효화 트리거"를 명시적으로 목록화하는 게 버그를 막는다.
DECISION 05

iOS xcconfig :// 주석 문제 — Project.swift 직접 하드코딩으로 우회

문제

xcconfig에서 Supabase URL 값을 읽으려 했는데, xcconfig 문법에서 ://가 줄 주석(//)으로 해석되어 URL이 잘렸다. 예: https://xxx.supabase.co → https:로 잘림.

결정

Project.swift의 settings 블록에서 직접 값을 주입하도록 전환했다. Config.xcconfig는 .gitignore에 추가. tuist generate 후 project.pbxproj에 xcconfig 경로가 반영되도록 커밋까지 포함했다.

xcconfig의 주석 파싱 문제는 공식 우회 방법이 없다. URL이 들어가는 값은 xcconfig로 관리하기 어렵다. 대안: Project.swift에서 직접 설정하거나, URL에 scheme을 분리한 두 키로 나눠서 조합하는 방법이 있다. 이번엔 직접 하드코딩이 가장 단순하고 빠른 선택이었다.
