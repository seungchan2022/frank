API 설계
요약 실패 에러 코드 원인별 분리 — 500 통일 → 422 / 503 / 408
기존 요약 실패는 전부 500으로 반환했다. 클라이언트가 500만 받으면 "서버 에러"라는 동일 메시지밖에 표시할 수 없다.

422(크롤링 실패 — 기사 본문 없음) / 503(LLM 장애 — OpenRouter/Groq 오류) / 408(타임아웃)으로 분리하면 클라이언트가 원인에 맞는 메시지를 선택할 수 있다.
HTTP 상태 코드는 클라이언트와의 계약이다. 500으로 모든 에러를 뭉개면 클라이언트는 맥락을 잃는다. 원인 분류가 가능한 에러는 코드 수준에서 분리해야 한다.
Silent 버그 발견
좋아요 요청 tag_id 누락 — fire-and-forget의 조용한 실패
M2 작업 중 서버 코드를 보니 keyword_weights 저장 시 tag_id NOT NULL 조건이 있었다. 클라이언트(웹+iOS)가 tag_id를 보내지 않아 keyword_weights 저장이 조용히 실패하고 있었다.

좋아요 동작 자체는 성공했다(hearts 테이블은 정상 저장). 부수 효과(keyword_weights)만 실패하는 구조라 UI에서 아무 에러도 보이지 않았다.
fire-and-forget 패턴은 실패가 무음으로 지나간다. 중요도에 따라 실패 로깅 또는 알림 정책을 명시적으로 정해야 한다.

캐시 설계
피드 TTL 캐시 — Arc<Mutex<HashMap>> + Instant, Redis 제외
Redis 대신 Rust 인메모리로 TTL 캐시 구현. 5분 TTL, 최대 100 엔트리.

캐시 키: "{user_id}:{sorted_tag_ids}" — 사용자 격리 + 태그 조합 구분. UUID prefix 기반으로 invalidate_user() 구현 가능. eviction은 만료 엔트리 우선 제거 → 초과 시 oldest-first.

Redis 제외 이유: 현재 사용자 규모에서 네트워크 왕복 비용이 캐시 이득보다 크다. 추후 다중 인스턴스 배포 시 Redis 전환 — FeedCachePort 교체만으로 가능.
포트/어댑터 패턴의 가치는 구현 교체 비용에 있다. "지금은 인메모리, 나중에 Redis"를 미룰 수 있는 건 FeedCachePort가 경계를 만들기 때문이다.
아키텍처 인사이트
NoopFeedCache — 기존 테스트 8곳 무수정 통과
FeedCachePort 구현체로 NoopFeedCache(항상 miss, 저장 무시)를 도입하고 테스트 픽스처에 주입했다. InMemoryFeedCache를 기존 테스트에 주입하면 테스트 간 캐시 공유 문제가 생기므로 Noop이 가장 안전한 기본값이다.

결과: make_test_state() 수정 없이 Noop 주입만으로 기존 테스트 8곳 전부 통과.
포트/어댑터의 실전 가치는 "기존 테스트를 건드리지 않고 새 구현체를 추가할 수 있다"는 데 있다. 이것이 TDD와 아키텍처가 교차하는 지점이다.
캐시 우회 전략
수동 새로고침 = Cache-Control: no-cache — 캐시 우회 vs 무효화 구분
피드 TTL 캐시 도입 후 "새로고침인데 캐시 HIT → 오래된 데이터"가 표시되는 문제. 웹 새로고침 버튼 / iOS pull-to-refresh 시 Cache-Control: no-cache 헤더를 붙여 서버 캐시를 우회하도록 변경.

no-cache 요청의 결과도 캐시에 저장 — no-cache 응답이 가장 최신 데이터이므로 다음 일반 요청에서 재활용 가능.
"캐시 우회"와 "캐시 무효화"는 다르다. 우회는 이번 요청만 신선하게, 무효화는 캐시 전체를 버리는 것. 사용자 새로고침 의도에는 우회로 충분하다.

Keep
NoopFeedCache 패턴 포트/어댑터가 "기존 테스트 무수정 통과"를 실제로 증명한 날. 인터페이스 먼저 설계하는 습관이 이 순간을 만든다.
에러 코드 원인별 분리 클라이언트가 맥락을 잃지 않게 하는 서버 설계 원칙. 앞으로 새 에러 케이스마다 적용할 기준이 됐다.
Problem
fire-and-forget silent 실패 tag_id 누락 버그가 오래 잠들어 있었던 건 부수 효과 실패를 모니터링하는 체계가 없었기 때문. 중요한 fire-and-forget은 실패 로깅이 필수다.
캐시 키 미래 확장성 {user_id}:{sorted_tag_ids} 포맷은 날짜 범위 등 필터가 추가되면 전면 수정이 필요하다. 지금 한계를 Known Issues로 문서화해야 한다.
Surprise
인메모리 캐시의 단순함 Arc<Mutex<HashMap<String, (Vec<Article>, Instant)>>> 한 줄로 동시성 안전 TTL 캐시가 완성됐다. Rust 타입 시스템이 복잡한 동시성 보일러플레이트를 대신한다.
신규 기능 0개의 날 버그 5건 + 성능 개선 1건만으로 하루가 꽉 찼다. 이런 "완성도 전용" 날이 주기적으로 필요하다는 걸 다시 확인했다.
