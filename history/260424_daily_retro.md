## 핵심 의사결정과 그 이유

### 결정 1: BUG-001 — `emitLocalSessionAsInitialSession: true` 옵션으로 세션 타이밍 수정

**상황**: 앱 첫 실행 시 Supabase 세션 복원이 완료되기 전에 API 요청이 먼저 나가서 401 → 태그 로딩 실패. 콘솔에 경고 메시지가 이미 찍혀 있었다.

```
Initial session emitted after attempting to refresh the local stored session.
To opt-in to the new behavior now, set `emitLocalSessionAsInitialSession: true`
```

**선택지 비교**:
| 방법 | 장점 | 단점 |
|------|------|------|
| `emitLocalSessionAsInitialSession: true` | 단 1줄, Supabase 공식 권장 경로 | 동작 방식을 신뢰해야 함 |
| 세션 준비 이벤트 직접 대기 | 내부 흐름을 직접 제어 | 비동기 타이밍 관리 복잡, 경쟁 조건 여전히 존재 |

**결정**: `emitLocalSessionAsInitialSession: true` + `AppDependenciesTests` 테스트 2개 추가

**왜**: SDK가 직접 안내한 경로였다. 경고 메시지 자체가 "이렇게 고쳐라"였으니 신뢰할 수 있었다.

**인사이트**: 라이브러리 경고 메시지는 대부분 이미 답을 갖고 있다. 복잡하게 우회하기 전에 SDK 문서/경고 메시지를 먼저 읽는 게 맞다.

### 결정 2: BUG-002 — 컴파일 타임 `#if targetEnvironment(simulator)` 분기 선택

**상황**: `Config.xcconfig`의 `SERVER_URL`이 실기기용 LAN IP로 고정돼 있어서, 시뮬레이터 전환 시 API 연결 실패 → 스플래시 무한 대기.

**선택지 비교**:
| 방법 | 장점 | 단점 |
|------|------|------|
| 컴파일 타임 분기 (`#if simulator`) | 잘못된 URL 요청 자체가 불가능 | 두 환경 코드가 항상 빌드됨 |
| 런타임 분기 | 동적 유연성 | 잘못된 URL로 실제 요청이 나갈 위험 |
| `deploy.sh` LAN IP 자동 감지 | 코드 변경 없음 | xcconfig 자동 수정 → git dirty, 부작용 |

**결정**: 컴파일 타임 분기 + `AppBootstrap` enum + `ConfigErrorView` 추가

**왜**: 런타임 분기는 "잘못된 URL로 요청이 나가는 순간"이 존재한다. 컴파일 타임이면 그 가능성 자체를 없앨 수 있다. 그리고 `targetEnvironment(simulator)`는 Swift 표준 관용구라 읽기도 쉽다.

**인사이트**: 환경 분리는 런타임보다 컴파일 타임이 안전하다. 특히 "잘못된 환경으로 요청이 나가는 것"을 막아야 할 때는 컴파일 타임 강제가 맞다.

### 결정 3: Codex step-7 무한 루프 사고 → 방지 규칙 추가

**상황**: step-7 리팩토링 단계에서 Codex가 `xcodebuild test`를 반복 실행하면서 124+ tool uses 루프 발생. 수동으로 중단해야 했다.

**근본 원인**: Codex에게 "분석"이 아니라 "검증(실행)"까지 맡긴 것. 종료 조건이 없는 상태에서 테스트가 실패할 때마다 다시 시도하는 루프에 빠졌다.

**처리**:
- `step-7/SKILL.md`에 무한 루프 방지 섹션 추가 (테스트 실행 명령 전달 금지, 재호출 최대 1회)
- `rules/sub/agents.md §3`에 Codex 3개 제한 항목 추가

**왜**: "같은 문제가 두 번 발생하면 문서가 아니라 기계로 막는다"는 원칙에 따라, 메모리 기록만이 아니라 스킬 파일과 rules 파일을 직접 수정했다.

**인사이트**: 에이전트에게 실행 권한을 줄 때는 반드시 종료 조건을 명시해야 한다. "분석 전용"과 "실행 포함" 역할을 명확히 분리하지 않으면 루프 위험이 생긴다.

## 오늘의 솔직한 어려움

구현 자체는 기획 문서(BUG-001/002 상세 문서)가 미리 작성돼 있어서 흔들리지 않았다. 어려웠던 건 다른 곳이었다.

이번 MVP11이 워크플로우 세팅을 새로 튜닝한 후 처음 진행한 MVP였다. 세팅을 바꿨다고 해서 곧바로 이전처럼 자연스럽게 흘러가는 게 아니었다 — 중간에 Codex 무한 루프 사고가 터졌고, 작업 흐름이 살짝 달라진 부분들을 실시간으로 판단하면서 조정해야 했다. 그 판단 자체가 피로했다.

---

## 인사이트 & 피드백

1. **기획 문서가 있으면 구현은 흔들리지 않는다.** BUG-001/002가 빠르게 끝난 이유는 이미 "왜, 어떻게"가 문서에 적혀 있었기 때문이다. 반대로 Codex 사고는 "어떻게 막을지"가 명시돼 있지 않아서 발생했다.

2. **튜닝은 선언이 아니라 실행 속에서 검증·보정하는 것이다.** 세팅을 바꾼 뒤 "이제 됐겠지"가 아니라, 실제로 써보면서 달라진 부분을 발견하고 다시 수정해야 한다. 이번 MVP가 그 첫 번째 검증 사이클이었다.

3. **에이전트 역할 경계는 미리 그어놔야 한다.** Codex = 분석 전용. 실행(빌드, 테스트)은 Claude가 직접. 이 경계가 흐려지면 루프가 생긴다.
