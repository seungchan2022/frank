01

인증 흐름 vs 서비스 흐름

⚠️
멘토 발표에서 가장 많이 헷갈렸던 부분.
두 흐름을 하나로 섞어서 설명하면 Rust 서버가 왜 등장하는지 이해가 안 됨.
인증은 Supabase SDK가, 데이터 요청은 Rust API가 처리한다.
인증 흐름 — Rust 서버 없음
// 사용자 → 앱 → Apple → Supabase SDK
sequenceDiagram participant U as 사용자 participant App as iOS / 웹 participant Apple as Apple participant SB as Supabase SDK U->>App: Apple로 로그인 버튼 탭 App->>Apple: 인증 요청 Apple-->>App: idToken (iOS) / code (웹) App->>SB: SDK.signIn(idToken) SB-->>App: access_token + refresh_token 발급
  • 인증은 Supabase SDK가 처리 — Rust 서버는 전혀 관여하지 않음
  • 결과물은 access_token (이후 API 호출에 쓸 JWT)
서비스 흐름 — Rust 서버 거침
// 앱 → Rust API → Supabase DB
sequenceDiagram participant App as iOS / 웹 participant Rust as Rust API participant SB as Supabase DB App->>Rust: API 요청 + Bearer access_token Rust->>Rust: JWT 검증 (Supabase 공개키) Rust->>SB: Supabase REST API로 DB 조회 SB-->>Rust: 데이터 반환 Rust-->>App: JSON 응답
  • 앱은 Authorization: Bearer {access_token} 헤더에 토큰을 담아 보냄
  • Rust API는 토큰을 검증하고 Supabase REST API로 DB에 접근
  • 비즈니스 로직은 전부 Rust 안에 존재

MVP2 — 분산된 데이터 접근

  • Supabase SDK 직접: 태그 목록, 기사 목록/상세, 프로필 — 단순 CRUD
  • Rust API: 수집(POST /me/collect), 요약(POST /me/summarize) — 크롤링·AI 처리가 필요한 것만
  • 웹/iOS 양쪽 모두 동일하게 분산 — 비즈니스 로직이 클라이언트에 중복 존재
MVP3 — Rust API 단일 창구
graph LR iOS([iOS 앱]) Web([웹 SvelteKit]) Rust[Rust API] DB[(Supabase DB)] SDK[Supabase SDK] iOS -->|모든 데이터 요청| Rust Web -->|모든 데이터 요청| Rust Rust -->|Supabase REST API| DB iOS -.-|인증만| SDK Web -.-|인증만| SDK style Rust fill:#065f46,stroke:#10b981,color:#f1f5f9 style DB fill:#1e293b,stroke:#3b82f6,color:#94a3b8 style SDK fill:#1e293b,stroke:#64748b,color:#94a3b8
  • 모든 데이터 요청 → Rust API (기사 조회, 태그 관리, 스크랩 등)
  • 인증만 → 여전히 Supabase SDK 직접 사용 (변경 없음)
  • 이유: 비즈니스 로직 한 곳 관리 / Supabase 서비스 키 클라이언트 노출 방지
03

iOS Apple 로그인 — ASAuthorizationController

iOS 인증 흐름 (브라우저 없음)
sequenceDiagram participant U as 사용자 participant iOS as iOS 앱 participant AC as ASAuthorizationController participant Apple as Apple 서버 participant SB as Supabase SDK U->>iOS: Apple로 로그인 탭 iOS->>AC: 실행 (nonce 생성) AC->>Apple: Face ID 확인 + 인증 요청 (+ nonce 해시) Apple-->>AC: idToken + nonce 반환 AC-->>iOS: idToken 전달 iOS->>SB: signInWithIdToken(idToken, nonce) SB-->>iOS: access_token + refresh_token iOS->>iOS: Keychain에 토큰 저장
핵심 개념
  • ASAuthorizationController: Apple이 iOS에 기본 제공하는 "Apple로 로그인" 프레임워크. 브라우저 팝업 없이 Face ID로 직접 처리
  • idToken: Apple이 발급. "이 사용자 진짜 Apple 계정 맞음" 증명. 1회용
  • nonce: 앱이 생성하는 일회용 랜덤 값 → 해시해서 Apple에 보냄 → idToken에 포함되어 돌아옴 → idToken 탈취 후 재사용 방지
✅
웹과의 차이: iOS는 브라우저 리다이렉트 없이 idToken을 직접 받음.
웹은 code를 받고 → Supabase SDK가 토큰으로 교환하는 단계 추가 필요.
04

웹 Apple 로그인 — OAuth + PKCE

PKCE 없을 때 — code 탈취 위험
sequenceDiagram participant Web as 웹 (SvelteKit) participant Apple as Apple participant H as 해커 Web->>Apple: 로그인 요청 Apple-->>Web: /callback?code=abc123 H->>H: code 탈취! H->>Apple: code로 토큰 교환 시도 Apple-->>H: 토큰 발급 (계정 탈취 성공)
PKCE 있을 때 — 처음 요청자만 교환 가능
sequenceDiagram participant Web as 웹 (SvelteKit) participant Apple as Apple participant H as 해커 Web->>Web: code_verifier 생성 → SHA256 해시 → code_challenge Web->>Apple: 로그인 요청 + code_challenge Apple->>Apple: code_challenge 기억해둠 Apple-->>Web: /callback?code=abc123 H->>H: code 탈취! H->>Apple: code만으로 교환 시도 Apple-->>H: code_verifier 없음 → 거부 Web->>Apple: code + code_verifier 제출 Apple->>Apple: SHA256(code_verifier) == code_challenge 확인 Apple-->>Web: 토큰 발급 성공
핵심 개념
  • PKCE (Proof Key for Code Exchange): code를 교환할 때 "처음 요청한 사람"만 가능하도록 보장하는 보안 장치
  • code_verifier: 브라우저(SvelteKit)가 로그인 시작 시 생성하는 랜덤 문자열
  • code_challenge: code_verifier를 SHA256으로 해시한 값 → Apple에 먼저 전달
  • Service ID: 웹 OAuth용 Apple 앱 식별자. iOS의 Bundle ID와 다름
✅
직접 구현 불필요: Supabase SDK가 PKCE 전체를 자동 처리함.
supabase.auth.signInWithOAuth({ provider: 'apple' }) 한 줄로 끝.

JWT / 토큰 종류

토큰 흐름
graph TD Apple([Apple]) SB[Supabase] App([iOS / 웹 앱]) Rust[Rust API] Apple -->|idToken 발급| SB SB -->|access_token JWT| App SB -->|refresh_token| App App -->|Bearer access_token| Rust App -->|만료 시 refresh_token| SB SB -->|새 access_token| App style Apple fill:#1e293b,stroke:#64748b,color:#94a3b8 style SB fill:#065f46,stroke:#10b981,color:#f1f5f9 style Rust fill:#7c3aed,stroke:#8b5cf6,color:#f1f5f9 style App fill:#1e3a5f,stroke:#3b82f6,color:#f1f5f9
토큰 비교
토큰 발급 주체 역할 수명
idToken Apple 신원 증명. Supabase 로그인할 때만 사용 후 폐기 1회용
access_token Supabase API 호출용 JWT. Bearer 헤더에 담아서 Rust에 전달 짧음 (1시간)
refresh_token Supabase access_token 만료 시 새 것 발급받는 열쇠 김 (수 주~수 개월)
암호화 vs 서명 (JWT)
  • 암호화: 내용을 숨김. 키 없으면 읽을 수 없음
  • 서명 (JWT): 내용은 읽을 수 있음. 변조하면 서명이 안 맞아서 서버가 거부.
    → Rust API는 내용을 읽고 변조 여부만 확인
💡
nonce 복습: idToken 안에 포함된 일회용 랜덤 값.
같은 idToken을 탈취해서 다시 쓰려 해도 nonce가 맞지 않아서 Supabase가 거부.

저장소 비교 — httpOnly 쿠키 vs Keychain

XSS 공격 — localStorage 취약점
sequenceDiagram participant H as 해커 participant Web as 웹페이지 participant LS as localStorage H->>Web: 악성 스크립트 심기 (XSS) Note over Web: 다른 사용자가 페이지 접속 Web->>LS: localStorage.getItem('token') LS-->>Web: 토큰 반환 Web->>H: 토큰 전송 → 계정 탈취 성공
🚫
localStorage는 JS로 접근 가능 → XSS 공격에 토큰 탈취 위험.
access_token을 localStorage에 저장하면 안 됨.

쿠키 vs Keychain 개념 정리
저장소 플랫폼 JS 접근 특징
쿠키 웹 가능 (기본) 브라우저가 관리. 매 요청마다 자동 첨부. 브라우저마다 독립
httpOnly 쿠키 웹 불가능 서버가 Set-Cookie 헤더로 생성. JS에서 읽기 불가 → XSS 방어
localStorage 웹 가능 JS로 자유롭게 읽기/쓰기. XSS 취약
Keychain iOS 해당없음 OS 레벨 암호화. 앱 삭제해도 유지. 다른 앱 접근 불가
UserDefaults iOS 해당없음 단순 키-값 저장. 암호화 없음 → 민감 정보 저장 금지
✅
Frank에서의 결론:
웹 → httpOnly 쿠키에 access_token 저장 (서버가 Set-Cookie로 관리)
iOS → Keychain에 access_token + refresh_token 저장
