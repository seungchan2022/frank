# 서브에이전트 사용 규칙 (v1.0)

## 1) 목적
Task tool을 통한 서브에이전트 사용 시 프로젝트 규칙 준수를 강제하고, 일관된 품질을 유지한다.

## 2) 적용 범위
서브에이전트 사용이 권장되는 시나리오:
- **병렬 테스트 실행**: 모듈별 독립 테스트
- **대규모 리팩토링**: 10개 이상 파일에 걸친 패턴 추출 또는 구조 변경
- **문서 자동 생성**: Progress 문서, 테스트 리포트, API 문서
- **품질 검증 자동화**: 린트 + 타입체크 + 아키텍처 규칙 병렬 검증

서브에이전트 사용이 **금지**되는 시나리오:
- 룰북 수정 (rules/0_CODEX_RULES.md, rules/sub/*.md)
- 프로덕션 배포 (수동 검증 필수)
- 시크릿 관리 (env, credentials)
- 데이터베이스 DDL 변경 (수동 검토 필수)

## 3) 에이전트 유형별 사용 규칙

### 3.1 에이전트 유형 매트릭스

| 작업 유형 | 권장 에이전트 | 모드 | 사용 시나리오 |
|----------|--------------|------|--------------|
| 테스트 실행 | `quality-engineer` | delegate | 모듈별 테스트 실행, 커버리지 검증 |
| 코드 품질 검증 | `quality-engineer` | delegate | 린트/타입체크/아키텍처 규칙 병렬 검증 |
| 리팩토링 계획 | `refactoring-expert` | **plan** | 10개+ 파일 패턴 추출, 중복 코드 제거 |
| 리팩토링 실행 | `refactoring-expert` | delegate | 승인된 계획 기반 코드 수정 |
| 아키텍처 검토 | `backend-architect` | **plan** | 계층 의존성 검증, 설계 개선안 |
| 성능 분석 | `performance-engineer` | delegate | 병목 식별, 최적화 제안 |
| 문서 생성 | `technical-writer` | delegate | Progress 문서, API 문서 |
| 버그 분석 | `root-cause-analyst` | delegate | 복잡한 버그의 근본 원인 분석 |
| 보안 검토 | `security-engineer` | **plan** | 취약점 스캔, 보안 규칙 준수 확인 |
| 코드베이스 탐색 | `Explore` | delegate | 파일/패턴 검색, 구조 파악 |

### 3.2 모드 선택 기준

**plan 모드 필수** (아키텍처/구조 변경):
- 10개 이상 파일 수정
- 공통/shared 모듈 변경
- 새로운 패턴/추상화 도입
- 계층 구조 변경
- 보안 관련 변경

**delegate 모드 허용** (실행/검증 작업):
- 테스트 실행 및 결과 수집
- 린트/타입체크 실행
- 문서 생성
- 버그 분석 (수정 제외)
- 코드베이스 탐색

### 3.3 금지된 에이전트-작업 조합

| 에이전트 | 금지 작업 | 사유 |
|---------|----------|------|
| 모든 에이전트 | 룰북 수정 (rules/*) | 수동 검토 필수 |
| 모든 에이전트 | 프로덕션 배포 | 수동 검증 필수 |
| 모든 에이전트 | 시크릿/credentials 처리 | 보안 위험 |
| delegate 모드 | 아키텍처 변경 | plan 모드 필수 |
| Explore | 코드 수정 | 탐색 전용 |

## 4) 결과 검증 체크리스트

서브에이전트 작업 완료 후 반드시 확인:

- [ ] 테스트 통과
- [ ] 린트/타입체크 통과
- [ ] 프로젝트 아키텍처 패턴 준수
- [ ] 계층 의존성 위반 없음
- [ ] 에러 처리 패턴 준수

## 5) 금지 사항

### 5.1 절대 금지
- **서브에이전트에게 룰북 수정 권한 부여**: rules/0_CODEX_RULES.md, rules/sub/*.md 변경 금지
- **프로덕션 배포 자동화**: 수동 검증 없이 배포 금지
- **시크릿 자동 생성/수정**: .env, credentials, API keys 자동 처리 금지
- **DDL 자동 실행**: 데이터베이스 스키마 변경은 수동 검토 필수

### 5.2 주의 필요
- **아키텍처 변경**: 반드시 plan 모드로 먼저 승인 받기
- **공통 모듈 변경**: shared/common 모듈 변경 시 영향 범위 확인 필수
- **테스트 스킵**: 검증 체크리스트 생략 금지
