# 개발 사고 가이드 (Thinking Guide)

이 문서는 본 저장소의 **9단계 워크플로우** (step-1~step-9, 진입점 `.claude/skills/workflow/SKILL.md`) 내부에서 참고하는 사고 가이드다. 별도의 단계 시스템이 **아니다**.

(Codex 내부 사고 모델인 5단계 — Inspect → Specify → Implement → Verify → Report — 도 본 가이드 내용을 자연스럽게 충족한다.)

---

## 적용 범위

- 새 기능 추가 / 기존 동작 변경 / 버그 수정 / API·비즈니스 로직·데이터 흐름 변경 시 **반드시** 참고
- 문서 수정 / 포맷팅 / 주석 / 설정 값 변경만 있는 경우 간소화 가능

---

## Progress 문서 (권장)

비단순 신규 작업 시 `progress/` 아래에 작업 문서를 생성하여 git에 커밋한다.
- 파일명: `{number}_<topic>.md` 또는 `{YYMMDD}_<topic>.md`
- 기록 항목: 작업 목표·범위, 결정 사항(무엇을/왜), 논의된 대안과 선택 이유, 남은 리스크·후속 작업

---

## Inspect (분석) 단계에서의 사고

- 해결하려는 문제를 **한 문장**으로 정리한다
- 영향받는 코드 영역(모듈·함수·API)을 식별한다
- 기존 동작과 기대 동작의 차이를 구분한다
- 아직 해결책을 코드로 옮기지 않는다
- active MVP의 `## KPI` 섹션을 확인하여 이 변경이 영향을 줄 지표를 파악한다

---

## Specify (테스트 우선) 단계에서의 사고

- 기대 동작을 작고 검증 가능한 단위로 분해한다
- 가능하면 단위 테스트 우선, 최소 1개 이상의 실패 케이스(엣지·권한 없음·잘못된 입력) 포함
- 테스트는 구현 전에 반드시 실패해야 한다

---

## Implement (구현) 단계에서의 사고

- 테스트를 만족시키는 **가장 단순한 구현**을 우선한다
- 불필요한 리팩터링·구조 변경 금지 (§3에서 별도 요청이 없는 한)
- 새로운 동작은 반드시 테스트로 설명되어야 한다

---

## Verify (검증) 단계에서의 사고

- 테스트·린트·타입체크 실행 결과를 신뢰 가능한 1차 판단 기준으로 삼는다
- active MVP가 Hard KPI를 선언했다면 `scripts/kpi-check.sh`로 현재값 확인
- UI/웹 변경은 Chrome DevTools MCP로 스크린샷·네트워크 검증 (svelte-scope 기준)
- DB 변경은 Supabase MCP `get_advisors`로 정적 점검

---

## Report (보고) 단계에서의 사고

- 변경 요약(1~3문장), 수정 파일 목록, 실행한 테스트 명령과 결과, 스모크 테스트 여부, 남은 리스크
- PR 생성 시: 무엇을·왜·어떻게 테스트했는지 포함

---

## 셀프 리뷰 체크리스트

코드를 PR 리뷰하듯 다시 읽을 때 확인한다.
- 이 변경이 꼭 필요한가?
- 코드가 요구사항을 정확히 충족하는가?
- 테스트가 의도를 충분히 설명하는가?
- 보안(입력 검증·인증·인가·정보 노출)에 문제는 없는가?
- 불필요하게 복잡한 로직은 없는가?
- 미래의 자신이나 동료가 이해할 수 있는가?

---

## 중단·재검토 조건

다음 중 하나라도 해당하면 **Inspect 단계로 되돌아간다**:
- 테스트로 명확히 표현할 수 없는 요구사항 발견
- 기존 동작을 충분히 이해하지 못한 상태
- 변경의 영향 범위를 명확히 설명할 수 없는 상태
- active MVP의 KPI 중 Hard 지표가 미달 상태인데 근본 원인이 파악되지 않은 경우

---

## 작업 범위 원칙

- 한 작업은 하나의 문제·목적에 집중한다
- 변경 범위가 커지면 작업을 분리한다 (커밋 단위도 플랫폼별/관심사별로 분리 — `feedback_commit_granularity`)
- 여러 관심사가 섞이기 시작하면 Inspect 단계로 돌아간다

---

## 자동화 우선 원칙

- 사람이 판단해야 할 부분을 제외하고는 자동 검증을 우선한다
- 테스트·린트·CI·KPI 게이트·MCP 기반 검증(스크린샷·DB advisors)을 1차 판단 기준으로 삼는다
- 수동 확인이 필요한 경우 그 이유를 보고한다

---

## 원칙 요약

- 분석 없는 구현 금지
- 테스트로 설명되지 않는 동작 추가 금지
- 큰 변경은 작은 단계로
- 코드보다 의도가 먼저 드러나야 한다
- 마일스톤이 선언한 KPI는 커밋을 가로막는 **실제 게이트**다
