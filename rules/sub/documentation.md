# 문서화 규칙 (v1.0)

이 문서는 progress/ 문서 구조, 서브태스크 문서 형식, 토론 로그 형식을 정의한다.

---

## 1. progress/ 문서 구조

### 1.1 디렉터리 구조

```
progress/
├── milestones/                          # 마일스톤 플로우 산출물
│   └── {프로젝트명}/
│       ├── _vision.md                   # 비전 + 씨앗 인터뷰 결과
│       ├── _discovery.md                # 영감 보드 + 아이디어 풀 + 수렴 결과
│       ├── _roadmap.md                  # 로드맵 (타임라인+의존성)
│       └── M{N}_{마일스톤명}.md          # 마일스톤 상세 (아이템+유형 태그)
├── {YYMMDD}_{메인태스크명}.md           # 메인태스크 문서
├── subtask/
│   └── {YYMMDD}_{메인태스크명}/
│       ├── {넘버링}-{제목}.md           # 서브태스크 문서
│       └── ...
└── debate/
    └── {YYMMDD}_{단계}_{주제}.md       # 토론 로그
```

> 마일스톤 문서 상세 형식은 `rules/sub/milestone.md` 참조.

### 1.2 파일명 규칙
- 날짜 형식: `YYMMDD` (예: 260209)
- 한글/영문 혼용 가능
- 공백 대신 언더스코어(`_`) 사용

## 2. 메인태스크 문서 형식

```markdown
# {메인태스크명}

> 생성일: {YYMMDD}
> 상태: 진행중 | 완료

## 요약
{요약 내용}

## 상세 내용
{인터뷰 결과 반영}

## 서브태스크
| # | 서브태스크 | 의존성 | 상태 |
|---|-----------|--------|------|
| 1 | {태스크명} | 없음 | 대기 |
| 2 | {태스크명} | 1 | 대기 |

## 워크플로우 설정
### 자동 토론
| 단계 | 상태 |
|------|------|
| step-1 | OFF |
| step-3 | OFF |
| step-5 | OFF |
| step-7 | OFF |

토론 깊이: standard
```

## 3. 서브태스크 문서 형식

```markdown
# {서브태스크 제목}

> 상위: {메인태스크명}
> 번호: {넘버링}
> 상태: 대기 | 진행중 | 완료
> 유형: 버그수정 | 문서업데이트 | 추가기능 | 리팩토링 | 설정변경

---

## 스코프

### In-Scope
- {포함 항목}

### Out-of-Scope
- {제외 항목}

---

## Definition of Done (DoD)

### 코드 품질
- [ ] 린트 통과
- [ ] 타입 체크 통과
- [ ] 테스트 통과

### 문서/프로세스
- [ ] 인/아웃 스코프 명확
- [ ] 문서와 코드 일치
- [ ] 리뷰 완료 (Claude + Codex)
- [ ] 사용자 승인

---

## 상세 구현 명세

### 변경 파일
| 파일 | 변경 유형 | 설명 |
|------|----------|------|
| `src/xxx.py` | 신규 | {설명} |

### 구현 상세
{상세 구현 내용}

---

## 테스트 계획
| 테스트 | 설명 | 유형 |
|--------|------|------|
| {테스트1} | {설명} | 단위 |

---

## 리뷰 결과

### Claude 리뷰
- {검토 결과}

### Codex 리뷰
- {검토 결과}

### 최종 결정
- {승인/조건부 승인/반려}
```

## 4. 토론 로그 형식

```markdown
# 토론 로그: {주제}

> 일시: {YYMMDD}
> 단계: step-{N}
> 참여자: Claude, Codex, Serena

## 주제
{토론 주제와 맥락}

## 가설

### Claude
{사용자 의도 관점 가설}

### Codex
{기술적 타당성 관점 가설}

### Serena
{구조적 영향 관점 가설}

## 반박
{상호 반박 내용}

## 합의
{공통 동의 사항}

## 결론
{채택된 접근법과 이유}
```

## 5. 다이어그램 작성 규칙

- 플로우차트/아키텍처 다이어그램은 **D2 MCP** 또는 **Mermaid MCP**로 작성한다.
- D2 언어로 다이어그램 코드를 작성하면 자동 레이아웃으로 SVG/PNG를 렌더링한다.
- 산출물: 소스 파일 + 렌더링된 다이어그램

## 6. 원칙

- progress 문서는 git에 커밋하여 추적한다.
- 결정/근거/리스크를 작업 기록으로 남긴다.
- 문서만 변경 시 테스트 생략 가능 (사유 명시 필수).
- 구현 단계에서는 문서에 명시되지 않은 기능은 절대 개발 금지.
