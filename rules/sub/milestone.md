# 마일스톤 규칙 (v2.0)

이 문서는 마일스톤 플로우의 핵심 원칙, 문서 형식, 아이템 유형, 라우팅 규칙을 정의한다.

---

## 1. 핵심 원칙: Discovery-First

- 마일스톤 플로우의 목적은 **"무엇을 만들지 함께 발견하는 것"**이다
- 탐색과 브레인스토밍이 전체 시간의 **60%**를 차지해야 한다
- 사용자가 "뭘 만들지 모르는 상태"에서 시작하는 것이 정상이다
- MCP 웹 리서치(Tavily, Exa, Firecrawl)는 **필수** 단계이다 (선택 아님)

---

## 2. 계층 구조

```
비전(Vision)
 └→ Discovery (영감 보드 + 아이디어 풀 + 수렴 결과)
     └→ 로드맵(Roadmap)
         └→ 마일스톤(Milestone)
             └→ 아이템(Item) — 유형에 따라 다른 스킬로 실행
                 ├─ feature  → /workflow 메인태스크
                 ├─ research → /deep-analysis
                 ├─ decision → /debate
                 ├─ chore    → 직접 실행
                 └─ spike    → 탐색적 구현
```

- 비전은 프로젝트당 1개
- Discovery 문서는 프로젝트당 1개 (영감 보드 + 아이디어 풀 보존)
- 로드맵은 프로젝트당 1개 (갱신 가능)
- 마일스톤은 로드맵 내 N개
- **아이템이 전부 메인태스크가 되지 않는다** — feature 유형만 /workflow로 간다

---

## 3. 아이템 유형과 라우팅

| 유형 | 설명 | 실행 스킬 | 산출물 |
|------|------|----------|--------|
| `feature` | 구현이 필요한 기능 | `/workflow` | 메인태스크 → 서브태스크 → 코드 |
| `research` | 조사/분석이 필요한 것 | `/deep-analysis` | `progress/analysis/` 보고서 |
| `decision` | 의사결정이 필요한 것 | `/debate` | 의사결정 기록 |
| `chore` | 설정/인프라/단순 작업 | 직접 실행 | 커밋 1~2개 수준 |
| `spike` | 기술 검증 프로토타입 | 탐색적 구현 | 프로토 + 판단 기록 → feature 전환 또는 폐기 |

### 라우팅 규칙
- 아이템 유형은 마일스톤 문서의 아이템 테이블에 명시
- feature 아이템만 `/workflow`의 메인태스크로 전환된다
- spike는 결과에 따라 feature로 승격되거나 폐기된다
- chore는 별도 워크플로우 없이 바로 실행 + 커밋

## 4. 디렉토리 규칙

```
progress/
└── milestones/
    └── {프로젝트명}/              # snake_case, 한글 허용
        ├── _vision.md             # 접두사 _ = 메타 문서
        ├── _discovery.md          # 영감 보드 + 아이디어 풀 + 수렴 결과
        ├── _roadmap.md
        ├── M1_{마일스톤명}.md      # M + 숫자 + _ + 이름
        ├── M2_{마일스톤명}.md
        └── ...
```

### 네이밍 규칙

| 항목 | 규칙 | 예시 |
|------|------|------|
| 프로젝트명 | snake_case, 한글 허용 | `frank_mvp2`, `뉴스앱` |
| 마일스톤 파일 | `M{N}_{이름}.md` | `M1_개인화_학습.md` |
| 메타 문서 | `_` 접두사 | `_vision.md`, `_discovery.md`, `_roadmap.md` |

## 5. 상태 전이

### 마일스톤 상태

```
대기 → 진행중 → 완료
         ↓
       보류 → 진행중 (재개)
         ↓
       취소
```

### 아이템 상태

```
대기 → 진행중 → 완료
                 ↓
               폐기 (spike가 실패한 경우)
```

---

## 6. 로드맵 갱신 규칙

### 갱신 시점
- 마일스톤 완료 시
- 마일스톤 리뷰(`/milestone-review`) 시
- 스코프 변경 발생 시
- spike 결과에 따른 아이템 유형 변경 시

### 갱신 필수 항목
1. `_roadmap.md`의 타임라인 테이블 업데이트
2. 변경 이력에 기록 추가 (날짜 + 변경 내용 + 사유)
3. 의존성 그래프 재생성 (변경된 경우)

---

## 7. 마일스톤 크기 가이드

| 규모 | 마일스톤 수 | 마일스톤당 기간 | 아이템 수 |
|------|-----------|---------------|----------|
| 소규모 (1~2주) | 2~3개 | 3~5일 | 2~4개 |
| 중규모 (1~2개월) | 4~8개 | 1~2주 | 3~7개 |
| 대규모 (3개월+) | 8~15개 | 1~3주 | 5~10개 |

### 분할 기준
- 예상 기간 3주 초과 시 분할 권장
- 아이템 7개 초과 시 분할 권장
- 기술적 불확실성이 높은 부분은 별도 spike 아이템으로 분리

## 8. 리스크 관리

| 영향 | 의미 | 대응 수준 |
|------|------|----------|
| H (High) | 마일스톤 실패 가능 | 대응 계획 필수, 매 리뷰 시 확인 |
| M (Medium) | 일정 지연 가능 | 대응 계획 권장 |
| L (Low) | 경미한 영향 | 인지만 |

---

## 9. 두 가지 진입점

| 진입점 | 씨앗 인터뷰 | 탐색 범위 |
|--------|-----------|----------|
| 신규 (제로 → MVP1) | 열린 탐색: 불편, 영감, 이상적 모습 | 넓음: 도메인 전체 |
| 진화 (MVP{N} → MVP{N+1}) | 회고 + 탐색: 아쉬운 점, 새 욕구, 영감 | 기존 코드/문서 + 새로운 가능성 |

진화 진입점에서는 기존 마일스톤 문서, 코드베이스, Git 히스토리를 자동 분석하여 컨텍스트를 먼저 확보한다.

---

## 10. 원칙 요약

- Discovery가 핵심이다 — 탐색 없는 마일스톤은 "할 일 정리"에 불과하다
- 마일스톤은 **라우터**다 — 아이템 유형에 따라 적절한 스킬로 분배한다
- feature만 /workflow로 간다 — 모든 아이템에 9단계를 강제하지 않는다
- 로드맵은 살아있는 문서다 — 변경을 두려워하지 말 것
- 비전은 안정적이어야 한다 — 자주 바뀌면 프로젝트가 흔들린다
