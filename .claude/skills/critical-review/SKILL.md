# 구멍 찾기 리뷰 (/critical-review)

> **목표**: 지금 잘 돌아가는 것 같지만 실제로는 무너질 지점을 **적대적·비판적 시각**으로 선제 발굴.
> 구현 전 단계일수록 ROI가 높다 (종이 수정 vs 벽 부수기).

## 일반 리뷰와의 차이

| 스킬 | 성격 | 발견 시점 |
|---|---|---|
| `/review` (플러그인) | 중립적 PR 리뷰 (정상 경로 확인) | 구현 후 |
| `/deep-analysis` | 구조·아키텍처 이해 중심 | 언제든 |
| **`/critical-review` (본 스킬)** | **적대적 시각** (숨은 함정 발굴) | **구현 전이 최적** |

## 호출 방법

```
/critical-review {대상}
```

대상 예시:
- 서브태스크 분리 결과 (step-5 자동 호출)
- 마일스톤 기획 문서 (M{X}_*.md)
- 하네스·룰 개편 직후
- 설계 초안·아키텍처 스케치
- 최근 구현 코드 섹션 (선택적)

## [2] Codex 적대적 리뷰 (핵심)

Codex에게 **다음 프롬프트**로 의뢰:

> "이 문서/코드는 겉보기에 문제없어 보인다. 하지만 실제로 배포·운영하면 뭐가 터질까?
> - 놓친 엣지 케이스 (실패 경로·권한 없음·부분 실패·동시성)
> - 구조적 결함 (의존 역전·계층 위반·순환·단일 실패점)
> - 기획 전제의 함정 (잘못된 가정·범위 오인·사용자 실사용 패턴 불일치)
> - 외부 충격 시나리오 (API 응답 변경·플랫폼 정책·네트워크 장애)
> - 문서와 실제의 불일치 (룰북은 X라 하지만 실제는 Y)
>
> 각 구멍에 대해: (a) 발생 조건 (b) 영향도 (c) 수정 비용.
> 칭찬·보완 제안 금지. 구멍만 적나라하게."

병렬로 Claude도 같은 관점으로 자체 검토 (자기검열 없이).

## [3] 외부 시너지 탐색 (선택)

다음 상황에서만:
- 하네스·인프라 개편 대상일 때 → 웹서치로 "같은 문제를 다른 팀이 어떻게 풀었나" 확인
- 새 기술 도입 관련일 때 → context7·tavily로 최신 베스트 프랙티스
- 코드 구조 관련일 때 → serena로 내부 유사 패턴 탐색

불필요하면 스킵.

## [4] 구멍 분류 + 대안

각 구멍을 **영향도 × 발생확률**로 분류:

| 우선순위 | 기준 |
|---|---|
| **치명** | 배포 차단 · 데이터 손실 · 보안 · 시스템 전체 무력화 가능 |
| **중대** | 특정 시나리오에서 실패 · 성능 저하 · UX 파손 |
| **경미** | 엣지 케이스 · 개선 여지 · 문서 불일치 |

**각 구멍에 반드시 대안 동반** (비판만 ❌):
- 즉시 실행 가능한 최소 수정
- 근본 해결 (시간 비용 명시)
- 둘 중 권장안 선택

## [5] 우선순위 표 + 사용자 확인

```markdown
## 구멍 리뷰 결과: {대상명}

### 치명 (N건)
| # | 구멍 | 발생 조건 | 대안 | 비용 |
|---|---|---|---|---|
| C1 | {설명} | {조건} | {대안} | 저/중/고 |

### 중대 (N건)
...

### 경미 (N건)
...

## 제안 실행 순서
1. 치명 C1 (30분): {즉시 조치}
2. 중대 M1 (1~2시간): {수정안}
3. 경미 T1: 후속 과제로 기록

진행할 것은 어떤 것인지 알려주세요.
```

## 원칙

1. **자기 검열 없음** — "내가 만든 거라 좋아 보이지만 실제로는..." 강제
2. **칭찬 금지** — 잘된 점은 리뷰 대상이 아님
3. **사용자 실사용 패턴 우선** — 원론적 비판보다 이 사용자/이 프로젝트에서 실제 문제인지 판단
4. **대안 없는 비판 금지** — 구멍만 나열하면 가치 없음
5. **우선순위 필수** — 전부 고칠 수는 없으니 순서 제시
