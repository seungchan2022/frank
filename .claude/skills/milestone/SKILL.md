# 마일스톤 플로우 (/milestone)

> **Discovery-First 전략 도구.**
> "뭘 만들지 같이 찾는" 플로우. 탐색과 브레인스토밍이 핵심이다.
> 각 마일스톤의 아이템은 유형에 따라 적절한 스킬로 라우팅된다.

## 핵심 철학

```
마일스톤 ≠ "사용자가 할 일을 알려주면 정리해주는 도구"
마일스톤 = "무엇을 만들지 함께 발견하는 도구"
```

전체 시간의 **60%는 탐색과 브레인스토밍**, 40%가 정리와 구조화다.

---

## 워크플로우와의 관계

```
/milestone (발견 + 전략)
─────────────────────────
비전 → 탐색 → 브레인스토밍 → 수렴 → 로드맵
                                        ↓
                               마일스톤 아이템들
                                ├─ feature  → /workflow
                                ├─ research → /deep-analysis
                                ├─ decision → /debate
                                └─ chore    → 직접 실행
```

- `/milestone`: "무엇을 만들지 발견하고 우선순위를 정한다" (발견 + 전략)
- `/workflow`: "결정된 기능을 어떻게 구현할지 실행한다" (전술)
- **마일스톤의 모든 아이템이 /workflow로 가지 않는다** — 유형에 따라 분배

---

## 두 가지 진입점

### A. 신규 프로젝트 (제로 → MVP1)

```
/milestone "AI 뉴스 스터디앱"
```

씨앗 인터뷰가 **열린 탐색** 중심.

### B. 기존 프로젝트 진화 (MVP1 → MVP2)

```
/milestone "frank MVP2"
```

씨앗 인터뷰가 **회고 + 탐색** 중심. 기존 코드/문서를 자동 분석하여 컨텍스트를 먼저 수집한다.

---

## 실행 흐름

```
/milestone {프로젝트 설명}
       ↓
[1] 씨앗 인터뷰 (열린 질문, 3~5개)
       ↓
[2] 탐색 발산 (MCP 웹 리서치) ← 핵심! 전체의 30%
       ↓
[3] 아이디어 브레인스토밍 (3자 토론) ← 핵심! 전체의 30%
       ↓
[4] 수렴: 우선순위 매기기
       ↓
[5] 로드맵 + 마일스톤 구성
       ↓
[6] 산출물 생성
       ↓
→ 아이템별 라우팅 안내
```

## [1] 씨앗 인터뷰

**목적**: 탐색의 방향을 잡기 위한 최소한의 씨앗을 얻는다. 답을 정하는 게 아니라 **탐색 범위를 좁히는 것**.

1문1답, 열린 질문 중심. A/B/C 선택지는 최소화.

### 신규 프로젝트일 때

1. **불편/욕구**: "어떤 불편함이나 필요를 느끼고 있나요?"
2. **영감**: "영감을 받은 서비스나 경험이 있나요?"
3. **이상적 모습**: "이게 완성되면 어떤 모습이면 좋겠어요?"

### 기존 프로젝트 진화일 때

먼저 자동으로 수행:
- `progress/` 기존 마일스톤/기획 문서 로드
- 코드베이스 구조 분석 (Serena)
- 최근 Git 히스토리 요약

그 후 질문:
1. **회고**: "현재 버전에서 가장 아쉬운 점은?"
2. **욕구**: "다음 버전에서 꼭 해보고 싶은 것은?"
3. **영감**: "최근에 본 서비스 중 '이거 넣고 싶다'고 느낀 게 있나요?"

### 규칙
- 질문은 **3~5개**로 제한 (인터뷰 피로 방지)
- 사용자가 "모르겠다"고 하면 **그게 정상** — [2]단계에서 같이 찾으면 된다
- 핵심은 **탐색 키워드**를 얻는 것 (정확한 기능 명세가 아님)

---

## [2] 탐색 발산 (MCP 웹 리서치)

> **이 단계가 마일스톤 플로우의 존재 이유다.**

씨앗 인터뷰에서 얻은 키워드를 기반으로 **폭넓게 탐색**한다.

### 자동 수행 항목 (병렬)

| 탐색 영역 | MCP 도구 | 수집 대상 |
|-----------|----------|----------|
| 유사 서비스 | Tavily, Exa | 경쟁/유사 서비스 10개+, 각각의 핵심 기능 |
| 트렌드 | Tavily | 해당 도메인의 최신 기술/UX 트렌드 |
| 오픈소스 | Exa | 관련 오픈소스 프로젝트, 라이브러리 |
| UX 패턴 | Firecrawl | 참고할 만한 UI/UX 디자인 패턴 |
| 기술 가능성 | Tavily | 적용 가능한 새로운 기술 (API, AI 모델 등) |

### 기존 프로젝트 진화일 때 추가

| 탐색 영역 | 도구 | 수집 대상 |
|-----------|------|----------|
| 기존 코드 확장점 | Serena | 현재 아키텍처에서 쉽게 확장 가능한 방향 |
| 기존 기획 문서 | Read | 이전 마일스톤에서 Out-of-Scope로 빠진 아이템 |

### 산출물: 영감 보드

탐색 결과를 **영감 보드**로 정리하여 사용자에게 보여준다:

```markdown
## 영감 보드

### 유사 서비스 분석
| 서비스 | 핵심 기능 | 우리에게 영감을 주는 점 |
|--------|----------|----------------------|
| {서비스A} | {기능} | {인사이트} |
| {서비스B} | {기능} | {인사이트} |

### 기술 트렌드
- {트렌드1}: {설명} → 적용 가능성: {H/M/L}
- {트렌드2}: {설명} → 적용 가능성: {H/M/L}

### 발견한 가능성
- {아이디어1}: {설명}
- {아이디어2}: {설명}
- {아이디어3}: {설명}
```

### 규칙
- 최소 **3개 MCP 도구**를 병렬로 호출
- 유사 서비스는 **10개 이상** 수집 목표
- 사용자가 언급하지 않은 가능성도 적극적으로 발굴
- 영감 보드를 보여주고 **사용자 반응을 확인**한 후 다음 단계로

## [3] 아이디어 브레인스토밍 (3자 토론)

> 영감 보드를 기반으로 **"이런 것도 가능하다"**를 폭발적으로 발산한다.

### 참여자와 역할

| 참여자 | 역할 | 관점 |
|--------|------|------|
| Claude | 사용자 옹호자 | "사용자가 진짜 원하는 건 뭐지?" |
| Codex | 기술 탐험가 | "기술적으로 이런 것도 가능하다" |
| Serena | 현실주의자 | "현재 코드베이스에서 이건 쉽고, 저건 어렵다" |

### 브레인스토밍 기법

1. **SCAMPER 적용**: 영감 보드의 각 아이디어에 대해
   - Substitute(대체), Combine(결합), Adapt(적용), Modify(변형)
   - Put to other use(다른 용도), Eliminate(제거), Reverse(뒤집기)

2. **"Yes, and..." 확장**: 각 아이디어를 부정하지 않고 확장
   - "북마크 기능 + 그리고 스페이스드 리피티션까지?"
   - "요약 기능 + 그리고 퀴즈 자동 생성까지?"

3. **와일드카드 아이디어**: 현실성 무시하고 "만약에..." 아이디어 3개 이상

### 산출물: 아이디어 풀

```markdown
## 아이디어 풀 ({N}개)

| # | 아이디어 | 출처 | 흥미도 | 실현성 |
|---|---------|------|--------|--------|
| 1 | {아이디어} | 유사서비스A 영감 | ★★★ | ★★☆ |
| 2 | {아이디어} | SCAMPER-Combine | ★★☆ | ★★★ |
| 3 | {아이디어} | 와일드카드 | ★★★ | ★☆☆ |
```

### 규칙
- 아이디어는 **최소 15개** 이상 도출
- 이 단계에서는 **판단하지 않는다** (수렴은 [4]에서)
- 사용자에게 "이 중에 끌리는 게 있나요?"로 반응 확인
- 사용자의 "그거!" 모먼트를 유도하는 것이 목표

## [4] 수렴: 우선순위 매기기

> 아이디어 풀에서 **이번 마일스톤에 넣을 것**을 사용자와 함께 결정한다.

### Impact/Effort 매트릭스

```
        높은 임팩트
            │
   Quick Win │ Big Bet
   (바로 넣기) │ (핵심 기능)
  ────────────┼────────────
   Fill-in    │ Money Pit
   (여유 시)   │ (보류/제외)
            │
        낮은 임팩트

← 낮은 노력     높은 노력 →
```

### 분류 프로세스

1. 아이디어 풀의 각 아이디어를 매트릭스에 배치
2. 사용자와 함께 3개 버킷으로 분류:
   - **이번에 넣을 것** (In): Quick Win + Big Bet 중 선택
   - **다음에 할 것** (Next): 좋지만 이번엔 아닌 것
   - **안 할 것** (Out): 임팩트 낮거나 비용 과다

3. "이번에 넣을 것"에 대해 **아이템 유형** 결정:

| 유형 | 설명 | 실행 스킬 |
|------|------|----------|
| `feature` | 구현이 필요한 기능 | `/workflow` (메인태스크 → 서브태스크) |
| `research` | 조사/분석이 필요한 것 | `/deep-analysis` |
| `decision` | 의사결정이 필요한 것 | `/debate` |
| `chore` | 설정/인프라 등 단순 작업 | 직접 실행 |
| `spike` | 기술 검증 (프로토타입) | 탐색적 구현 후 판단 |

### 규칙
- 사용자가 최종 결정 — AI는 제안만
- "이번에 넣을 것"이 너무 많으면 경고: "마일스톤당 아이템 7개 이하 권장"
- 각 아이템에 유형 태그 필수

## [5] 로드맵 + 마일스톤 구성

"이번에 넣을 것"을 마일스톤으로 묶는다.

### 묶기 기준
- **독립적 가치**: 마일스톤 완료 시 사용 가능한 것이 있는가
- **의존성**: 기술적으로 선후 관계가 있는 아이템끼리
- **적정 크기**: 마일스톤당 1~2주 (아이템 3~7개)

### `sequential_thinking`으로 구조화

1. 아이템 간 의존성 분석
2. 마일스톤 경계 결정
3. 순서 배치 (단순 → 복잡, 인프라 → 기능 → 품질)
4. 사용자에게 초안 제시 후 확정

### 로드맵 초안 포맷

```
## 로드맵 초안

M1: {마일스톤명}                    [1주]
 ├─ [feature] 아이템A              → /workflow
 ├─ [chore] 아이템B                → 직접 실행
 └─ [decision] 아이템C             → /debate

M2: {마일스톤명}                    [2주]  ← 의존: M1
 ├─ [feature] 아이템D              → /workflow
 ├─ [feature] 아이템E              → /workflow
 └─ [research] 아이템F             → /deep-analysis

총 예상: {N}주
```

## 아이템 라우팅 규칙

마일스톤 착수 시, 아이템의 유형에 따라 자동으로 적절한 스킬을 안내한다:

| 유형 | 실행 방법 | 산출물 |
|------|----------|--------|
| `feature` | `/workflow "M{N}-{아이템명}"` | 메인태스크 → 서브태스크 → 구현 |
| `research` | `/deep-analysis {대상}` | `progress/analysis/` 분석 보고서 |
| `decision` | `/debate {주제}` | 의사결정 기록 |
| `chore` | 직접 실행 (커밋 1개 수준) | 코드 변경 |
| `spike` | 탐색적 구현 → 결과에 따라 feature 전환 또는 폐기 | 프로토타입 + 판단 기록 |

### feature 아이템 → 메인태스크 전환 규칙
- feature 유형 아이템만 `/workflow`의 메인태스크가 된다
- 메인태스크 문서에 마일스톤 참조를 추가한다:
  ```markdown
  > 마일스톤 참조: progress/milestones/{프로젝트명}/M{N}_{마일스톤명}.md
  ```
- 워크플로우 완료 시 마일스톤 아이템 상태를 `완료`로 갱신
