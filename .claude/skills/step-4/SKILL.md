# Step 4: 서브태스크 인터뷰

## 수행 작업

```
[1] 의존성이 해결된 서브태스크 선택
       ↓
[2] A/B/C(추천) 형식으로 1개씩 상세 인터뷰
       ↓
[3] 큰 덩어리 → 작은 세부사항 순으로 정의
       ↓
[4] 우선순위 및 구현 순서 결정
       ↓
[5] 변경 파일 표 초안 확정
       ↓
[6] ★ Feature List 초안 생성 (크기 판정 → 초안 → 훑어보기 승인)
       ↓
[7] ★ active_subtask.txt 기록 (skip 모드 제외)
       ↓
[8] 서브태스크 문서 생성/업데이트 (`## Feature List` 섹션 포함)
```

## 인터뷰 규칙
- **1개씩** 질문한다
- **A/B/C(추천)** 형식으로 선택지 제공
- 추천 옵션에는 **(추천)** 표시

## 기본 질문

- 이 서브태스크의 구체적인 목표는?
- 어떤 파일/모듈을 수정해야 하나요?
- 테스트 방법은 무엇인가요?
- 예외 케이스가 있나요?

## 중요

> 구현 단계에서는 문서에 명시되지 않은 기능은 절대 개발 금지

# Feature List 초안 생성 (단계 [6])

인터뷰 + 변경 파일 표 확정 후 서브태스크별 검증 체크리스트 **초안**을 자동 생성한다.
세부 확정은 `/step-8` 실측 단계에서 이루어지므로, 이 단계는 **훑어보기 승인만** 받는다.

## 흐름 (순서 고정)

```
[A] 변경 파일 표가 확정된 상태 전제
       ↓
[B] 파일 수 + 위험도로 크기 A/B/C 제안 → 사용자 확인
       ↓
[C] 카테고리 7종 중 조건부 3종(UI·UX/회귀/플랫폼) 포함 여부 판정
       ↓
[D] 카테고리 분배 비율대로 항목 초안 생성 (15~25개 기준)
       ↓
[E] 사용자에게 전체 리스트 제시 → "대략 이런 느낌" 훑어보기 승인
       ↓
[F] 서브태스크 문서에 `## Feature List` 섹션 삽입
```

## [B] 크기 판정 규칙

### 1차 — 변경 파일 수

| 파일 수 | 크기 | 항목 수 | 스킵 가능 |
|---------|------|---------|-----------|
| 1~2개 | 소형 | 10~15 (skip 승인 시 **3~5로 축약**) | ✅ (사용자 확인 시) |
| 3~5개 | 중형 | 15~25 | ❌ |
| 6개 이상 | 대형 | 25~35 | ❌ |

### 2차 — 위험도 예외 (1차 결과 한 단계 상향)

다음 조건 중 하나라도 해당하면 **1차 결과를 한 단계 올림**:

- 단일 파일 500라인+ 수정
- 상태 관리 / 인프라 / DB 마이그레이션 / 상태머신 영역
- 공통 컴포넌트 / 공용 라이브러리 변경
- 외부 API 계약 변경 (스키마 추가·삭제)

예: 파일 2개지만 그 중 하나가 상태머신이면 → 소형(1차) → **중형(2차 강제)**

### [B] 사용자 확인 포맷

```
변경 파일 {N}개, {위험도 예외 설명(있을 시)} → {크기} 제안.

A) 소형 (10~15개)
B) 중형 (15~25개)  (추천)
C) 대형 (25~35개)

어떤 크기로 진행할까요?
```

## [C] 카테고리 조건부 판정

### 필수 4종 (항상 포함)
- **기능** F — 구현되어야 할 기능 목표
- **엣지** E — 경계 조건, 입력 극단값
- **에러** R — 실패 경로, 예외 처리
- **테스트** T — 단위 테스트 / 통합 테스트 / 회귀 테스트 커버리지

### 조건부 3종 (서브태스크 성격에 따라)
- **UI·UX** U — 프론트엔드 변경 포함 시
- **회귀** G — 공통 컴포넌트/공용 라이브러리 변경 시
- **플랫폼** P — iOS/Swift/Xcode 빌드/Tuist 등 플랫폼 의존 작업 시

### 판정 예시

| 서브태스크 성격 | 필수 | 조건부 |
|-----------------|------|--------|
| Svelte 컴포넌트 추가 | F, E, R, T | U |
| Rust API 핸들러 추가 | F, E, R, T | - |
| iOS 화면 추가 | F, E, R, T | U, P |
| 공통 라이브러리 리팩토링 | F, E, R, T | G |
| 문서 업데이트 | F, E, R, T | - |

> **예외 없음**: 필수 4종(F/E/R/T)은 서브태스크 성격과 무관하게 **항상 포함**. 문서 업데이트도 엣지(오타/깨진 링크), 에러(렌더링 실패), 테스트(맞춤법 검증) 항목이 유효함. 비율이 낮아져도 섹션 자체는 유지한다.

## [D] 항목 초안 생성 규칙

### 안정 ID 규약

```
카테고리 prefix 1자리 + 하이픈 + 2자리 순번

F-NN : 기능
E-NN : 엣지
R-NN : 에러
T-NN : 테스트
U-NN : UI·UX
G-NN : 회귀
P-NN : 플랫폼
```

- 순번은 **카테고리 내부에서 01부터** (예: `F-01`, `F-02`, ..., `E-01`, `E-02`)
- ID는 한 번 부여되면 재활용 안 함 (항목 삭제돼도 뒤 항목 재번호 안 함)
- Grep 앵커: `- \[([ x~\-])\] ([FERTUGP])-\d{2}\b`

### 4상태 모델

| 상태 | 기호 | 의미 | 사유 필드 | 갱신 주체 |
|------|------|------|-----------|-----------|
| 미완료 | `- [ ] F-01 ...` | 기본값 | - | - |
| 완료 | `- [x] F-01 ...` | 실측 통과 | - | step-8 사용자 보고 기반 Claude 갱신 |
| 연기 | `- [~] deferred (사유) F-02 ...` | 필요하지만 이번엔 보류 | **필수** | 사용자 명시 사유 지시 시 갱신 |
| 해당없음 | `- [-] N/A (사유) F-03 ...` | 서브태스크 범위 밖 | **필수** | 사용자 명시 사유 지시 시 갱신 |

**초안 생성 시 모든 항목은 `- [ ]` 미완료 상태**. `[x]`/`[~]`/`[-]`는 step-8 실측 단계에서만 전환.

### 카테고리 분배 비율 (중형 15~25 기준 권장)

| 카테고리 | 비율 | 비고 |
|----------|------|------|
| F (기능) | 30~40% | 서브태스크 목표 반영 |
| E (엣지) | 15~20% | 경계/극단값 |
| R (에러) | 10~15% | 실패 경로 |
| T (테스트) | 20~25% | 커버리지 90% 룰 반영 |
| U/G/P | 각 5~15% | 해당 시 |

AI 초안은 이 비율에 따라 분배. "기능 20개 + 엣지 0개" 편중 방지.

## [E] 사용자 훑어보기 승인

전체 리스트를 한 번에 제시하고 **세부 수정은 step-8에서 가능**함을 명시.

```
Feature List 초안 18개입니다. (중형, skip: false)

[전체 리스트 제시]

**세부 수정/추가/삭제는 /step-8 실측 단계에서 가능합니다.**
지금은 "대략 이런 느낌이 맞다" 수준 승인만 받으면 됩니다.

진행할까요? (Y/n)
```

## [F] 섹션 포맷 (서브태스크 문서 삽입)

```markdown
## Feature List
<!-- size: 중형 | count: 18 | skip: false -->

### 기능
- [ ] F-01 로그인 후 피드 진입 시 스켈레톤 UI 표시
- [ ] F-02 스크롤 무한로딩 정상 동작
...

### 엣지
- [ ] E-01 빈 피드 상태 메시지 노출
- [ ] E-02 네트워크 재연결 시 자동 재시도
...

### 에러
- [ ] R-01 API 5xx 응답 시 재시도 버튼 노출
...

### 테스트
- [ ] T-01 Vitest 단위 테스트 신규 컴포넌트 커버
- [ ] T-02 E2E 골든 패스 시나리오 통과
...

<!-- 조건부 카테고리 (해당 시만) -->
### UI·UX
- [ ] U-01 다크모드 색상 대비 확인
...
```

### HTML 메타 (한 줄 SSOT)

```
<!-- size: {소형|중형|대형} | count: {총 개수} | skip: {true|false} -->
```

- `size`: 소형 / 중형 / 대형
- `count`: 체크박스 총 개수
- `skip`: true일 때 step-8 순회 생략 가능 (소형 + 사용자 확인)

**이 메타 한 줄만 파서 SSOT**. 다른 형식의 메타 주석 사용 금지.

## 파싱 실패 정책 (step-8/향후 §8.2 공유)

step-8이 Feature List 파싱 시 다음 경우는 **파싱 실패 = 차단 + 복구 제안**:

| 실패 유형 | 감지 기준 | 동작 |
|-----------|-----------|------|
| ID 포맷 위반 | prefix 외 문자 / 순번 2자리 아님 / 중복 ID | 차단 + "canonical formatter로 재생성" 제안 |
| 상태 기호 위반 | `[ ]` / `[x]` / `[~]` / `[-]` 외 기호 | 차단 + 허용 기호 안내 |
| `[~]`/`[-]` 사유 누락 | 괄호 없음 or 빈 괄호 | 차단 + 사유 보정 요청 |
| HTML 메타 누락/부족 | `<!-- size ... \| count ... \| skip ... -->` 없음 or 필드 부족 (shorthand 금지) | 차단 + 메타 복구 제안 |
| count 불일치 | 메타 `count` 값과 실제 체크박스 개수 불일치 | 차단 + 자동 재계산 제안 |
| 카테고리 순서 위반 | H3 순서 고정 규칙 위반 | 경고만 (파싱 진행) |

**Silent pass 금지** — 위반 감지 시 반드시 사용자에게 알리고 복구 경로 제시.

## 소형 스킵 모드

소형(1~2 파일) 판정 후 사용자가 "스킵" 확인 시:

```markdown
<!-- size: 소형 | count: 4 | skip: true -->

### 기능
- [ ] F-01 ...
- [ ] F-02 ...

### 테스트
- [ ] T-01 ...
- [ ] T-02 ...
```

- 최소 3~5개만 생성
- step-8에서 순회 루프 생략, 전체 일괄 확인만 받음
- `skip: true` 시 feature-list-check.sh도 통과 처리

## [G] active_subtask.txt 기록 (단계 [7])

Feature List 초안 생성 후 **즉시** `progress/active_subtask.txt`에 서브태스크 문서 경로를 기록한다.

### 기록 조건

| 조건 | 동작 |
|------|------|
| `skip: false` (중형/대형) | **기록 필수** — step-9/pre-commit hook의 Feature List 검증 대상 |
| `skip: true` (소형 스킵) | **기록 생략** — feature-list-check.sh가 skip=true 시 통과 처리하므로 기록 불필요 |

### 기록 포맷

```
progress/subtask/{상위폴더}/{파일명}.md
```

- **progress 상대경로** (절대경로·repo 루트 상대경로 금지)
- 예: `progress/subtask/260420_g8_2_commit_block/01-feature-list-check-sh.md`

### Write 도구 사용

```
Write("progress/active_subtask.txt", "progress/subtask/{경로}/{파일명}.md")
```

### 클리어 시점

- step-9 커밋 성공 직후 `none`으로 초기화 (step-9 스킬이 담당)

### 메타태스크 면제

하네스 룰 자체를 변경하는 태스크(스킬 SKILL.md, hook 스크립트, rules/ 파일 수정 등)는
`active_subtask.txt` 기록을 생략해도 무방하다.
이유: 메타태스크는 Feature List 체계 도입 이전부터 존재하거나, Feature List 자체를 관리하는 코드를 변경하는 경우라 자기참조 오류가 발생할 수 있다.
