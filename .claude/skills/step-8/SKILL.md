# Step 8: 테스트

## 프로젝트 타입 자동 감지

```bash
git diff --cached --name-only 2>/dev/null || git diff --name-only HEAD
```

변경된 파일 경로를 기반으로 어떤 테스트를 실행할지 자동 결정한다.

## 테스트 명령어

프로젝트에 맞는 테스트 명령어를 실행한다:

### Swift (iOS/macOS)
```bash
swift test                          # 단위 테스트
swiftlint lint                      # 린트
```

### Svelte (프론트엔드)
```bash
cd {프론트엔드_경로}
npm run lint                        # ESLint
npm run check                       # svelte-check
npm run test                        # Vitest
```

### Rust (서버)
```bash
cargo clippy -- -D warnings         # 린트
cargo test                          # 테스트
```

### Python ML (서버)
```bash
ruff check .                        # 린트
mypy .                              # 타입체크
pytest tests/ -v                    # 테스트
```

## 통과 기준

| 테스트 | 기준 |
|--------|------|
| 린트 | 0 errors |
| 타입체크 | 0 errors |
| 단위 테스트 | 100% pass |

## 자동화 테스트 통과 후 — 배포 + Feature List 순회 검증 (필수)

자동화 테스트가 모두 통과하면 **반드시** 아래 순서로 진행한다. 건너뛰지 않는다.

### 1. 배포 스크립트 실행

```bash
scripts/deploy.sh
```

- Docker 미실행 시 `y` 입력하여 네이티브 모드로 실행
- 브라우저(`http://localhost:5173`)와 iOS 시뮬레이터가 자동으로 열린다

### 2. Feature List 순회 검증 (step-4에서 생성된 초안 기반)

`step-4`에서 생성된 서브태스크 문서의 `## Feature List` 섹션을 로드하여 **한 항목씩 순회**하며 사용자 실측 결과를 반영한다.

#### 2-A. 파싱 + 사전 검증

1. 서브태스크 문서의 `## Feature List` 섹션 로드
2. HTML 메타 한 줄 파싱 (step-4 SSOT): `<!-- size: {소형|중형|대형} | count: {N} | skip: {true|false} -->` — **shorthand/부분 메타 금지**
3. **count 검증**: 메타 `count` 값과 실제 체크박스 개수가 일치하지 않으면 파싱 실패 (step-4 "HTML 메타 필드 부족"에 준함)
4. 파서 규약 검증은 step-4 SKILL.md **"파싱 실패 정책" 그대로 준수** (ID 포맷 / 상태 기호 / 사유 필드 / HTML 메타 / 카테고리 순서)
5. 실패 감지 시 **차단 + 복구 제안** (Silent pass 금지)

#### 2-B. 소형 스킵 모드 분기 (`skip: true`)

step-4 SSOT 한 줄 메타에서 `skip: true`로 파싱된 경우:
- 순회 루프 **생략**
- 전체 체크박스 요약 표로 한 번에 제시
- 사용자 보고 3가지 경로:
  - **전체 통과**: "통과" 한 마디 → 모든 항목 일괄 `[x]` 갱신
  - **부분 예외**: "F-02는 N/A (사유), T-01은 연기 (사유), 나머지 통과" → 명시된 항목만 `[-]`/`[~]`로 개별 갱신 + 나머지는 일괄 `[x]`
  - **전체 보류**: "아직" / "검증 못 함" → 변경 없이 step-9 진입 (`[~]` 일괄 전환 안 함 — 명시 사유 없으면 AI 독단 금지)
- step-9 진입 가능 (향후 §8.2 차단 로직에서 skip=true는 통과 처리)

#### 2-C. 순회 루프 (`skip: false`)

순회 시작 시 **ID prefix legend 한 줄 제시** (사용자 prefix 혼동 방지):

```
Feature List 순회 시작. ID 규약: F=기능 / E=엣지 / R=에러 / T=테스트 / U=UI·UX / G=회귀 / P=플랫폼
```

```
for 각 카테고리 in [기능, 엣지, 에러, 테스트, (UI·UX, 회귀, 플랫폼)]:
    for 각 항목 in 카테고리:
        if 이미 [x]/[~]/[-] 상태면 skip (재확인 안 함)
        ※ 단, 사용자가 "F-03 재측정" / "reopen F-03" 등 명시 요청 시
          해당 항목을 [ ]로 되돌리고 순회 큐에 다시 포함

        Claude 제시:
            "{ID} {항목 내용}
             → 실제 동작 확인 후 '통과/실패/N/A (사유)/연기 (사유)' 보고"

        사용자 결과 수신 후 상태 갱신:
            "통과"             → - [x] {ID} {내용}
            "실패"             → - [ ] {ID} {내용} (변경 없음, 미완료 유지)
            "N/A (사유)"       → - [-] N/A (사유) {ID} {내용}
            "연기 (사유)"      → - [~] deferred (사유) {ID} {내용}
            "미완료로 복귀"    → - [ ] {ID} {내용} (사유 토큰 제거)

        Write로 문서 즉시 반영
```

#### 2-D. 사용자 일괄 보고 허용

Frank 규모(15~25개 기준)에서 사용자가 여러 항목을 한 번에 보고 가능:

```
사용자 예시:
"F-01, F-02, E-01 통과.
 E-02 N/A (백엔드라 UI 검증 무관).
 T-01 연기 (다음 마일스톤에서 E2E 재작성)."
```

Claude는 위 보고를 한 번에 파싱하여 일괄 Write. 항목별 재확인 프롬프트 생략.

#### 2-E. 실측 중 항목 추가 / 수정 / 삭제 / 상태 복귀

실측 중 초안에 없던 중요 검증 포인트 발견 또는 부정확 판단 시:

| 동작 | 트리거 (사용자 지시) | 처리 |
|------|------------------------|------|
| 추가 | "이거 빠졌어. 추가: ..." | 해당 카테고리 **마지막 순번 + 1** ID 자동 부여 (예: `F-18`), 초기 상태 `[ ]`. 사용자가 ID 직접 지정해도 **무시**하고 시스템 순번 사용 (중복/혼동 방지, 경고 1회 출력) |
| 내용 수정 | "F-03 문구 바꿔: ..." | ID 유지, 내용만 교체 |
| 삭제 | "F-05 삭제" | 해당 라인 제거. 뒤 항목 ID 재번호 **안 함** (파싱 안정성 유지) |
| 상태 복귀 | "F-03 미완료로 복귀" / "reopen F-03" | 사유 토큰 제거하고 `- [ ] F-03 ...`로 환원. 순회 큐에 재삽입 |
| `[x]` 재측정 | "F-03 재측정" | 예외적으로 상태 복귀 후 순회 큐 재삽입 가능 (기본은 `[x]` 재확인 안 함) |

**ID는 한 번 부여되면 삭제돼도 재활용 안 함**. 순번에 간격이 생겨도 무방. HTML 메타 `count`는 갱신 후에도 실제 체크박스 개수와 일치해야 함 (자동 재계산).

#### 2-F. `[~]` / `[-]` 상태 전환 금지 규칙

- Claude는 **사용자가 명시 사유를 제공한 경우에만** `[~]`/`[-]` 갱신
- AI 독단으로 "이건 N/A 같은데?" 판단 금지
- `[x]`도 동일 — 사용자 "통과" 보고 없이 Claude가 찍지 않음

### 3. 잔여 미체크 요약 출력

순회 완료 시 카테고리별 `[ ]` 잔여 개수 요약:

```
잔여 미체크 요약:
- 기능: 2/8 미체크
- 엣지: 1/3 미체크
- 에러: 0/2 ✅
- 테스트: 0/4 ✅
- UI·UX: 1/3 미체크

총 4개 [ ] 남음. step-9 진입 전 검토 필요.
```
