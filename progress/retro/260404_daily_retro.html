🔑 핵심 의사결정과 그 이유

결정 1: "자동 큐레이션"으로 좁히기

상황: 넓은 "AI 스터디앱"에서 MVP 핵심 가치 선택

선택지:
스크랩 중심 학습 경로 인사이트 대화

결정: 키워드 선택 → 자동 웹서치 → LLM 요약

왜: AI 정보 과잉이 출발점. "좋은 콘텐츠를 자동으로 모아주는 것"이 먼저.

📐 기획/설계 과정

오늘은 코드를 한 줄도 안 짜고, 하루 종일 "무엇을 만들 것인가"만 논의했다.

인터뷰 기반 기획

멘토가 하나씩 질문을 던져줬다. "AI 기반 스터디앱"이라는 막연한 아이디어가 질문에 답하며 구체화됐다. 특히 "수동 스크랩이 아니라 자동 수집"으로 방향이 바뀐 순간이 인상 깊었다.

경쟁 서비스 리서치

"나만 이런 생각을 한 게 아닐 텐데"에서 시작. Artifact(실패), Feedly(유료), Readwise(수동) 등을 분석하고 포지셔닝이 명확해졌다:

💡 "키워드 기반 자동 수집 + LLM 인사이트를 하나로 묶은 서비스는 아직 없다"

아키텍처 논쟁

Claude는 혼용 전략을 제안, 나는 에코 서버를 제안. "심판해달라"고 요청해서 8개 기준 채점 비교 → 내 제안이 이겼다. AI와 "토론"하는 것과 "수용"하는 것은 완전히 다른 경험.

💡 인사이트 & 피드백

1. "다 모르면 오히려 유리하다"

기존 지식에 얽매이지 않아서 더 깔끔한 결정. "Swift에서는 이렇게 하는데..."라는 편견이 없었다.

2. "바이브 코딩은 의사결정의 연속"

코드 0줄이지만 오늘 결정들이 수천 줄의 방향을 결정. "올바른 방향을 빠르게 찾는 것"이 핵심.

3. "AI에게 심판을 맡기면 객관적 비교"

"8개 기준으로 채점해달라"고 요청하니 정량적 비교와 명확한 근거가 나왔다.

4. "문서 분리의 중요성"

CLAUDE.md(작업 가이드) vs progress/(상세 기획). 목적에 맞게 분리하니 둘 다 읽기 좋아졌다.

5. "포트/어댑터는 모든 플랫폼에 적용"

Rust(trait), Swift(Protocol), Svelte(Interface) 전부 같은 패턴으로 통일. 진짜 아키텍처의 가치.

📚 배운 것

기술적으로

  • Axum — 2026년 Rust 웹 사실상 표준
  • Supabase REST — SQL 없이 DB 접근, RLS 자동
  • 검색 API — Tavily/Exa/Firecrawl 각각의 강점
  • OpenRouter — 다양한 LLM 하나의 API로

프로세스에서

  • 인터뷰 기반 기획 → 막연함을 구체화에 효과적
  • 경쟁 리서치는 기획 중간에 해야 함
  • AI와 토론 ≠ AI 수용

💭 느낀 점

"iOS 개발자가 Rust 서버를?" 처음엔 그런 생각이었다. 근데 멘토와 하루 종일 기획하고 나니, 코드 전에 방향 잡는 게 얼마나 중요한지 느꼈다.

에코 서버 패턴을 직접 제안하고 채택된 건 뿌듯했다. "모르니까 질문 → 답에서 또 질문" 사이클이 재밌었다.

내일부터 진짜 코드를 짜야 하니까 긴장되지만, 기획이 탄탄하니 방향을 잃을 일은 없을 것 같다.
