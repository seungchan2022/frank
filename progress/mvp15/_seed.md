## 주제

**기사 양·다양성 확보 + 내 커스텀 인사이트**

두 축:
1. **(E) 기사 양·다양성**: 검색엔진 활용 전략 변경 + 쿼리 다양화로 피드 양·다양성 확대
2. **(A-1) 내 커스텀 인사이트**: 사용자 프로필 기반 맞춤 시각으로 요약/인사이트 생성

## 채택 배경

### 발견 (코드 점검으로 확인)

1. **피드는 ephemeral**: `server/src/api/feed.rs:69` — DB 저장 안 됨, 메모리 캐시 5분 TTL만. `articles` 테이블은 즐겨찾기 한 기사만 저장
2. **태그당 5개만 검색**: `server/src/api/feed.rs:216` — `chain.search(&search_query, 5)`
3. **검색 쿼리 단일**: `"{tag} latest news"` 형식 고정 → 같은 기사 반복
4. **Tavily 실패 → Exa 폴백**: `search_chain.rs`는 첫 성공만 사용. 사용자 관찰상 현재 Exa로 빠짐 — 실패 원인 미점검

### 정체성 정합

선언: "내 관점으로 트렌드를 보는 개인 레이더"
현재 토대: 태그당 5개 · 5분 후 사라짐 · 누적 0
→ 정체성과 토대 사이의 갭이 MVP15 채택 이유

---

## 사용자 디자인 의도 (보존 필수)

### 기사 ephemeral 유지

**DB 영속 저장 안 함**. 이유:
- "훑고 넘기는" 가벼움 보존 (트렌드 레이더의 정체성)
- 영속 저장 시 "이미 본 기사 어떻게 표시/삭제할지" 결정이 줄줄이 따라옴 → 정체성 흐려짐
- 양·다양성만 늘리고 누적은 안 하는 방식

→ MVP15에서 양은 늘리되, **기사 자체는 영속 저장 안 함**.

### 사용자 본인 사용 전제

- 클라우드 배포 우선순위 낮음 — 맥 켜놓고 본인이 사용
- B(클라우드 배포)는 MVP15 제외

## Discovery에서 결정할 것

1. **프로필 깊이**:
   - a. 직업 한 줄 ("iOS 개발자")
   - b. 직업 + 관심 분야 여러 개
   - c. 자유 텍스트로 직접 작성
2. **인사이트 표시 위치**:
   - 기존 요약 카드 안에 한 단락 추가
   - 별도 카드 ("내 시각" 카드)
   - 토글로 켜고 끄기
3. **양 확대의 비용·속도 트레이드오프**: 메모리 `feedback_api_cost_policy` (비용 증가 금지 원칙)
4. **검색엔진 활용 방식**:
   - 폴백 → 병합 (각 엔진 결과 합치기)?
   - 병행 (다른 쿼리 분배)?
   - 엔진별 역할 분리 (Tavily=뉴스, Exa=기술/연구)?
5. **검색 쿼리 다양화 전략**:
   - 시간대 변형 ("this week", "today")
   - 관점 변형 ("announcement", "release")
   - 몇 개 쿼리 동시?
