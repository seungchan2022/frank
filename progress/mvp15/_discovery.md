## 발견 (시드 단계 코드 점검)

1. **피드는 ephemeral**: `server/src/api/feed.rs:69` — DB 저장 안 됨, 메모리 캐시 5분 TTL만
2. **태그당 5개 limit**: `server/src/api/feed.rs:216` — `chain.search(&search_query, 5)`
3. **검색 쿼리 단일**: `"{tag} latest news"` 형식 고정 → 같은 기사 반복 가능성
4. **Tavily 실패 → Exa 폴백**: `search_chain.rs`는 첫 성공만 사용. 사용자 관찰상 현재 Exa로 빠짐 — 실패 원인 미점검
5. **사용자 통점 (인터뷰 추가 발견)**: 일부 태그는 결과가 **1개만** 나옴. limit 5에서 잘리는 게 아니라 검색 자체가 부족할 가능성

## 인터뷰 결정 사항 (Q1~Q5)

### Q1. 인사이트 디자인

- **결정**: **객관 요약 + 내 시각 단락** (같은 카드 안). 액션 곁들임은 MVP15 제외
- **시각 결**: 일반 요약은 "**무엇이** 일어났는지", 내 시각은 "**그래서 iOS 개발자 입장에서 뭐가 달라지는지**" — 역할 분담, 정보 중복 없음
- **백엔드 응답**: 분리된 필드로 (UI 통합/분리 자유)
- **프롬프트**: 별도 파일/상수로 분리 (변경 비용 최소)

### Q2. 프로필

- **결정**: **직업 한 줄만** (예: "iOS 개발자"). 자유 텍스트·태그·계층 다 X
- **입력 위치**: 설정 페이지에 항목 1개 추가
- **강제·안내**: **없음** (선택적). 미설정 시 인사이트 단락 생략, 일반 요약만 표시
- **LLM 컨텍스트**: 직업 한 줄만 주입. 피드 태그는 검색용으로만 (LLM 컨텍스트 X)
- **이유**: "잘 모르겠다" 상태에서 결정 미루기. 써보면서 진짜 욕구 발견 시 자유 텍스트·태그 추가 (변경 비용 한나절)

### Q3. 양 확대 + 비용

- **결정**: **양 limit 5 → 10** (코드 변경, 호출 수 변화 없음 — Tavily는 호출 단위 카운트)
- **나머지**: 검색 엔진·캐시·태그 다 지금 유지
- **비용 원칙**: **$0 유지** (무료 리소스만)
- **이유**: limit 변경 자체는 호출 수 안 늘림. 무료 안 100% 안전. M1 진단 후 양 더 늘릴지 결정

### Q4. 검색엔진 활용 (보류 → M1 진단 후 결정)

- 시드 M1 ("검색엔진 진단")의 정확한 미션
- 본인 통점 "1개만 나오는 태그"는 limit 병목 아닌 검색 결과 자체 부족 가능성
- M1에서 실측: limit 100으로 호출 시 진짜 결과 수, 다른 쿼리·다른 엔진의 결과량 비교

### Q5. 검색 쿼리 다양화 (보류 → M1 후 결정)

- Q3 결정("지금 유지")과 정합 → 단일 쿼리 유지
- M1 진단에서 "쿼리 다양화가 1개 태그 문제 해결에 효과적인지" 측정
- 효과 있고 무료 한도 여유 있으면 M2 또는 후속 MVP에서 추가

## 인터뷰 인사이트

- **사용자가 "잘 모르겠다"고 한 영역에서 결정 강요 X** — 단순 형태로 시작 + 써보고 진화 (Q2)
- **변경 비용을 미리 보여주면 결정 부담 ↓** — Q1 답변 패턴: 형태 보여주기 → 변경 비용 짚기 → 결정 (반복)
- **사용자 통점이 진짜 디자인 결정자** — "1개만 나오는 태그"가 양 확대 디자인의 진짜 니즈
