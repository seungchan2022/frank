## 2. 패턴 분석

### 2.1 결과량 분포 (Tavily 기준 — Exa는 §2.2 참조)

| 결과 수 | 셀 개수 | 대표 태그·변형 |
|---|---:|---|
| 19~20 (max) | 47/60 | 대부분 — Tavily가 충분히 풍부 |
| 14~16 | 13/60 | 모바일 개발(year2026, announcement), 데이터 사이언스(announcement, english_baseline=Data Science), 보안(english_baseline=Security), 블록체인(baseline, simple), 오픈소스(baseline), 클라우드/인프라(baseline), 웹 개발(announcement), 투자/VC(english_baseline=Investment/VC) |
| ≤10 | 0/60 | **없음** — 진단 시점엔 어떤 태그도 결과 부족 아님 |

**핵심**: Tavily는 어떤 태그·변형 조합에서도 14~20개 결과 제공 (효율 70~100%).

### 2.2 엔진별 격차 — Exa 전부 실패

- **Tavily**: 60/60 응답 (실패율 0%)
- **Exa**: **0/60 응답** (실패율 100%)
  - 모든 60셀에 동일 에러: `HTTP 402 Payment Required — You have exhausted your free credits`
  - 즉 **Exa 무료 한도 완전 소진**. 진단 시점 기준 결제 전까지 모든 호출 실패.
- **운영 영향 평가**:
  - `SearchFallbackChain`은 Tavily 우선, Tavily 실패 시 Exa 폴백
  - Tavily가 진단에서 실패율 0% → 운영에서도 대부분 Tavily가 답을 주고 있어 사용자 경험 영향 미미
  - 단 Tavily가 가끔 실패하는 케이스에선 **폴백 보험 없는 상태**. 결과 0개 표시될 가능성.

### 2.3 변형별 효과 — 매우 작음

- 한국어 4변형(baseline / simple / 2026 / announcement)과 영어 번역(english_baseline) 사이 결과 수 차이 미미 (대부분 ±2 범위)
- "모바일 개발" 같이 한국어가 약간 적은 (14) 케이스도 영어 번역 시(Mobile Development) 20으로 회복 — 단 이런 패턴 일관되지 않음
- 시간 변형(2026), 관점 변형(announcement)도 baseline 대비 큰 변화 없음

**해석**: H1 가설("Tavily 한국어 약점") 부분적 시사하나 강하지 않음. **시드 가설보다 Exa 한도 소진이 더 결정적 발견**.

### 2.4 사용자 통점 ("1개만 나오는 태그") 진단 재현 안 됨

**원인 후보 (확률순)**:
1. **운영 limit=5 + Exa 폴백 실패의 조합** — Tavily가 5개 미만 주는 시점에 Exa 폴백도 죽어 있어 결과 적게 보였을 가능성. **유력**
2. **시간 갭** — 통점 시점에 Tavily가 일시적으로 적게 줬을 가능성 (4-7일 전). 검증 불가.
3. **캐시 영향** — 사용자가 같은 시점에 새로고침 시 5분 캐시 HIT으로 동일 결과 반복.

**결론**: 진단 시점 데이터로는 통점 직접 재현 불가. 다만 **limit=5 + Exa 죽음** 조합이 통점의 운영 환경 원인일 가능성 강함.

### 2.5 Tavily 실패 분류

- **실패 0건** — 영구/간헐 분류 대상 없음
- 진단 시점 한정. 다른 시간대 실패 패턴 알 수 없음.

## 3. 권고 (3블록 강제)

### 3.1 M2 권고 — 양 limit 결정

#### 근거
- Tavily는 무료 티어 max=20 (코드의 `effective` 컬럼). 우리가 `limit=100` 요청해도 20으로 clamp됨
- 운영 코드(`feed.rs:216`) `limit=5` → 20개 중 5개만 받음
- **15개 결과를 매번 버리고 있음**

#### 결정
- **운영 `limit` = 20으로 상향** (Tavily max 정확히 활용)
- `feed.rs:216`의 `chain.search(&search_query, 5)` → `chain.search(&search_query, 20)`

#### 위험
- 응답 페이로드 4배 증가 → 네트워크·메모리 영향 (단 20개도 작은 양, 위험 미미)
- 무료 한도 소비량 1회당 동일 (한도는 호출 수 기준이지 결과 수 아님)
- **Exa 결제 결정과 결합 시**: Exa max 확인 후 양 엔진 일관 limit 적용 (현재는 Tavily 단독으로 limit=20 의미)

### 3.2 Q4 권고 — 검색 엔진 활용 방식

#### 근거
- **Exa 무료 한도 완전 소진 (60/60 실패, HTTP 402)**
- Tavily 단독으로도 진단 시점 60/60 응답 → 단독 운영 가능성 검증
- 단 Tavily 실패 시 폴백 보험이 없음 = 단일 의존 위험

#### 결정 (사용자가 다음 마일스톤 진입 전 결정)
- **옵션 A**: Exa 결제 (월 $X 발생, 메모리 `project_api_cost_policy` "$0 유지" 원칙 위반)
- **옵션 B**: Exa 빼고 Tavily 단독 + 다른 무료 엔진 추가 (예: Brave Search, SerpAPI 무료 티어)
- **옵션 C**: Tavily 단독 + 한도 80% 도달 시 캐시 강화 (메모리 정책 그대로)
- **추천**: B 또는 C. A는 비용 정책 위반.

#### 위험
- B: 새 엔진 어댑터 작성 비용 + 무료 티어 한도 검증
- C: Tavily 일시 장애 시 사용자 검색 결과 0건 (사용자 경험 직접 영향)
- A: 비용 발생 + 정책 위반 (별도 동의 필요)

### 3.3 Q5 권고 — 쿼리 다양화

#### 근거
- 5변형(baseline / simple / 2026 / announcement / english_baseline) 간 Tavily 결과 수 차이 미미
- 영어 번역 변형이 일부 한국어 약한 태그에서 회복 효과 보였으나 일관되지 않음 (예: 모바일 개발 14→20, 데이터 사이언스 16→15)
- 시간 변형(2026), 관점 변형(announcement) 효과 거의 없음

#### 결정
- **현 단일 쿼리(`{tag} latest news`) 유지**
- 변형 다양화 도입 안 함 — 호출량 증가 대비 결과 풍부도 향상 없음

#### 위험
- 한국어 약한 태그에서 결과 부족 시 영어 번역 fallback이 도움 될 수도 있으나, 진단 데이터로 명확한 패턴 안 보임 → 도입 근거 약함
- 향후 다른 사용자 추가 시 한국어 외 언어 태그 들어오면 재진단 필요
