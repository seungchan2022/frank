# M2: 양 확대 (서버)

> 프로젝트: Frank MVP15
> 상태: 대기
> 예상 기간: 2~3일
> 의존성: M1 (진단 결과)

## 목표

서버에서 피드 양 limit을 5→10 (또는 M1 진단 결과값)으로 확대하고, 무료 한도 안에서 안전하게 운영되도록 모니터링·안전장치를 추가한다.

## 배경

- Q3 결정: 양 limit 5→10. 호출 수는 변화 없음 (Tavily는 호출 단위 카운트)
- 비용 원칙: $0 유지 (무료 리소스만)
- 한도 자동 감지·완화 안전장치 필요 (사용량 폭증 시 자동 캐시 강화)

## M1 진단 결과별 분기

M1 진단 보고서의 "원인" 결론에 따라 M2 작업 범위가 달라진다:

| M1 결론 | M2 작업 |
|---|---|
| **(A) limit 5가 병목**, 검색 엔진은 충분히 결과 줌 | limit 5→10 단순 변경. 카운터·안전장치는 그대로 진행 |
| **(B) 검색 결과 자체 부족**, limit 늘려도 무의미 | limit 변경 보류 또는 5→10만. 핵심 작업을 검색 전략 변경(병합/병행)으로 재스코프. M3 전 단계로 들어옴 |
| **(C) Tavily 실패 원인 외부 (서비스 장애 등)** | 폴백 안정화 우선 (재시도 로직). limit·분배는 후속 |
| **(D) 혼합 (일부 태그는 A, 일부는 B)** | 우선 limit 5→10 적용 + 가장 결과 부족한 태그를 위한 검색 전략 추가 검토 |

→ M2 진입 시점에 M1 보고서를 먼저 읽고 분기 결정.

## 성공 기준 (Definition of Done)

- [ ] M1 진단 보고서 읽고 위 분기표 기반으로 M2 작업 범위 확정
- [ ] `server/src/api/feed.rs:216` limit 5 → 10 (또는 M1 결정값)
- [ ] 검색 호출 카운터 추가 — **반드시 영속 저장** (파일 또는 DB). in-memory 금지 (재시작 시 리셋되어 $0 검증 불가)
  - 권장: `progress/mvp15/cost_log.json` (일자별 + 엔진별 카운터) 또는 신규 테이블 `api_call_counters`
  - **엔진별 분리 카운터** (`tavily_calls_this_month`, `exa_calls_this_month`, `firecrawl_calls_this_month`)
- [ ] 한도 모니터링: **각 엔진별 80% 도달** 시 캐시 TTL 5분 → 30분 자동 전환
- [ ] 한도 **각 엔진별 100% 도달** 시 해당 엔진 검색 호출 차단 + 다른 엔진/캐시만 사용
- [ ] 카운터 조회 엔드포인트 (`GET /admin/cost`) 또는 본인 확인용 로그
- [ ] 서버 테스트 통과 (`cargo test`)
- [ ] 통합 테스트: 한도 시뮬레이션 (mock으로 80%/100% 트리거 후 동작 검증)
- [ ] 재시작 후 카운터 보존 검증 (영속 저장 동작 확인)
- [ ] 변경 후 1회 실측: 본인 사용으로 양 10개 표시 확인
