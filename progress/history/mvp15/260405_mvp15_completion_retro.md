## 심층 분석

### 1. http_utils.rs — 이번 작업의 핵심 산출물

559줄짜리 신규 모듈이 전체 변경량의 20%를 차지한다. RetryConfig + 지수 백오프 + Content-Length/chunked 양쪽 size check를 하나의 공통 유틸로 통합했다.

**Before (MVP1)**: 4개 어댑터가 각자 에러 처리 → 타임아웃 미설정, 무한 대기 가능, 응답 크기 무제한

**After (MVP1.5)**: 모든 외부 HTTP 호출이 동일 retry/size-limit 정책 적용
```
tavily.rs  ─┐
exa.rs     ─┼→ send_with_retry() → read_body_limited()
firecrawl.rs─┤     ↑ RetryConfig::for_search/for_crawl/for_llm
openrouter.rs┘
```

**의의**: 외부 API 장애 시 "무한 대기 → 로그 없이 실패" 시나리오가 "3회 재시도 → 지수 백오프 → 명확한 에러"로 전환. 프로덕션 안정성의 기반.

### 2. 아키텍처 의존 방향 복원

MVP1에서 2건의 의존 방향 위반이 있었다:

| 위반 | 해결 방법 | 결과 |
|------|----------|------|
| `sqlx::FromRow`가 domain에 존재 | infra에 로컬 Row 구조체 + 수동 매핑 | domain에서 sqlx import 0건 |
| `SearchFallbackChain` 구체 타입이 services에서 참조 | `SearchChainPort` trait 추가 | services에서 infra import 테스트만 |

**현재 의존 그래프**:
```
api → services → domain(ports)
                      ↑
                  infra(adapters)
```

역방향 의존 0건. "에코 서버" 원칙 완전 복원.

### 3. 테스트 증가 패턴 분석

| 시점 | 서버 | 웹 | 합계 | 증가율 |
|------|------|-----|------|--------|
| MVP1 완료 | 69 | 66 | 135 | - |
| A 완료 | 124 | 72 | 196 | +45% |
| C 완료 | 137 | 77 | 214 | +9% |
| 최적화 완료 | 139 | 77 | 216 | +1% |

마일스톤 A에서 서버 테스트가 +55건 급증한 이유: `http_utils.rs`의 retry/size-limit 로직이 정밀한 단위 테스트를 요구했고, auth 미들웨어 리팩토링에 포괄 테스트가 동반됐기 때문. **안정성 확보와 테스트 보강이 실질적으로 동시에 진행된 셈.**

### 4. 병렬 처리 도입 효과

| 작업 | Before | After | 이론적 개선 |
|------|--------|-------|------------|
| 크롤링 5건 | 직렬 15초+ | `buffer_unordered(5)` 3초 | ~5배 |
| LLM 요약 5건 | 직렬 25초+ | `buffer_unordered(5)` 5초 | ~5배 |

`futures::stream::buffer_unordered`로 동시성 상한을 두면서 병렬화. 메모리 누수 방지 + API rate limit 준수.

### 5. DB 이중 조회 제거 — 작지만 의미 있는 최적화

`summary_service`에서 알림 전송 시 `get_user_articles`를 한 번 더 호출하던 것을 메모리에서 직접 Article을 조립하는 방식으로 대체.

- 쿼리 1회 절약 (요약 완료 시마다)
- N+1 패턴 사전 차단
- 코드 의도도 명확해짐: "방금 요약한 기사를 그대로 알림에 사용"

## 잘한 것 (Keep)

### 1. 순서가 맞았다: 안정성 → 아키텍처 → 테스트

A에서 타임아웃/retry를 먼저 잡지 않았다면, B의 병렬화 작업 중 외부 API 장애로 디버깅 지옥에 빠졌을 것이다. C를 마지막에 둔 것도 맞았다 — B에서 변경된 코드에 대해 테스트를 작성해야 의미 있으니까.

### 2. "새 기능 0개" 원칙을 지킨 것

기능 추가 유혹 없이 순수 품질 개선에만 집중했다. 결과적으로 코드가 2,140줄 늘었지만, 전부 안전장치(retry, size-limit, 테스트)와 구조 개선이다. 사용자 관점에서 보이는 건 "더 빨라짐"과 "안 죽음" 뿐.

### 3. MVP1 회고의 지적사항을 정확히 해결한 것

MVP1 회고에서 지적한 핵심 문제 3가지를 정확히 타격했다:

| MVP1 회고 지적 | MVP1.5 해결 |
|---------------|------------|
| "테스트 커버리지 90% 미달" | 웹 99.28% 달성, 서버 +70 테스트 |
| "코드 리뷰 이슈 23건 미해결" | 9+2=11건 해결 |
| "의존 방향 위반 2건" | 0건으로 복원 |

### 4. 향후 개선 메모를 즉시 해결한 것

B 코드리뷰에서 발견한 2건(DB 이중 조회, chunked size check)을 "MVP2에서"로 미루지 않고 바로 해결. 기술 부채를 해소하는 마일스톤에서 새 기술 부채를 만들지 않겠다는 의지.

## 아쉬운 것 (Problem)

### 1. 서버 커버리지 정량 미측정

`cargo-tarpaulin`이 환경 문제로 실행되지 않아 서버 커버리지를 정확한 수치로 확인하지 못했다. 테스트 69→139로 2배 증가했지만, 실제 라인 커버리지 90%를 달성했는지는 미확인 상태.

**교훈**: CI에 tarpaulin을 포함시켜 커밋마다 자동 측정해야 한다.

### 2. 마일스톤 A 비중 과다 (65%)

안정성 확보가 전체 작업량의 65%를 차지했다. `http_utils.rs` 559줄이 핵심이지만, 4개 어댑터 각각의 retry 통합 코드까지 합치면 infra 레이어에 변경이 집중됐다. 아키텍처(B)와 테스트(C)에 더 균등한 투자가 있었다면 좋았을 것.

### 3. 테스트 내 구현체 결합

`collect_service` 테스트에서 `SearchFallbackChain` 구현체를 직접 import해서 사용한다. 포트/어댑터 패턴의 취지상 trait mock으로 대체하면 테스트 격리가 향상되지만, 현실적으로 FakeSearchAdapter + SearchFallbackChain 조합이 충분히 동작하므로 trade-off.

## 놀랐던 것 (Surprise)

### 1. 기술 부채 해소가 기능 추가보다 코드량이 많다

새 기능 0개인데 +2,646줄. 대부분 테스트 코드와 안전장치다. "기술 부채 해소 = 코드 줄이기"라는 직관과 반대. 실제로는 "빠진 안전장치를 채우는 것"이 핵심.

### 2. 테스트가 2배로 늘어도 실행 시간은 1초 미만

서버 139개 테스트가 0.86초, 웹 77개가 0.05초. Fake 어댑터 기반 테스트의 진가 — 외부 의존 없이 메모리에서 완결되므로 테스트 수가 늘어도 실행 시간이 선형으로 증가하지 않는다.

### 3. 회고 문서의 실전 가치

MVP1 회고에서 지적한 Critical 3건(타임아웃, auth Client, 에러 노출)이 그대로 MVP1.5의 A1~A3이 됐다. 회고를 "느낌 정리"가 아닌 "다음 작업의 입력"으로 활용한 첫 사례. 앞으로도 회고 → 기획 파이프라인을 유지할 것.

## 회고를 마치며

MVP1.5는 "만든 것을 다듬는 시간"이었다. 새 기능 없이 하루를 온전히 품질에 투자한 것은 속도 압박 속에서 쉽지 않은 결정이지만, 결과적으로 맞았다.

MVP1이 "핵심 가치 검증"이었다면, MVP1.5는 "그 위에 안전하게 쌓을 수 있는 기반 확보"다. 타임아웃 없이 프로덕션에 올렸다면 첫 번째 외부 API 장애에서 전체 서버가 멈췄을 것이고, 의존 방향 위반을 그대로 뒀다면 MVP2에서 새 포트 추가할 때마다 순환 의존과 싸웠을 것이다.

테스트 135 → 216, 의존 위반 2 → 0, retry/size-limit 0 → 전체 적용. 이제 MVP2에서 학습 기능을 올릴 준비가 됐다.

> *"기능을 추가하기 전에, 기존 코드가 넘어지지 않을 자신이 있어야 한다" — 2026.04.05, MVP1.5 완료*
