## 핵심 의사결정과 그 이유

### 결정 1: MVP 범위를 어디서 끊을 것인가

- **상황**: M0~M6으로 잘 쪼갠다고 생각했다. 근데 막상 만들기 시작하니까, "이것도 있어야 하지 않나?" 같은 유혹이 계속 왔다.
- **선택지들**:
  - A) 기능 하나 더 넣자 (태그 자동 추천, 북마크 등)
  - B) 지금 범위에서 칼같이 끊자
- **결정**: B) 칼같이 끊기 (하지만 완벽하진 못했다)
- **왜**: 중간에 태그 필터 탭이나 레이지 로딩 같은 걸 추가했는데, 이건 "기능 추가"라기보다 "기본 UX가 안 되면 사용 자체가 불가능"한 것들이었다. "스터디 기능", "AI 채팅" 같은 큰 기능들은 확실히 MVP2로 미루었다.
- **인사이트**: MVP 범위를 끊는 건 한 번에 되는 게 아니다. 만들면서 계속 재조정하는 거였다. 중요한 건 "이게 핵심 루프에 필요한가?"라는 질문을 매번 던지는 것.

### 결정 2: Supabase REST → sqlx 직접 연결로 전환

- **상황**: 아침에 에코 서버 패턴으로 Supabase REST API만 쓰기로 했었다. 근데 실제로 구현해보니 REST API의 한계가 나왔다 — 복잡한 쿼리, 트랜잭션, 타입 안전성에서 마찰이 컸다.
- **선택지들**:
  - A) REST API를 계속 밀고 가기
  - B) sqlx로 전환 (직접 PostgreSQL 연결)
- **결정**: B) sqlx로 전환
- **왜**: "기획 때 결정한 거니까 밀어야지"라는 생각과 "실제로 써보니 안 맞는데 왜 고집하지?"라는 생각이 충돌했다. 포트/어댑터 패턴의 진짜 가치는 "REST든 sqlx든 어댑터만 교체하면 된다"는 거였고, 실제로 그걸 증명한 셈이다.
- **인사이트**: 아키텍처를 잘 잡아놓으면 기술 선택을 바꿔도 충격이 적다. 포트 추상화 덕분에 서비스 레이어는 한 줄도 안 고쳤다. 이게 포트/어댑터의 진짜 힘이었다.

### 결정 3: 버그를 다 고치고 나서야 "완료"

- **상황**: M0~M6을 다 만들고 나서 버그가 쏟아졌다. JWT 인증 실패, 중복 기사, 날짜 타입 불일치, LLM 요약 실패(Qwen3 thinking 모드), 태그 필터 소멸... 커밋 11개가 전부 버그 수정이다.
- **왜 중요했나**: 기능만 만들고 "MVP 완료!"라고 할 수 있었다. 근데 버그가 있으면 데모도 못 하고, 자기 자신도 쓸 수 없다. "작동하는 소프트웨어"가 MVP의 핵심인데, 작동 안 하면 MVP가 아니다.
- **인사이트**: 기능 구현보다 버그 수정에 시간이 더 걸렸다. 특히 서버-클라이언트 연결 쪽 버그가 많았는데, 이 영역에 대한 경험이 부족해서 삽질을 많이 했다.

## 인사이트 & 피드백

### 1. "전부 다 어려웠다" — 그리고 그게 정상이다

Rust도 처음, Svelte도 처음, Supabase도 처음, 서버 개발도 처음. 근데 되돌아보면 "전부 다 모르는 상태에서 전부 다 만들었다"는 것 자체가 오늘의 가장 큰 성과다.

어려운 게 당연한 거고, 어려운데도 진행할 수 있었던 건 바이브 코딩 덕분이다. 멘토(Claude)가 코드를 짜주니까 나는 "뭘 만들지"와 "왜 이렇게 만들지"에 집중할 수 있었다.

### 2. 서버-클라이언트 연결이 가장 큰 삽질 영역

iOS 개발자로서 앱 내부 로직은 익숙하지만, "브라우저 → SvelteKit 프록시 → Rust API → Supabase DB"라는 체인은 처음이었다. JWT 인증, CORS, RLS 정책, 프록시 라우팅... 각각은 이해할 수 있는데 체인으로 엮이면 디버깅이 힘들었다.

**다음에**: 체인의 각 단계를 독립적으로 먼저 검증하고 나서 연결해야 한다. 한꺼번에 연결하고 나서 어디서 문제인지 찾느라 시간을 많이 썼다.

### 3. 기획 때의 결정을 뒤집는 건 실패가 아니다

아침에 "Supabase REST만 쓴다!"고 했는데 오후에 sqlx로 바꿨다. 처음에는 "내가 틀렸나?"라는 생각이 들었지만, 포트/어댑터 패턴 덕분에 전환이 매끄러웠다. 좋은 아키텍처는 결정을 바꿔도 충격이 적은 구조를 만드는 것이다.

### 4. 워크플로우에 아직 익숙하지 않다

`/workflow`의 9단계를 따라가면서 만들었는데, 중간에 다른 길로 많이 샜다. 버그가 터지면 당장 그걸 고치느라 다음 마일스톤을 잊고, 마일스톤을 진행하다가 옆길로 빠지고. 하루 만에 완주하긴 했지만 효율은 떨어졌던 것 같다.

## 배운 것

### 기술적으로

| 영역 | 배운 것 |
|------|---------|
| **Rust/Axum** | 핸들러 구조, State 주입, trait 기반 포트 패턴 |
| **SvelteKit** | 서버 프록시 (+server.ts), Svelte 5 runes ($state, $derived, $effect) |
| **Supabase** | Auth(JWT), RLS 정책, REST API → sqlx 전환 |
| **OpenRouter** | LLM 호출 구조, Qwen3 thinking 모드 이슈 |
| **Docker** | Docker Compose, 헬스체크, Cloudflare Tunnel |
| **인프라** | 배포 스크립트, iMessage 자동화 (AppleScript) |

### 프로세스에서

- 기획과 구현은 선형이 아니라 반복적 — 만들면서 기획이 바뀐다
- MVP 범위는 "한 번 정하고 끝"이 아니라 계속 재조정하는 것
- 바이브 코딩에서 내 역할은 "의사결정자" — 코드는 AI가, 방향은 내가

## 느낀 점

솔직히 오늘은 "만족하는 결정"보다는 "하나씩 흐름을 찾아가는" 느낌이었다. 기획할 때는 자신 있었는데, 막상 코드가 나오기 시작하니까 모르는 게 쏟아져 나왔다. 특히 서버와의 연결 부분은 iOS 개발에서는 경험할 수 없었던 영역이라 많이 헤맸다.

그래도 "전부 다 어려운데 전부 다 해냈다"는 게 뿌듯하다. 커밋 28개가 그 증거다. 완벽하진 않지만 **작동하는 소프트웨어**가 있다. 그게 MVP의 핵심 아닌가.

바이브 코딩이 "쉬운 코딩"은 아니라는 것도 느꼈다. 코드를 직접 안 짜는 것뿐이지, 모든 결정은 내가 내려야 한다. 오히려 결정의 빈도가 높아서 더 집중력이 필요한 작업이었다.
