## 0. 한 줄 요약

프로젝트는 **하네스 원칙을 문서로는 잘 반영**했다. 서브 룰북, 에이전트 역할 분리, `/step-7` 리팩토링 3R + 리뷰 3R 같은 장치가 이미 갖춰져 있다. **남은 과제는 문서화된 원칙을 자동 실행 가능한 기계 장치로 격상**하는 방향이다. 즉 "해야 한다"고 쓴 것을 "어기면 막는다 / 사람 없이도 굴러간다"로 바꾸는 일이다.

## 1. 두 문서 핵심 원리

### 왜 이 두 문서인가

둘 다 2026년 초에 AI 에이전트가 **혼자서 오래(수 시간~수 개월) 일하게 하는 법**을 다룬 실전 보고서다. Anthropic은 Claude 연구를 통해 "왜 AI가 긴 작업에서 실패하는지"를 4가지 유형으로 정리했고, OpenAI는 "5개월 동안 코드 한 줄도 직접 쓰지 않고 AI로만 100만 줄 프로그램을 완성한" 실제 실험 결과를 공개했다. 두 문서가 강조하는 원리는 다르지만 **공통 철학은 하나**: "AI를 더 똑똑하게 만들기보다 AI가 일할 환경을 잘 만드는 것이 중요하다".

### 1.1 Anthropic — 실패 4유형 × 해결

AI가 긴 작업에서 실패하는 4가지 방식과 해결책.

| 실패 | 해결 |
|---|---|
| ① 대충 끝내려 함 | **체크리스트** |
| ② 다 했다고 거짓말 | **체크리스트 + 다른 AI 검사** |
| ③ 원래 목표 잊음 | **인수인계서** |
| ④ 자기 답안 자기 채점 | **다른 AI 검사 + 5~15회 반복** |

**원문 용어 매핑** (헷갈림 방지):
- **체크리스트** = Feature List = 기능 목록 = 피쳐 리스트
- **다른 AI 검사** = 평가자 분리 = 다중 에이전트
- **인수인계서** = 컨텍스트 리셋

**②와 ④는 원인이 같다** — 둘 다 "자기 평가 편향". 그래서 같은 약(다른 AI 검사)이 통함.

### 1.2 OpenAI — 4대 원리 (100만줄/5개월 프로젝트)

3~7명이 5개월간 AI로만 100만 줄 프로그램을 만들면서 쌓은 교훈.

| 원리 | 무엇인가 | 왜 중요한가 |
|---|---|---|
| ① **AGENTS.md (지도+도서관)** | 100줄짜리 짧은 안내서 + 상세 내용은 별도 `docs/` 폴더 | AI가 읽어야 할 안내서가 너무 길면 정작 작업에 집중 못 함. 얇은 가이드 + 필요 시 찾는 도서관 구조 |
| ② **기계적 규칙 강제** | 6-layer 아키텍처를 린터가 자동 검사. 위반 시 거부 + **"이렇게 고치면 됨" 힌트까지 제공** | AI는 상황 따라 다르게 판단함. "해야 한다"만으론 안 지켜짐. 기계가 막아야 함 |
| ③ **피드백 루프** | AI가 Chrome DevTools로 자기가 만든 화면을 직접 보고, 성능 기준치(예: "0.8초 이내 기동")를 통과할 때까지 스스로 고침 | 사람이 매번 확인해주면 속도 안 남. AI가 자기 결과를 자기가 검증하는 자율 루프 필요 |
| ④ **가비지 컬렉션** | AI가 매일 낡은 코드·문서 청소 PR을 자동으로 올림. 1분 내 자동 반영 | 쓰레기가 쌓이면 전체 시스템이 느려짐. 큰 리모델링 대신 매일 조금씩 청소 |

> **엔지니어의 새 역할**: OpenAI가 말한 "요리사 → 주방 설계자". 직접 코드를 쓰지 않고 **AI가 일할 환경을 설계**하고, **실수가 나오면 환경에 박제**하여 재발 방지한다.

## 2. 채점 루브릭

### 왜 루브릭이 필요한가

이 노트의 스코어(★, n/10)가 **"왜 그 점수인지"** 설명 없으면 주관적 느낌으로 읽힌다. 루브릭을 먼저 정해두면 같은 기준으로 재채점 가능하고, 나중에 봐도 "내가 왜 이렇게 매겼는지" 추적 가능하다.

### 2.1 정합도 (★ 기준)

| 등급 | 정의 | 언제 이 등급인가 |
|---|---|---|
| ★★★★★ | 문서 + 기계 강제 양쪽 | 원칙이 룰북에 있고 hook/lint/CI가 자동으로 막음 |
| ★★★★☆ | 문서화 + 일부 기계 강제 | 원칙은 문서에 있으나 강제 장치가 일부만 있거나 약함 |
| ★★★☆☆ | 문서만, 강제 없음 | 문서에 있지만 어겨도 알림 없음 |
| ★★☆☆☆ | 부분 구현 | 원칙 자체가 일부만 반영됨 |
| ★☆☆☆☆ | 언급 수준 | 이름만 있고 실질 구현 없음 |

### 2.2 스코어 (n/10)

| 점수 | 의미 | 해석 |
|---|---|---|
| 9~10 | 업계 모범 사례 | 이 영역에선 더 개선할 여지 적음 |
| 7~8 | 실무 적용 충분 | 핵심은 갖춰짐, 일부 자동화 여지 있음 |
| 5~6 | 기계 강제 부족 | 문서·운영 수준은 있으나 어기면 막는 장치가 약함 |
| 3~4 | 실천 산발적 | 원칙은 인지하고 있으나 일관성 부족 |
| 1~2 | 부재 | 해당 장치 없음 |

### 한계

같은 항목을 다른 시점에 다시 채점하면 **1~2점 변동 가능**. 특히 "실무 적용 충분(7)"과 "기계 강제 부족(6)" 경계가 주관적이다. 따라서 **절대 등급이 아닌 상대적 진단**으로만 사용한다. "X는 Y보다 부족하다"가 핵심이지 "X는 객관적으로 6점이다"가 아니다.

## 4. Strong Fit (정합된 부분)

### 이 섹션의 의미

두 문서가 강조하는 원칙 중 프로젝트가 **이미 잘 반영하고 있는 부분**을 정리한다. ★5가 아닌 항목에는 "감점 사유"를 ⚠로 표시하되, **수정할 필요가 있는지**를 명확히 한다 — 감점 사유가 있다고 다 고쳐야 하는 건 아니다.

### ⚠ 행동 권고 3종

- **[넘어가기]**: 감점 사유이지만 지금 수정 불필요 (이유 명시)
- **[선택적]**: 여유 있을 때 개선하면 ★5 가능
- **[Gap 이관]**: 별도 Gap 항목에서 관리. Strong Fit 자체는 유지

### 4.1 AGENTS.md = 지도+도서관 (OpenAI ①) — ★★★★☆

**왜 강점인가**: `CLAUDE.md`(173줄)는 "지도" 역할, `rules/sub/*.md` 9개는 상세 "도서관" 역할을 맡는다. 언어 스코프 룰(`.claude/rules/*-scope.md`)은 rust/swift/svelte/python-ml 작업 시 자동 로드되어 컨텍스트 효율이 높다. OpenAI가 100만줄 프로젝트에서 도입한 구조와 거의 동일하다.

**⚠ 감점 사유**: CLAUDE.md가 OpenAI 권장 "100줄"을 초과(173줄). **[넘어가기]** — OpenAI 기준은 100만줄 대형 프로젝트에서 컨텍스트 과부하를 막기 위한 수치다. 현재 프로젝트는 MVP 규모라 173줄이 오히려 적절한 밀도다. 세션 시작 지연이나 가독성 문제가 체감될 때 분할한다.

### 4.2 기계적 규칙 강제 (OpenAI ②) — ★★★★★

**왜 강점인가**: 프로젝트는 두 계층의 hook과 deny 규칙으로 **위험 명령·형식 위반·테스트 미통과를 기계적으로 차단**한다.

| 계층 | 어디 | 역할 |
|---|---|---|
| Git pre-commit hook | `.git/hooks/pre-commit` | main 직접 커밋 차단, 테스트 미통과 차단 |
| Git commit-msg hook | `.git/hooks/commit-msg` | 커밋 메시지 형식(tag: 제목) 강제 |
| Claude Code deny | `settings.json` | `git push`, `rm -rf`, `.env 수정` 등 위험 명령 자동 거부 (7건) |
| Claude Code PreCompact hook | `settings.json` | 컴팩션 전 대화 로그 자동 백업 |

"해야 한다"가 아니라 **"어기면 막는"** 수준이다. OpenAI가 강조한 "기계적 규칙 강제"의 상당 부분이 이미 실현됨.

> 참고: `settings.json`의 Stop hook은 이모지 출력만 하는 placeholder다. 하지만 **실질 검증(테스트·린트·형식)은 Git hook이 모두 담당**하므로 Stop hook이 비어있어도 기능상 공백은 없다. §9에서 정리 예정.

### 4.3 다중 에이전트 · 평가자 분리 (Anthropic ③) — ★★★★★

**왜 강점인가**: `/debate` 스킬이 Claude(Anthropic) + Codex(OpenAI) + Serena(MCP 심볼 분석) 3자 토론을 진행한다. **서로 다른 회사·다른 방법**이라 편향이 부분적으로 독립되어 있다. 역할 에이전트 7종도 작성자와 평가자를 분리한다. 특히 `/step-7`은 Claude 작성 / Codex 리뷰 구조로 "자기 답안을 자기가 채점"하는 실패 유형 ④를 방지한다. MVP 단계 코드 리뷰 영역에서는 이 조합만으로 충분한 독립성이 확보된다.

### 4.4 반복적 개선 (Anthropic ④) — ★★★★☆

**왜 강점인가**: `/step-7` 스킬이 리팩토링 3R(R1/R2/R3) + 리뷰 3R(보안/아키텍처/최종) 구조를 갖는다. Anthropic이 프론트엔드 디자인 작업에서 5~15회 반복 적용해 품질을 끌어올린 방식과 동일한 아이디어다. 한 번에 완벽한 결과를 요구하지 않고 **만들고→평가하고→다시 만들기**를 반복한다.

#### G3. UI 자율 피드백 루프 `[상태: 미구현 — Chrome DevTools MCP 연결 완료로 도입 시점 낮아짐]`

**무엇인가**: Playwright로 AI가 **자기가 만든 화면을 직접 보고, 스스로 수정 반복**하는 자율 루프. OpenAI가 "밤새 일하는 AI"라 부른 구조.

```
AI UI 구현 → 스크린샷 자동 촬영 → 기준과 비교 →
다르면 AI가 자동 코드 수정 → 다시 스크린샷 → 일치할 때까지 반복
```

**핵심 전제 — "기준"이 있어야 작동**:

비교는 반드시 **기준**이 있어야 가능하다. 기준 없이 AI가 자기 구현을 자기가 평가하면 Anthropic이 지적한 **"자기평가 실패"** 원칙 위반.

| 상황 | 기준 | G3 효과 |
|---|---|---|
| **A. 사용자가 이미지·디자인 시안 제공** | 시안 | ✅ 가장 이상적 |
| **B. AI 자율 판단만** ("좋아요 버튼 추가해줘") | **없음** | ❌ 비교 불가 |
| **C. 회귀 테스트** | 이전 상태 | ✅ 가능 |
| **D. 명백한 시각 버그** | 정상 렌더 상식 | ✅ 가능 |

**C. 회귀 테스트란?** "이전엔 잘 되던 게 새 변경으로 깨졌는지" 확인. 기준은 **"변경 전 상태"**.

```
어제: 피드 화면 정상 → 스크린샷 찍어둠 (기준)
오늘: 좋아요 버튼 추가 (코드 변경)
     → 오늘 피드 화면 스크린샷
     → 어제 것과 비교
     → 내가 안 건드린 다른 부분이 깨졌는지 확인
```

**D. 명백한 시각 버그**: 시안 없이도 누가 봐도 깨진 경우.
- 텍스트가 화면 밖으로 튀어나옴 (overflow)
- 버튼이 다른 요소 위에 겹침
- 이미지가 빈 박스로 보임
- 레이아웃이 무너짐

→ AI가 스크린샷 보고 "이거 이상하다" 감지 가능. 기준은 **"정상적 렌더 상식"**.

**비용 — 토큰 많이 씀**:

**Playwright를 쓰면 반환값(이미지·DOM)이 커서 토큰을 많이 쓴다.**

| 도구 | 반환값 | 토큰 |
|---|---|---|
| `browser_take_screenshot` | 이미지 (비전 처리) | 1회당 수백~수천 |
| `browser_snapshot` | DOM 트리 전체 | 1회당 수천~수만 |

자율 루프 5~15회 반복하면 **1작업당 수만~수십만 토큰** 사용.
→ 모든 UI 작업에 적용 = **과투자**.

**어떻게 채울까**: `/step-8`에 UI 변경 감지 시 자동 스크린샷 + 기준과 diff + 불일치 시 `/step-6` 재진입 파이프라인.

**언제 꺼낼까**: 다음 **모두** 해당될 때 (둘 중 하나만으론 부족):
- 디자인 시안/기준 이미지가 있음
- 같은 UI를 3번 이상 반복 수정 중 (수동보다 자동이 싸짐)

**현재 MVP 단계에서의 가치**:

개인 MVP라 디자인 시안 없이 기능 중심으로 작업한다면 **A 시나리오가 거의 없음**. 따라서 G3의 실용 가치는 **제한적**이고, 더 낮은 우선순위가 맞다. 디자인 중요도가 높아지는 시점이 오면 재고.

#### G4. MVP 완료 아카이빙 자동 알림 `[상태: ❌ 진행 안 함 — /workflow step-9 자동 제안으로 종결 (2026-04-20)]`

**무엇인가**: MVP 모든 마일스톤이 끝났을 때, **"이제 아카이빙하세요"를 AI가 자동 알림**. 사용자는 승인만 누르면 `/progress-cleanup` 실행.

**현재 작업 흐름**:
```
마일스톤 리뷰 → MVP + 내부 마일스톤 흐름 잡기
   ↓
마일스톤 스킬 → 각 마일스톤 상세 기획
   ↓
세션 단위로 마일스톤 하나씩 진행 → 완료 시 로드맵에 "완료" 표시
   ↓
모든 마일스톤 끝 → /progress-cleanup 으로 아카이빙 (progress/ → history/)
```

**지금 무엇이 부족한가**: `/progress-cleanup` 스킬은 존재하지만 **사용자가 "MVP 끝났지"를 기억해서 직접 실행**해야 한다. 잊으면 `progress/`에 낡은 파일이 계속 쌓임.

**실측 — G4 감지에 필요한 자산이 이미 갖춰져 있음**:

1. **규칙 명시**: `rules/sub/milestone.md`에 *"로드맵은 프로젝트당 1개"*, *"마일스톤 완료 시 `_roadmap.md` 타임라인 업데이트 필수"* 명시됨
2. **실제 파일 존재**: `history/mvp4/`, `history/mvp9/`, `history/mvp10/`에 `_roadmap.md` 확인
3. **상태 기록 형식**: 마일스톤별 "완료" 상태가 명확한 테이블로 기록됨

실제 `history/mvp10/_roadmap.md` 예시:
```markdown
| 마일스톤 | 목표 | 기간 | 상태 |
|----------|------|------|------|
| M1 | iOS 퀴즈 UX 완성 | 1~2일 | 완료 |
| M2 | 버그 수정 | 1~2일 | 완료 |
| M3 | 피드 성능 | 2~3일 | 완료 |
```

→ **파싱 쉬운 고정 형식**. G4는 새 구조를 만들 필요 없이 **기존 자산 위에 얹기만** 하면 됨.

**어떻게 채울까 — 감지 + 알림**:

> **참고**: Claude Code 훅은 **파일 변경 이벤트를 직접 수신하지 않는다**. 실제 구현은 **PostToolUse 훅**(`Edit`/`Write` 후)에서 경로가 `_roadmap.md`인지 체크하는 방식. 훅은 JSON `additionalContext` 필드로 Claude 컨텍스트에 제안 주입 가능 (2026-04-17 공식 문서 확인).

```
사용자가 마일스톤 완료 시 _roadmap.md 에서 해당 M{N} 상태를 "완료"로 갱신
   ↓
[PostToolUse 훅] Edit/Write 파일 경로가 _roadmap.md인지 확인
   ↓
[스크립트] 로드맵 파싱 → 모든 마일스톤 "완료" 상태?
   ↓ YES
[system reminder 주입] "📢 MVP의 모든 마일스톤 완료 확인.
            아카이빙할까요? (progress/ → history/mvp{N}/)
            [승인] [거부] [나중에]"
   ↓
사용자: 승인 → /progress-cleanup 스킬 호출
```

**왜 완전 자동이 아닌가 — 롤백 비용**:

"완료" 감지되자마자 바로 아카이빙하면 편하겠지만 **롤백 비용이 크다**.

| 완전 자동 | 알림 + 승인 |
|---|---|
| 체크 실수 시 수십 개 파일 즉시 이동 → 되돌리기 번거로움 | 승인 1번만 누르면 안전망 확보 |
| 회고 더 쓰고 싶거나 문서 추가하고 싶은 타이밍에도 이미 이동됨 | 사용자가 타이밍 조절 가능 |
| 진행 중인 다른 작업에 혼란 가능 | 적절한 시점에 실행 |

원칙: **"강제 적용 범위 큰 작업(파일 이동 등)은 승인 필수"**.

즉 G4는 **"까먹지 않게 알림 + 실수 방지 승인"**의 조합. 완전 자동은 편의 ↔ 롤백 비용의 트레이드오프에서 불리.

**언제 꺼낼까**: MVP 여러 개를 돌리면서 아카이빙을 자주 잊는 시점.

- **단점**: 사소한 작업도 체크리스트 생성 → 약간의 오버헤드
- **범위**: 이 항목은 **UI/동작 수동 테스트 강제** 전용. 코드 리뷰(step-7)는 이미 LLM 3자 분리로 충분하므로 무관
