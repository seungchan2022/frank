## 목표

피드를 실사용 수준으로 끌어올린다.  
성능(병렬 검색), 시각적 완성도(썸네일·카드 UI), 탐색 편의성(태그탭 독립 새로고침), 콘텐츠 품질(마크다운 렌더링) 순으로 개선한다.

## UX 확정 사항

| 항목 | 결정 |
|------|------|
| 카드 레이아웃 | 왼쪽 썸네일 + 오른쪽 텍스트 (뉴스앱 표준) |
| 썸네일 없을 때 | 동일 비율 플레이스홀더 표시 (레이아웃 고정) |
| stale-while-revalidate 스크롤 | 새 데이터 도착 시 상단으로 자동 스크롤 |
| 태그탭 전환 캐시 | 이전 탭 결과 캐시 유지, 재요청 없음 |
| 마크다운 렌더링 수준 | 굵게 / 이탤릭 / 목록 + 줄바꿈 (기본 마크다운) |
| 마크다운 미사용 요소 | 코드블록 / 테이블 제외 |

## 실행 순서 근거 (재작업 방지)

`feed.rs`와 `FeedItemResponse` 모델이 M1~M3 공통 변경 대상이므로 순서가 중요하다:

```
M1 서버: FeedItemResponse에 image_url 추가 → 모델 최종 확정
M2 서버: 확정된 모델 위에서 join_all 로직 추가
M3 서버: 완성된 feed.rs 위에서 tag_id 파라미터 추가
M4 서버: summarize.rs만 건드림 — 완전 독립
```

클라이언트도 동일 원칙:
```
M1 클라이언트: 카드 UI 최종 레이아웃 확정 (썸네일 포함)
M2 클라이언트: 확정된 카드 위에서 stale-while-revalidate UX
M3 클라이언트: 완성된 피드 페이지 위에서 태그탭 재요청
M4 클라이언트: 디테일 뷰만 변경 — 독립
```

## M1: 썸네일 — 모델 확정 + Tavily image_url + DB image_url

### 목표
`FeedItemResponse`와 `favorites` DB에 `image_url`을 추가해 이후 마일스톤의 모델 기준을 확정한다.  
카드 UI를 썸네일 포함 최종 레이아웃으로 전환한다.

### 서버

**변경 파일**: `infra/tavily.rs`, `domain/models.rs`, `api/feed.rs`

- `TavilyResult`에 `image_url: Option<String>` 추가 파싱
- `FeedItemResponse`에 `image_url: Option<String>` 추가
- `favorites` 테이블 `image_url text` 컬럼 마이그레이션
- `POST /me/favorites` 요청 바디에 `image_url` 추가
- `GET /me/favorites` 응답에 `image_url` 포함

### 웹 + iOS (서버 완료 후 동시 진행)

**웹**:
- `FeedItem` 타입에 `image_url?: string` 추가
- 카드 컴포넌트 레이아웃: `[썸네일 72×72] [제목 / source / 발행일]` (왼쪽 고정 썸네일)
  - `image_url` 있으면: `<img>` 표시
  - `image_url` 없으면: 동일 크기 회색 플레이스홀더 (아이콘 또는 배경색)
- 즐겨찾기 목록(`/favorites`)도 동일 카드 컴포넌트 사용

**iOS**:
- `FeedItem` 모델에 `imageUrl: String?` 추가
- `ArticleCardView` 레이아웃: `HStack { [AsyncImage 72×72] [VStack 제목·source·발행일] }`
  - `imageUrl` 있으면: `AsyncImage` 표시
  - `imageUrl` 없으면: 동일 크기 `RoundedRectangle` 플레이스홀더
- 즐겨찾기 목록도 동일 카드 컴포넌트 사용

### DoD
- [x] `TavilyResult` + `FeedItemResponse`에 `image_url` 추가
- [x] `favorites` DB `image_url` 컬럼 마이그레이션
- [x] 웹 카드 UI — 썸네일 표시 (없으면 플레이스홀더)
- [x] iOS 카드 UI — AsyncImage 썸네일 + 제목 + source
- [x] 즐겨찾기 목록 썸네일 표시 (웹 + iOS)
- [x] 테스트 통과 (서버 178 / 웹 146 / iOS 155)

> **완료**: 2026-04-10  
> **이미지 소스 변경**: Tavily `images[]` 배열 → 기사 URL `og:image` 직접 크롤링 (병렬, 5s 타임아웃)  
> **추가 버그픽스**: URL 정규화 기반 중복 제거 (http/https, www, trailing slash 통일)

## M2: 피드 성능 — 병렬 검색(join_all) + stale-while-revalidate

### 목표
태그 수에 관계없이 피드 응답시간을 일정하게 유지한다.  
로딩 중에도 빈 화면 없이 이전 피드를 표시한다.

### 서버

**변경 파일**: `api/feed.rs`, `services/feed_service.rs`

- 태그별 `search()` 호출을 `tokio::join_all()`로 병렬 전환
- 개별 태그 실패 시 해당 태그 skip (전체 에러 아님) — 기존 동작 유지
- 응답 구조 변경 없음 (M1에서 이미 확정)

### 웹 + iOS (서버 완료 후 동시 진행)

**웹**:
- `feedStore`: 피드 새로고침 시 기존 `items` 유지한 채 `isRefreshing: true` 상태로 백그라운드 요청
- 새 결과 수신 완료 시 `items` 교체 + 스크롤 상단으로 이동
- 로딩 인디케이터: 카드 목록 위 최상단 얇은 progress bar (카드 교체되지 않음)

**iOS**:
- `FeedFeature`: `isRefreshing` 상태 분리 (`feedItems`는 유지)
- 새 결과 수신 후 `feedItems` 교체 + `scrollPosition` 상단으로 reset
- Pull-to-refresh: 완료 후 자동으로 상단 스크롤

### DoD
- [ ] 서버 `join_all` 병렬 검색 — 태그 N개 동시 요청
- [ ] 웹 stale-while-revalidate (카드 유지 + 백그라운드 갱신)
- [ ] iOS stale-while-revalidate (feedItems 유지 + 갱신)
- [ ] 테스트 통과 (서버 / 웹 / iOS)

## M3: 태그 탭 필터링 — tag_id 파라미터 + 독립 새로고침

### 목표
특정 태그 탭 선택 시 해당 태그만 서버에 요청한다.  
탭별로 독립적으로 새로고침할 수 있다.

### 서버

**변경 파일**: `api/feed.rs`

- `GET /me/feed?tag_id=<uuid>` 쿼리 파라미터 추가
- `tag_id` 없으면 기존대로 전체 태그 검색 (하위 호환)
- `tag_id` 있으면 해당 태그 1개만 검색

### 웹 + iOS (서버 완료 후 동시 진행)

**웹**:
- `feedStore`: `tagCache: Map<tagId | 'all', FeedItem[]>` 로 탭별 캐시 관리
- 탭 전환 시 캐시 있으면 즉시 표시 (재요청 없음)
- 캐시 없으면 `GET /me/feed?tag_id=xxx` 요청 후 캐시 저장
- "전체" 탭은 `tag_id` 파라미터 없이 호출, 캐시 키는 `'all'`
- 클라이언트 측 tag_id 필터링 로직 제거

**iOS**:
- `FeedFeature`: `tagCache: [UUID?: [FeedItem]]` 딕셔너리로 탭별 캐시 관리
  - `nil` 키 = 전체 탭
- 탭 전환 시 캐시 있으면 즉시 표시
- 캐시 없으면 `GET /me/feed?tag_id=xxx` 요청 후 캐시 저장
- Pull-to-refresh: 현재 탭 캐시만 갱신

### DoD
- [ ] 서버 `tag_id` 쿼리 파라미터 — 단일 태그 검색
- [ ] 웹 태그탭 클릭 → 서버 재요청 (클라이언트 필터 제거)
- [ ] iOS 태그 칩 선택 → 서버 재요청
- [ ] "전체" 탭 동작 유지 (파라미터 없음)
- [ ] 테스트 통과 (서버 / 웹 / iOS)

## M4: 마크다운 렌더링 — LLM 프롬프트 + 클라이언트 렌더러

### 목표
요약/인사이트를 마크다운 형식으로 출력하고, 클라이언트에서 렌더링한다.  
원문 리드(snippet)도 동일하게 처리한다.

> M4 서버는 M1~M3와 독립적(`summarize.rs`만 변경). 서버 작업은 M3 서버와 병렬 진행 가능.

### 서버

**변경 파일**: `services/summary_service.rs` (LLM 프롬프트)

- LLM 프롬프트에 마크다운 출력 지시 추가:
  - 강조: `**굵게**`
  - 이탤릭: `*이탤릭*`
  - 목록: `- 항목`
  - 줄바꿈: 문단 사이 빈 줄
  - **미사용**: 코드블록, 테이블, 헤딩(`#`)
- 응답 구조 변경 없음 (`summary`, `insight` 필드 유지)

### 웹 + iOS (서버 완료 후 동시 진행)

**웹**:
- 패키지: `svelte-markdown` 추가 (`marked` 대신 — Svelte 컴포넌트로 바로 사용 가능)
- 적용 범위: 디테일 페이지 `summary` / `insight` / `snippet` 영역
- 기존 `<p>{text}</p>` → `<SvelteMarkdown source={text} />`로 교체

**iOS**:
- 적용 방식: `Text(try! AttributedString(markdown: text))` (iOS 15+ 네이티브, 서드파티 불필요)
- 적용 범위: `ArticleDetailView`의 `summary` / `insight` / `snippet` 텍스트
- 파싱 실패 시 fallback: 일반 `Text(text)` 표시

- [ ] iOS 요약/인사이트/snippet 마크다운 렌더링
