## A: 안정성 확보

> 목표: MVP2 개발 중 예상치 못한 장애/누수 방지

| # | 이슈 | 현재 문제 | 해결 방법 | 대상 파일 |
|---|------|----------|----------|----------|
| A1 | HTTP 타임아웃 미설정 | 외부 API 무응답 시 스레드 무한 대기 | reqwest Client에 timeout 설정 | `infra/tavily.rs`, `exa.rs`, `firecrawl.rs`, `openrouter.rs` |
| A2 | auth Client 매 요청 생성 | 요청마다 새 Client + Supabase 장애 시 전체 마비 | Client를 AppState에서 재사용 | `middleware/auth.rs` |
| A3 | Internal 에러 메시지 노출 | 내부 에러 상세가 클라이언트에 그대로 전달 | 500 응답에 generic 메시지, 상세는 로그만 | `domain/error.rs` |
| A4 | onAuthStateChange 구독 해제 누락 | 로그아웃/재로그인 시 이전 구독 메모리 누수 | onDestroy에서 unsubscribe 호출 | `lib/stores/auth.svelte.ts` |

## B: 아키텍처 정리

> 목표: "api → services → domain ← infra" 의존 방향 복원 + 성능 병목 해소

| # | 이슈 | 현재 문제 | 해결 방법 | 대상 파일 |
|---|------|----------|----------|----------|
| B1 | sqlx::FromRow domain 침투 | Domain 모델이 sqlx를 import | domain에서 sqlx::FromRow 제거, infra에서 수동 매핑 | `domain/models.rs`, `infra/postgres_db.rs` |
| B2 | SearchFallbackChain 구체 타입 참조 | services가 infra를 직접 import | SearchChainPort trait 추가, services는 trait만 참조 | `domain/ports.rs`, `services/collect_service.rs`, `infra/search_chain.rs` |
| B3 | 크롤링 직렬 처리 | N건 순차 HTTP → 수십 초 지연 | `futures::join_all`로 병렬 크롤링 | `services/collect_service.rs` |
| B4 | LLM 요약 직렬 처리 | N건 순차 API 호출 → 수십 초 지연 | `futures::join_all`로 병렬 요약 (동시 요청 수 제한) | `services/summary_service.rs` |

**완료 기준**: 기존 테스트 전부 통과 + domain/services에서 infra import 0건 + 수집/요약 체감 속도 개선

**결과 (260405)**: ✅ 완료. B1~B4 해결 + RetryConfig 프리셋 리팩토링. domain에서 sqlx import 0건, services에서 infra import 테스트만. 서버 124 + 웹 72 = 총 196 테스트 통과. 커밋: `402cba9`
