## 배경 지식 — 왜 OpenRouter에서 Groq로 바꿨나

### OpenRouter란?
OpenRouter는 AI 모델 중계 플랫폼이다.
GPT-4(OpenAI), Claude(Anthropic), Llama(Meta) 등 수백 개 모델을 하나의 API 키로 사용할 수 있게 해주는 서비스.
모델을 자동으로 골라주는 게 아니라, 내가 모델명을 지정하면 해당 회사로 요청을 중계해준다.

```
Frank 서버 → OpenRouter(중계) → 모델 회사 → 응답 반환
```

Frank에서 사용하는 기능 3가지가 모두 OpenRouterAdapter 하나를 통해 실행됨:
- 키워드 추출 (기사 즐겨찾기 시 자동 실행)
- 요약 (요약 버튼 탭)
- 퀴즈 생성 (퀴즈 풀기 버튼 탭)

### 왜 느렸나?
Frank는 OpenRouter **무료 티어**를 사용 중이었다.
무료 사용자는 유료 사용자 처리 후 대기열 뒤에서 처리됨 → 30초+ 지연 발생.
모델 자체가 느린 게 아니라 중간 대기열이 원인.

### Groq란?
Groq는 AI 추론 전용 칩(LPU)을 만든 회사다.
일반 GPU 대비 AI 연산 속도가 압도적으로 빠르며, 업계 1위 추론 속도를 자랑함.
Frank에서는 Meta의 Llama 3.3 70B 모델을 Groq에서 직접 실행한다.

```
Frank 서버 → Groq(직접 연결, 전용 칩) → 응답 (3~5초)
```

### 핵심
모델 자체(Llama 3.3 70B)는 동일하다.
달라지는 건 **어디서 실행하느냐** — OpenRouter 중계 대기열 없이 Groq 전용 칩에서 직접 실행.
무료 플랜: 하루 1,000회 한도, 초과 시 429 에러(비용 청구 없음, 자동 유료 전환 없음).

## 상세 구현 가이드

### 아이템 1: Groq LLM 어댑터 교체

**배경**
- 현재: `qwen/qwen3.5-plus-02-11` (OpenRouter free tier) → 대기열 30초+
- 목표: Groq `llama-3.3-70b-versatile` → 394 TPS, 3~5초 예상

**Groq API 스펙**
- Base URL: `https://api.groq.com/openai/v1`
- 엔드포인트: `/chat/completions` (OpenAI-compatible)
- 무료 플랜: 30 RPM / 1,000 RPD / 100,000 TPD (Frank 사용량에 충분)
  - 한도 초과 시: **429 에러 반환, 비용 청구 없음** (자동 유료 전환 없음)
- 비용: 입력 $0.59/M, 출력 $0.79/M (OpenRouter free와 달리 대기열 없음)
- 환경변수: `GROQ_API_KEY`
- `response_format: { "type": "json_object" }` → **지원 확인됨** (기존 코드 그대로 사용 가능)
- `reasoning: { "exclude": true }` → **Groq 미지원, 반드시 제거**

**LlmPort 3가지 메서드 모두 교체 대상**
- `summarize` — 기사 본문 → 한국어 요약
- `generate_quiz` — 기사 내용 → 퀴즈 3문제 생성
- `extract_keywords` — 기사 제목/스니펫 → 키워드 5개 추출
  - 현재 openrouter.rs에서 `KEYWORD_EXTRACT_MODEL = "meta-llama/llama-3.3-70b-instruct:free"` 별도 상수로 분리됨
  - Groq에서는 `llama-3.3-70b-versatile` 하나로 통일 (instruct:free 미존재)

**구현 방향**
- `server/src/infra/openrouter.rs` → `server/src/infra/groq.rs` 신규 파일
- 기존 `OpenRouterAdapter`와 동일한 `LlmPort` trait 구현
- 차이점:
  - `base_url`: `https://api.groq.com/openai/v1`
  - 모델명: `llama-3.3-70b-versatile` (summarize, generate_quiz, extract_keywords 모두 동일)
  - `reasoning: { "exclude": true }` 파라미터 **제거** (openrouter.rs 97, 195, 335번째 줄에 있던 것)
  - `response_format: { "type": "json_object" }` 유지 (Groq 지원 확인)
- `server/src/main.rs`에서 `AppState` 빌드 시 `GroqAdapter` 주입 (`OpenRouterAdapter` 대체)
- `server/src/infra/fake_llm.rs` 기존 유지 (테스트용)
- `.env`에 `GROQ_API_KEY` 추가, `OPENROUTER_API_KEY` 제거 또는 유지(미사용)

**테스트**
- `groq.rs` 단위 테스트: wiremock으로 응답 파싱 검증
- 기존 `openrouter.rs` 테스트는 유지 (어댑터 교체와 독립)
- 통합: mock LLM으로 서비스 레이어 테스트는 기존과 동일

### 아이템 2: Tavily `time_range: "week"` 추가

**배경**
- `server/src/infra/tavily.rs:66` body에 날짜 필터 없음
- 같은 키워드로 반복 검색 시 Tavily가 동일 상위 기사 반환

**구현**

```rust
// 변경 전 (tavily.rs:66)
let body = serde_json::json!({
    "query": query,
    "max_results": max_results,
    "search_depth": "advanced",
    "include_answer": false,
});

// 변경 후
let body = serde_json::json!({
    "query": query,
    "max_results": max_results,
    "search_depth": "advanced",
    "include_answer": false,
    "time_range": "week",   // 최근 7일 기사만
});
```

> **주의**: Tavily 문서 확인 결과, `days` 파라미터는 `topic: "news"` 설정 시에만 작동.
> 일반 검색(`topic` 없음)에서는 `time_range: "week"` 사용.

**테스트**
- wiremock 기존 테스트에 body 검증 추가: `"time_range": "week"` 포함 확인

### 아이템 3: snippet 노이즈 패턴 필터 강화

**배경 (실제 사례)**
- `TechCrunch –:–:–:– The first StrictlyVC...` → 사이트 구분자 패턴
- `arrow_forward - #Android Studio - #Compose...` → 아이콘이름 + 해시태그 목록
- `모바일 기기/앱 | PR Newswire ... 00:00 01:00 02:00...` → 시간표 패턴

**기존 `clean_snippet` 한계 (exa.rs:42)**
- HTML 태그, URL, 이메일만 제거
- 위 패턴은 일반 텍스트라 통과

**추가할 필터 규칙 (exa.rs `clean_snippet` 확장)**

```rust
// 1. 시간표 패턴 제거: "00:00 01:00 02:00..." 연속 시간 3개 이상
// 2. 해시태그 목록 제거: "#단어 - #단어 - ..." 3개 이상 연속
// 3. 구분자 패턴 제거: "–:–:–:–" 또는 "|:–" 같은 비문장 구분자
// 4. 아이콘이름 패턴 제거: "arrow_forward", "chevron_right" 등 snake_case 단독 토큰
// 5. 네비게이션 문구: "Skip Navigation", "이전 기사보기", "다음 기사" 로 시작하는 행 제거
```

**적용 범위 확장**
- 기존: `exa.rs`의 `clean_snippet`만 적용
- 추가: `tavily.rs`의 snippet에도 동일 함수 적용
  - `exa.rs`에서 `pub fn clean_snippet` 유지 (이미 pub)
  - `tavily.rs`에서 `use crate::infra::exa::clean_snippet;` import 후 적용

**테스트**
- `clean_snippet` 단위 테스트에 위 패턴별 케이스 추가
- tavily.rs의 snippet 필터링 통합 테스트
