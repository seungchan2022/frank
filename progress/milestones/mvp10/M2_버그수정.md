# M2: 버그 수정 — 키워드 오염 + 에러 메시지

> 프로젝트: Frank MVP10
> 상태: 대기
> 예상 기간: 1~2일
> 의존성: 없음
> 실행 순서: ST-1(서버/DB, M3-ST1과 함께) → ST-2(웹)

## 목표

좋아요 기반 `user_keyword_weights`가 태그와 무관하게 전역으로 누적되어 다른 태그 피드를 오염시키는 문제를 수정하고,
요약 실패 시 기술적 에러 메시지가 그대로 노출되는 문제를 원인별 친화적 메시지로 개선한다.

## 성공 기준 (Definition of Done)

- [ ] 태그 A에서 좋아요한 키워드가 태그 B 피드 쿼리에 붙지 않음
- [ ] 요약 실패 시 웹에서 원인별 사용자 친화적 메시지 표시 (`Internal server error` 미노출)
- [ ] 서버 테스트 통과 (`cargo test`)
- [ ] 웹 테스트 통과 (`npm run test`)
- [ ] 커버리지 90% 유지

## 아이템

| # | 아이템 | 레이어 | 실행 단계 | 유형 | 상태 |
|---|--------|--------|----------|------|------|
| 1 | 서버 `user_keyword_weights`에 `tag_id` 컬럼 추가 — 태그별 키워드 분리 | 서버/DB | 1단계 (M3-ST1과 동시) | feature | 대기 |

> ST-1은 M3-ST1(TTL 캐시)과 같은 `tags.rs`를 수정하므로 **동시에 처리**한다.

## 상세 분석

### ST-1: keyword_weights 태그별 분리 (안 2 확정)

**파일**: `server/src/api/feed.rs`, `server/src/infra/postgres_db.rs`, `supabase/migrations/`

**문제 원인**:

`user_keyword_weights` 테이블이 태그와 무관하게 전역으로 키워드를 누적한다.
태그 A(경제)에서 좋아요한 키워드("금리", "환율")가 태그 B(AI)로 전환해도 그대로 남아
전체 피드 쿼리에 의도치 않은 키워드가 붙는다.

**재현 시나리오**:
1. 태그 "경제" 구독 → 기사 3개 좋아요 → "금리", "환율" 누적
2. 태그를 "AI"로 교체
3. 전체 피드 요청 → 쿼리: `"AI latest news 금리 환율"` → 원하지 않는 결과 혼입

**수정 방향 — 안 2 확정 (tag_id 컬럼 추가)**:

`user_keyword_weights` 테이블에 `tag_id` 컬럼을 추가하여 태그별로 키워드를 분리한다.
좋아요 시 해당 기사의 태그 ID를 함께 저장하고, 피드 쿼리 시 현재 구독 태그 기준 키워드만 조회한다.

- **DB 마이그레이션**: `user_keyword_weights`에 `tag_id uuid NOT NULL` 컬럼 추가
- **좋아요 저장**: `tag_id`를 함께 INSERT
- **키워드 조회**: `get_top_keywords(user_id, tag_ids: Vec<Uuid>, limit)` — 현재 구독 태그 기준 필터
- **전체 피드(태그 없음)**: 현재 구독 태그 전체를 대상으로 키워드 조회

**안 1(태그 변경 시 전체 초기화) 대비 장점**:
- 태그 추가 시 기존 키워드 보존
- 태그 복귀 시 이전 개인화 데이터 재활용 가능
- 누적 데이터 손실 없음

**보완 조건**:
1. **TDD 순서 강제**: 실패 테스트 먼저 작성 → 구현 → 통과 확인
2. **포트 시그니처 변경**: `get_top_keywords` 시그니처에 `tag_ids` 추가 → FakeDbAdapter도 함께 수정
3. **마이그레이션 안전성**: 기존 레코드는 `tag_id = NULL` 허용 후 이후 신규 데이터부터 적용 (또는 전체 초기화)

### ST-2: 웹 요약 에러 메시지 개선 (A안 확정)

**파일**: `web/src/routes/feed/article/+page.svelte`, `server/src/api/summarize.rs`

**문제 원인**:

서버가 모든 실패를 `500 Internal Server Error + { "error": "Internal server error" }`로 반환.
웹 클라이언트가 이 메시지를 그대로 화면에 표시.

**수정 방향 — A안 확정 (서버 원인별 HTTP 코드 반환)**:

서버가 실패 원인을 구분하여 적절한 HTTP 상태 코드를 반환하고,
웹 클라이언트는 상태 코드 기반으로 사용자 친화적 메시지를 선택한다.

**원인별 HTTP 코드 매핑 (확정)**:

| 실패 원인 | HTTP 코드 | 웹 표시 메시지 |
|---------|----------|-------------|
| 크롤링 실패 (기사 내용 불러오기 불가) | 422 | '기사 내용을 불러오지 못했습니다.' |
| LLM 실패 (요약 생성 오류) | 503 | '요약 서비스에 문제가 생겼습니다. 잠시 후 다시 시도해주세요.' |
| 타임아웃 | 408 | '요약 시간이 초과됐습니다. 긴 기사일 수 있습니다.' |

**구현 방향**:
- **서버**: `summarize.rs` 핸들러에서 실패 원인별 분기 후 적절한 HTTP 코드 반환
- **웹**: `+page.svelte` catch 블록에서 `ApiError.status` 기반 메시지 매핑
- `realClient.ts`의 `ApiError` 클래스를 export하여 `instanceof` 체크 활용

**보완 조건**:
1. **TDD 순서 강제**: 실패 테스트 먼저 작성 → 구현 → 통과 확인
2. **서버 + 웹 동시 적용**: 서버 코드 코드 변경 후 웹 클라이언트 메시지 매핑 연동
3. 매핑 외 상태 코드(기타 4xx/5xx): 기존 fallback 메시지 유지
