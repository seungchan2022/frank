# M3: 피드 성능 개선 — 서버 TTL 캐시

> 프로젝트: Frank MVP10
> 상태: 대기
> 예상 기간: 2~3일
> 의존성: M2-ST1과 동시 처리 (1단계 서버/DB에서 함께)
> 실행 순서: M2-ST1과 묶어서 1단계에서 진행 — tags.rs를 두 번 건드리지 않기 위함

## 목표

앱 재시작 후 첫 피드 로딩 속도를 개선한다.
수동 새로고침 / pull-to-refresh / 새 기사 불러오기는 기존과 동일하게 항상 최신 기사를 반환한다.

## 현재 동작 vs 개선 후

| 시나리오 | 현재 | 개선 후 |
|---------|------|--------|
| 앱 재시작 후 첫 피드 | 외부 API 호출 (2~5초) | 서버 캐시 HIT → 수 ms |
| 수동 새로고침 | 외부 API 호출 (2~5초) | **동일** — 캐시 우회, 외부 API 호출 |
| pull-to-refresh | 외부 API 호출 (2~5초) | **동일** — 캐시 우회, 외부 API 호출 |
| 탭 전환 | 클라이언트 캐시 → 즉시 | **동일** — 클라이언트 캐시 처리 |
| 태그 변경 후 첫 피드 | 외부 API 호출 (2~5초) | 캐시 무효화 → 외부 API 호출 (2~5초) |

## 성공 기준 (Definition of Done)

- [ ] 앱 재시작 후 첫 피드 — 서버 캐시 HIT 시 즉시 응답 (외부 API 미호출)
- [ ] 수동 새로고침(`Cache-Control: no-cache`) — 캐시 우회 후 외부 API 호출, 최신 기사 반환
- [ ] TTL 만료(기본 5분) 후 요청 — 외부 API 재호출하여 최신 기사 반환
- [ ] 사용자별 캐시 키 분리 — 다른 사용자 피드 혼입 없음
- [ ] 태그 변경(`POST /api/me/tags`) 시 해당 사용자 캐시 즉시 무효화
- [ ] 캐시 적중/미스 `tracing` 로그 추가
- [ ] 웹/iOS 클라이언트: 수동 새로고침 요청 시 `Cache-Control: no-cache` 헤더 추가
- [ ] 서버 테스트 통과 — hit/miss/no-cache 경로 모두 커버
- [ ] 커버리지 90% 유지

## 상세 분석

### ST-1: 서버 TTL 캐시 (서버/DB)

**관련 파일**:
- `server/src/api/feed.rs` — 캐시 조회/저장 + no-cache 우회 로직
- `server/src/api/tags.rs` — 태그 변경 시 캐시 무효화 (M2-ST1과 동시 처리)
- `server/src/infra/` — InMemoryFeedCache 신규 구현
- `server/src/domain/ports.rs` — FeedCachePort trait 추가
- `server/src/api/mod.rs` (AppState) — FeedCache 필드 추가

**캐시 키 설계**:

```
"{user_id}:{sorted_tag_ids}"
예시: "550e8400:ai,economics,tech"

- 태그 없음(전체 피드): "{user_id}:all"
- 특정 태그 피드: "{user_id}:{tag_id}"
- 태그 ID는 정렬 후 쉼표 구분 (순서 무관 동일 캐시 보장)
```

**캐시 동작 방향**:
- TTL: 5분 (기본값)
- 최대 엔트리: 100개 (초과 시 만료된 항목 우선 제거)
- `Cache-Control: no-cache` 헤더 수신 시 캐시 조회 건너뛰고 외부 API 직접 호출
- 태그 변경 시 해당 user_id 프리픽스로 시작하는 캐시 전체 무효화
- 캐시 HIT/MISS 모두 tracing 로그 기록

**테스트용 NoopFeedCache**: AppState 생성 시 캐시 없이 테스트 가능한 Noop 구현체 제공

---

### ST-2: 웹 no-cache 헤더 (웹)

**관련 파일**: `web/src/routes/feed/+page.svelte` 또는 feed API 호출 부분

**수정 이유**:
서버 캐시(ST-1)가 도입되면 수동 새로고침도 캐시 HIT가 되어 최신 기사를 못 가져오게 됨.
기존 동작(수동 새로고침 = 항상 최신)을 유지하기 위해 클라이언트가 서버에 캐시 우회를 요청해야 함.

**수정 방향**:
- 수동 새로고침 요청 시 `Cache-Control: no-cache` 헤더 추가
- 일반 피드 로딩(앱 진입 시)은 헤더 없이 그대로 → 캐시 활용

---

### ST-3: iOS no-cache 헤더 (iOS)

**관련 파일**: `ios/Frank/Frank/Sources/Core/Adapters/APIFeedAdapter.swift` 또는 피드 네트워크 호출 부분

**수정 이유**: ST-2와 동일 — 서버 캐시 도입 후 pull-to-refresh / 새 기사 불러오기가 캐시 HIT로 최신 기사를 못 가져오는 것을 방지.

**수정 방향**:
- pull-to-refresh 요청 시 `Cache-Control: no-cache` 헤더 추가
- 새 기사 불러오기 요청 시 동일
- 일반 피드 로딩(앱 재시작 시)은 헤더 없이 그대로 → 캐시 활용

## 테스트 계획

| 테스트 | 파일 위치 | 검증 내용 |
|--------|----------|----------|
| 캐시 HIT 시 검색 API 미호출 | `server/src/api/feed.rs` tests | MockSearchChain 호출 횟수 = 0 |
| 캐시 MISS 시 검색 API 호출 후 저장 | 동일 | 호출 횟수 = 1, 이후 HIT |
| `Cache-Control: no-cache` 시 캐시 우회 | `server/src/api/feed.rs` tests | 캐시 있어도 검색 API 호출 |
| TTL 만료 시 재호출 | `server/src/infra/feed_cache.rs` unit tests | ttl=0 설정으로 항상 만료 시뮬레이션 |
| 태그 변경 시 캐시 무효화 | `server/src/api/tags.rs` tests | invalidate 호출 확인 |
| 사용자 격리 | unit | 다른 user_id 키는 별도 엔트리 |
| max_entries 초과 시 만료 엔트리 제거 | unit | store 크기가 max_entries 이하 유지 |

## 리스크

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 태그 변경 외 캐시 stale — 기사 내용 변경 불가 | L | TTL 5분이면 허용 범위. 사용자가 최신성보다 속도를 더 원함 |

| 서버 재시작 시 캐시 소멸 → 첫 요청 느림 | L | 인메모리 허용 범위. Redis는 MVP10 범위 밖 |
| AppState FeedCache 필드 추가 시 기존 테스트 make_test_state 수정 | M | 서버 전체 테스트의 make_test_state에 NoopFeedCache 주입 필요 |
