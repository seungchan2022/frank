## 앱 정체성 재정의 (2026-04-29 대화에서)

기존: "AI 기반 뉴스 스크랩 스터디앱"
→ 수정: **"내 관점으로 트렌드를 보는 개인 레이더"**

퀴즈/오답노트는 부가 기능. 핵심은 **내 배경(예: iOS 개발자)으로 뉴스를 읽고, 관련 인사이트를 쌓아가는 것**.

## 기능 아이디어

### [IDEA-01] 내 관점 기반 AI 인사이트
**상태**: 🟢 **MVP15 흡수** (M3) — 2026-05-01 확정

요약 시 단순 요약이 아닌 사용자 배경(iOS 개발자 등) 기반 맞춤형 인사이트 생성.
예: "iOS 개발자 시각에서 이 기사의 핵심은 X이고, 실무에서 Y로 연결됩니다"
- 구현: 요약 프롬프트에 사용자 프로필/역할 컨텍스트 추가
- 서버 변경 최소 (프롬프트 개선 수준)

### [IDEA-02] 기사에 내 생각 메모
**상태**: 🔴 **보류** (정체성 부적합) — 2026-05-01

기사를 읽고 나서 나만의 코멘트를 남기는 기능.
- 보류 사유: "훑고 넘기는" 트렌드 레이더 정체성과 충돌. 즐겨찾기 아카이빙으로 대체 가능
- DB: `article_memos` 테이블 (user_id, article_url, memo, created_at)
- 웹+iOS 양쪽 지원

### [IDEA-03] 트렌드 키워드 추적
**상태**: 🟡 **조건부 보류** (데이터 누적 후 재평가) — 2026-05-01

내 관심 태그에서 반복적으로 등장하는 키워드 시각화.
예: "이번 달 AI 탭에서 'Agent'가 12번 등장"
- 보류 사유: 사용자 의도 — 기사 ephemeral 유지. 키워드 누적 저장(`keyword_trends`)을 별도로 깔아야 함
- 재평가 시점: MVP17+ (앱 데이터 한 달 이상 누적 후)
- 서버에서 키워드 빈도 집계 필요

### [IDEA-04] 읽기 스트릭 / 학습 통계
**상태**: 🔴 **보류** (방향 부적합) — 2026-05-01

"오늘 3개 읽음 · 5일 연속" 같은 간단한 지표.
- 보류 사유: 앱 정체성이 "학습 동기부여"가 아닌 "트렌드 파악". 스트릭은 학습앱 패턴
- 꾸준히 앱을 열게 만드는 동기부여. 퀴즈 배지 시스템과 자연스럽게 연결 가능

### [IDEA-05] 알림 (클라우드 배포 후)
**상태**: 🔴 **보류** (전제 미충족 — 클라우드 배포 보류) — 2026-05-01

- 보류 사유: 사용자 본인 사용 전제로 클라우드 배포 우선순위 낮음
- 클라우드 배포 전: 의미 없음 (서버가 맥 로컬에만 있어서)
- 클라우드 후: 로컬 알림(Background App Refresh) 또는 APNs 푸시 알림
- 트리거: 새 기사 도착, 복습 스케줄 등

## 우선순위 판단 기준

- **앱 정체성 부합**: "내 관점 + 트렌드 레이더"에 맞는가
- **서버 부하**: 현재 로컬 서버 기준 감당 가능한가
- **클라우드 의존**: 클라우드 배포 전/후 중 언제 가능한가

### [IDEA-06] 검색엔진 활용 전략 변경
**상태**: 🟢 **MVP15 흡수** (M1, M2) — 2026-05-01 확정

현재: 폴백 체인(Tavily → Exa → ...) — 첫 성공만 사용, 태그당 5개
변경 방향: 병합/병행으로 양·다양성 확보, 검색 쿼리 다양화
- M1: Tavily 실패 원인 점검, 엔진별 결과 비교 실측
- M2: 폴백 → 병합 전환, 쿼리 다양화 (시간대/관점 변형), 양 5→20급
- 사용자 의도: 기사 ephemeral 유지 (DB 영속 저장 안 함)

### [IDEA-07] 모델 효율 정책 — 작업 종류별 모델 차별화
**상태**: 🟡 **다음 사이클 진행** (MVP15 완료 후 인프라 작업으로) — 2026-05-02 발견

#### 배경

워크플로우 진행 중 작업 종류에 따라 필요한 모델이 다름:
- **사고 집약** (critical-review, milestone-review, milestone, debate, deep-analysis, step-2/5/7) → 강한 모델 (Opus)
- **단순 작업** (step-6 구현, step-8 테스트, step-9 커밋, init, status) → 가벼운 모델 (Haiku/Sonnet)

현재는 세션 단위 단일 모델로 진행 → 토큰 비효율.

#### 심층 리서치 결과 (2026-05-02)

공식 문서 기반 강제 메커니즘 신뢰성 등급:

| 기법 | 강제 효과 | 비고 |
|---|---|---|
| **Subagent `model` 필드** | ★★★ 결정적 | `.claude/agents/*.md` frontmatter, 우선순위 최상 (env 변수 다음) |
| **Skill `model` 필드** | ★★ 턴 단위 | 해당 스킬 호출 턴만 적용, 다음 프롬프트엔 세션 모델 복귀 |
| `/model` 명령 | ★★★ 세션 단위 | 수동 전환 |
| **Hook** | ✗ 불가능 | 도구 차단만, 모델 변경 권한 없음 |
| CLAUDE.md 권고 | ✗ 자연어 | 강제력 0 |

#### 적용 옵션 (다음 사이클에서 결정)

1. **메인 세션 `--model opusplan` 시도** — Plan mode = Opus, Execution mode = Sonnet 자동 전환. 워크플로우 9단계와 자연 매칭
2. **사고 집약 스킬 9개에 `model: opus`/`effort: high` frontmatter 추가**:
   - critical-review, milestone, milestone-review, debate, deep-analysis
   - step-2 (룰즈 검증), step-5 (서브태스크 리뷰), step-7 (리팩토링+코드리뷰)
3. **단순 스킬에 `model: haiku`/`effort: low` 추가**:
   - step-6, step-8, step-9, init, status
4. **사고 집약 작업을 별도 subagent로 분리** (가장 강한 강제):
   - `.claude/agents/heavy-reviewer.md` 등에 `model: opus`
5. **`CLAUDE_CODE_SUBAGENT_MODEL` 환경변수**로 일괄 통제 (옵션)

#### 진입 조건

- MVP15 (현재) 완료 후
- 단독 인프라 사이클로 진행 (워크플로우 9단계 적용 가능)

#### 추가 발견

- **`effort` 필드** (Skill frontmatter): low/medium/high/xhigh/max — 모델과 별개로 사고 깊이 조절
- **`advisor()` 도구**: 시스템 프롬프트에 "stronger reviewer model"이라 적혀있으나 docs.anthropic.com에 공식 문서 없음. 내부 도구 가능성 — 강제 메커니즘으로 의존 X
