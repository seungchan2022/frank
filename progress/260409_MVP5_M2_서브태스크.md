## 확정 사항 (인터뷰 결과)

| 항목 | 결정 |
|------|------|
| SSRF 방어 | `summarize_service.rs` 내부 + `url_jail` 크레이트 |
| 타임아웃 | `AppError::Timeout(String)` 신규 추가 → HTTP 504 |
| 웹 라우트 | 신규 `/feed/article?url=...` + pushState state 전달 |
| FavoritesPort (M2) | `update_favorite_summary(user_id, url, summary, insight)` 단일 메서드만 (find 불필요) |
| Favorite 모델 | full struct — M3 DB 스키마 완전 반영 (id, user_id, title, url, snippet, source, published_at, tag_id, summary, insight, liked_at, created_at) |
| summarize 함수 시그니처 | free function: `summarize(url, title, user_id, crawl, llm, favorites)` |
| 타임아웃 범위 | crawl + LLM 전체를 `tokio::time::timeout(30s)` 로 감싸기 |
| favorites DB 업데이트 실패 시 | `warn!` 로그 + 200 정상 응답 반환 (사용자 경험 무영향, 개발자 모니터링) |
| SupabaseFavoritesAdapter | `PATCH /rest/v1/favorites?user_id=eq.{id}&url=eq.{url}` |

### ST3 — summarize_service.rs (TDD)
- **목적**: URL 크롤 + LLM 요약 오케스트레이션, SSRF 방어, 30초 타임아웃
- **파일**:
  - `server/src/services/summary_service.rs` — TDD (테스트 먼저)
- **산출물**: `summarize(url, title, user_id, crawl, llm, favorites)` free function
  - url_jail로 SSRF 차단 (private IP, loopback, cloud metadata, DNS rebinding)
  - `tokio::time::timeout(30초)` 로 crawl + LLM 전체 감싸기 → AppError::Timeout
  - `CrawlPort::scrape` 호출
  - `LlmPort::summarize` 호출
  - `FavoritesPort::update_favorite_summary` 무조건 호출 (favorites 없으면 0행 업데이트, 에러 없음)
  - update 실패 시: `warn!` 로그 + 요약 결과 정상 반환 (사용자에게 영향 없음)
- **테스트 케이스**:
  - 정상 요약 흐름 (favorites 업데이트 호출 확인)
  - SSRF 차단 (127.0.0.1, 10.0.0.1, 169.254.169.254 등)
  - 크롤 실패 → AppError::Internal
  - LLM 실패 → AppError::Internal
  - 30초 타임아웃 → AppError::Timeout
  - favorites DB 업데이트 실패 → 200 정상 반환 (warn 로그)
- **완료 기준**: 단위 테스트 모두 통과
- **예상 시간**: ~90분

### Codex 리뷰 (기술적 타당성)

- **[1] postgres_favorites.rs vs Supabase REST**: 문제 있음. sqlx PgPool 직접 UPDATE — 설계 확정안과 다름. 운영 시 DB 연결 관리 방식 차이로 배포/권한 리스크.
- **[2] summary_service 타임아웃 테스트 누락**: 문제 있음. 구현은 있으나 테스트 없어 회귀 보호 없음.
- **[3] summarize 핸들러 504 테스트 누락**: 문제 있음. 경로는 존재하지만 검증 없음.
- **[4] 웹 `+page.ts` published_at null fallback**: 경미한 문제. 새로고침 시 snippet/published_at 유실 — 기능에 치명적이지 않으나 비일관성.
- **[5] summaryCache 모듈 singleton 테스트 누출**: 경미한 문제. 현재 `vi.resetModules()`로 방지 중이나 다른 테스트와 공유 시 재발 가능.
- **[6] iOS 타임아웃 처리 책임 분리**: 경미한 문제. 어댑터가 HTTP 504 → `APISummarizeError.timeout` 변환, ViewModel이 메시지 변환 — 명확하나 `URLError.timedOut`(네트워크 타임아웃)은 별도 분기 없음.
- **[7] iOS `SummarySessionCache.shared` 격리**: 경미한 문제. 단위 테스트는 새 인스턴스 주입으로 안전하나 구조 자체는 격리 비친화적.

**전체 기술적 위험 수준**: 중간
