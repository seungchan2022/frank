## 목표

수집 시 크롤링/요약을 제거하고 메타데이터만 저장.  
피드는 제목 + source + 날짜만 표시. 백그라운드 주기 수집 추가.

## 의존성 DAG

```
S1(DB 마이그레이션)
  └→ S2(기존 데이터 삭제)
       └→ S3(모델 + collect_service 수정)
            └→ S4(백그라운드 스케줄러)
                 └→ S5(API 응답 경량화)
                      └→ S6(서버 테스트)
                           └→ [병렬 실행]
                                ├→ I1(FeedView UI) → I2(Pull-to-refresh) → I3(포그라운드 갱신) → I4(iOS 테스트)
                                └→ W1(/feed UI)    → W2(새로고침 버튼)                          → W3(웹 테스트)
```

**실행 전략**:
- S1~S6: 서버 레이어 순차 실행
- S6 완료 후: iOS(I1→I2→I3→I4) + 웹(W1→W2→W3) 병렬 에이전트로 실행

## DB 스키마 (확정)

### articles (재설계)

```sql
articles (
  id              uuid PK,
  user_id         uuid FK → profiles,
  tag_id          uuid FK → tags (nullable),
  title           text,
  url             text,
  snippet         text (nullable),
  source          text,
  published_at    timestamptz (nullable),
  created_at      timestamptz default now()
)
```

제거: `content`, `title_ko`, `llm_model`, `prompt_tokens`, `completion_tokens`, `summarized_at`, `search_query`

### favorites (신규)

```sql
favorites (
  id              uuid PK default gen_random_uuid(),
  user_id         uuid FK → profiles,
  article_id      uuid FK → articles,
  summary         text (nullable),
  insight         text (nullable),
  liked_at        timestamptz (nullable),
  created_at      timestamptz default now(),
  UNIQUE (user_id, article_id)
)
```

## Step 5 리뷰 결과

### Claude 리뷰 (문서 일치성)

**검토 항목**

1. **S3 범위 과소평가 위험** — Article 필드 제거는 단순 구조체 수정이 아니다. `summary_service`, `notification_service`, `DbPort.update_article_summary`, `DbPort.update_article_content`, `PostgresDb.ARTICLE_COLUMNS`, `FakeDb.update_article_*`, 테스트 fixture 전체가 연쇄 변경된다. 현재 서브태스크 명세에 이 영향 범위가 명시되지 않아 구현 시 범위 초과 가능성이 높다.

2. **summary/insight 귀속 모호성** — 설계상 summary/insight는 favorites 테이블로 이동한다. 그런데 `Article` 모델에서 제거하면 `summary_service.rs`가 즉시 컴파일 오류 난다. M1에서 `/me/summarize` 엔드포인트와 `summary_service`를 어떻게 처리할지 미결 상태다. **결정 필요**: M1 범위에서 요약 기능을 일시 비활성화(라우터에서 제거)할지, 기존 로직을 유지하되 favorites 기반으로 M2에서 옮길지.

3. **S4 스케줄러 대상 사용자 미명세** — 백그라운드 스케줄러가 "어떤 사용자"에 대해 수집하는지 문서에 없다. 전체 활성 사용자 순회가 필요하면 `DbPort`에 `list_active_users()` 같은 메서드가 추가로 필요하다. 이 사실이 S4 서브태스크에 반영되어 있지 않다.

4. **collect_for_user 시그니처 일관성** — 문서는 "CrawlPort trait 유지, collect_service에서 호출 제거"로 명시한다. 그런데 현재 `collect_for_user(db, search_chain, crawl, user_id)` 시그니처에서 `crawl` 인자가 남아 있으면 AppState/handler도 계속 crawl을 넘기게 된다. "호출 제거"와 "인자 제거" 여부를 S3 명세에 명시해야 한다.

5. **S5 DTO 분리 누락** — 현재 `ArticleResponse`가 목록(`/me/articles`)과 단건(`/me/articles/:id`) 양쪽에서 쓰인다. M2에서 상세 화면에 더 많은 정보가 필요할 때 다시 깨진다. S5에서 `ArticleListItem` DTO 분리를 검토할 필요가 있다.

6. **iOS FeedFeature 상태머신 미반영** — I2 Pull-to-refresh가 "POST /me/collect 후 목록 갱신"이지만, 현재 `FeedFeature.swift`는 `collectAndSummarize` 플로우를 핵심으로 들고 있다. 상태머신 변경 범위가 I1~I2보다 훨씬 크다.

**문서 일치성 판단**: 전반적 방향은 일치하나, S3 영향 범위, S4 대상 사용자 명세, `/me/summarize` 처리 결정이 문서에 빠져 있다.

### Codex 리뷰 (기술적 타당성)

**핵심 이슈**

1. **S3 연쇄 파괴**: `summary_service`, `notification_service`, `DbPort.update_article_summary/update_article_content`, `PostgresDb.ARTICLE_COLUMNS/ArticleRow`, `FakeDb` 구현이 모두 동시에 수정 대상이 된다. 서브태스크 명세가 `models.rs + collect_service`로 한정되어 있어 범위가 실제보다 좁게 표현되어 있다.

2. **S4 overlap 방지 한계**: `MissedTickBehavior::Skip` 단독으로는 "이전 collect가 실행 중인데 다음 tick 도달"을 막지 못한다. `AtomicBool` 또는 `Semaphore(1).try_acquire_owned()`를 추가 가드로 사용해야 한다.

3. **S5 클라이언트 계약 동기화**: 서버 DTO 축소는 iOS/웹 DTO 수정과 반드시 동기화해야 한다. 현재 `APIArticleAdapter.swift`와 `article.ts`가 `summary/insight/title_ko/search_query/summarized_at`를 명시적으로 파싱 중이다.

4. **collect_for_user crawl 인자**: trait은 유지하되 함수 인자에서도 제거해야 AppState에서 crawl 전달 경로가 정리된다.

5. **목록/단건 DTO 분리**: `ArticleResponse` 하나로 목록과 단건을 공유하면 M2에서 다시 설계 충돌이 난다. 지금 분리하는 것이 낫다.

**서브태스크별 기술 판단**

| # | 판단 | 주요 이슈 |
|---|------|-----------|
| S1 | 승인 | favorites FK 의존성 확인 후 실행, ARTICLE_COLUMNS와 동기화 필수 |
| S2 | 승인 | 마이그레이션과 분리된 운영 스크립트로 관리 권장 |
| S3 | 조건부 승인 | summary_service/notification_service/DbPort 처리 결정을 명세에 추가해야 함 |
| S4 | 조건부 승인 | 대상 사용자 명세 + AtomicBool overlap 가드 패턴 명시 필요 |
| S5 | 조건부 승인 | 목록/단건 DTO 분리 + 클라이언트 동기화 계획 추가 필요 |
| S6 | 승인 | summary 관련 테스트 제거 + M1 기준 계약 테스트 추가 방향으로 수정 |
| I1~I4 | 조건부 승인 | FeedFeature 상태머신(collectAndSummarize) 정리가 선행되어야 함 |
| W1~W3 | 조건부 승인 | summarize 타이머/API 제거 후 fixture/mock부터 수정 |

### 리뷰 반영 — 서브태스크 명세 보강

**S3 명세 추가**
- `summary_service.rs`, `notification_service.rs` M1에서 처리 방향: `/me/summarize` 라우터 제거, summary_service 비활성화 (M2에서 favorites 기반으로 재설계)
- `DbPort.update_article_summary` + `update_article_content` 제거
- `collect_for_user` 시그니처에서 `crawl: &dyn CrawlPort` 인자 제거
- `PostgresDb.ARTICLE_COLUMNS`, `ArticleRow` 필드 동시 수정

**S4 명세 추가**
- 스케줄러 수집 대상: `DbPort`에 `list_user_ids_with_tags()` 추가 (또는 기존 활성 유저 순회 방식 명시)
- overlap 방지: `MissedTickBehavior::Skip + AtomicBool` 조합

**S5 명세 추가**
- `ArticleListItem` DTO 신규 정의 (id, title, snippet, source, published_at, tag_id)
- `/me/articles/:id`는 별도 `ArticleDetailResponse` DTO로 분리 (현재는 동일 구조체 유지, M2에서 확장)

---

### 최종 결정: **조건부 승인**

아래 3가지 결정 사항을 확정하면 S6 착수 가능하다.

1. **M1에서 `/me/summarize` 처리**: 라우터에서 제거 + `summary_service` 비활성화 (M2에서 favorites 기반 재설계)
2. **S4 스케줄러 대상**: `DbPort`에 `list_user_ids_with_tags()` 추가하여 활성 태그가 있는 전체 사용자 순회
3. **S5 DTO 전략**: `ArticleListItem` DTO 분리 (목록용) + `/me/articles/:id`는 M2까지 임시로 동일 구조 유지

위 3가지 확정 후 S1부터 순차 착수.
