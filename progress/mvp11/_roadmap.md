## 마일스톤 구성

```
M1 (서버) ──────────────────────────────────────────▶ done
           │
           └─ M2 (iOS 초기화·환경설정) ──────────────▶ done
           └─ M3 (웹 즐겨찾기·오답 태그 필터) ────────▶ done   ← 병렬
```

## 기술 부채 / 후속 과제

### BUG-004 후속: 뉴스 품질 개선 근본 해결

**현재 방어 구조의 한계**

M1에서 적용한 3층 방어(Exa `category:"news"` / Tavily `topic:"news"` / URL 패턴 필터)는 부분적 개선이며, 근본 해결이 아니다. 각 웹사이트의 URL 구조가 달라 URL 패턴 필터는 두더지 잡기 방식이 된다. `published_at` 유무 기반 필터도 Exa 전용 커스텀이 되어 다른 엔진(Tavily, Firecrawl)의 정상 기사를 오탐 제거하는 문제가 있다.

**권장 후속 방법 (둘 중 하나)**

| 방법 | 설명 | 비용 | 엔진 독립 |
|------|------|------|----------|
| 뉴스 전용 API 교체 | NewsAPI / Bing News Search 등 — 기사만 반환하도록 API 계약 수준에서 보장 | 추가 API 비용 | ✅ |
| LLM 분류 후처리 | 검색 결과 스니펫·제목을 LLM이 "기사 vs 목록 페이지" 분류 | 요청당 LLM 비용 + 지연 | ✅ |
