# MVP5 로드맵 — 학습 기능 (기사 수집 · 요약 · 즐겨찾기)

> 작성일: 2026-04-09  
> 브랜치 기반: `main` (MVP4 완료 이후)

---

## 목표

현재 "모두 수집 → 모두 저장 → 모두 요약" 구조의 비용·UX·노이즈 문제를 해소하고,  
**"보고 싶은 것만 요약, 저장하고 싶은 것만 즐겨찾기"** 흐름으로 전환한다.

---

## MVP5 스코프

| 마일스톤 | 내용 | 플랫폼 |
|---------|------|--------|
| M1 | 피드 아키텍처 전환 | 서버 + iOS + 웹 |
| M2 | 디테일 페이지 + 온디맨드 요약 | 서버 + iOS + 웹 |
| M3 | 즐겨찾기 + 스크랩 목록 | 서버 + iOS + 웹 |

**MVP6으로 이동**: 유저 역할 프로필, 인사이트 개인화, 퀴즈, 연관 기사, 좋아요

## 인터뷰 결정 사항 (구현 기준)

| 항목 | 결정 |
|------|------|
| 기존 DB 기사 데이터 | **전부 삭제** (새 구조로 시작) |
| 피드 표시 | **제목만** (이후: 제목 + 썸네일 이미지) |
| 수집 방식 | **백그라운드 주기적 수집 + 수동 pull-to-refresh** |
| 디테일 원문 | **리드 3~5문장 + 앱 내 SafariView** (SFSafariViewController / WebView) |
| 요약 캐시 | **세션 캐시** (클라이언트 메모리, 앱 종료 시 소멸) |
| 즐겨찾기 안 하면 | 요약/인사이트 **저장 안 됨** |
| 즐겨찾기 하면 | **현재 상태 그대로 저장** (요약 전이면 메타만, 요약 후면 요약+인사이트 포함) |
| 즐겨찾기 후 요약 | **DB 업데이트** (summary, insight 컬럼 채움) |
| 즐겨찾기 버튼 | **하나만** (MVP6에서 좋아요 분리 예정) |
| 즐겨찾기 목록 접근 | **하단 탭바 별도 탭** |
| 즐겨찾기 목록 정렬 | **최신순** (이후 필터 추가) |
| 수집 언어 | **글로벌** (언어 구분 없음) |
| 요약/인사이트 출력 언어 | **한국어 고정** |
| 플랫폼 구현 순서 | **서버 먼저 → iOS + 웹 동시** |

## DB 스키마 재설계

### articles 테이블 (재설계)

기존에서 **제거**:
- `content` — 수집 시 크롤링 안 함
- `title_ko` — 온디맨드에서 처리
- `llm_model`, `prompt_tokens`, `completion_tokens`, `summarized_at` — favorites로 이동

**유지 + 추가**:

```sql
articles (
  id              uuid PK,
  user_id         uuid FK → profiles,
  tag_id          uuid FK → tags (nullable),
  title           text,
  url             text,
  snippet         text (nullable),   -- 원문 리드 (수집 시 저장, 3~5문장)
  source          text,
  published_at    timestamptz (nullable),
  created_at      timestamptz default now()
)
```

> `search_query` 제거 — 피드 표시에 불필요  
> `snippet`은 기존 컬럼 재활용 (검색 API가 리드를 이미 제공)

### favorites 테이블 (신규)

```sql
favorites (
  id              uuid PK default gen_random_uuid(),
  user_id         uuid FK → profiles,
  article_id      uuid FK → articles,
  summary         text (nullable),     -- 즐겨찾기 후 요약 시 채워짐
  insight         text (nullable),     -- 즐겨찾기 후 인사이트 시 채워짐
  liked_at        timestamptz (nullable),  -- MVP6 좋아요 기능 대비 (지금은 사용 안 함)
  created_at      timestamptz default now(),
  UNIQUE (user_id, article_id)
)
```

## M1: 피드 아키텍처 전환

### 목표
수집 시 DB 저장을 메타데이터만으로 경량화. 크롤링/요약 선행 제거.

### 서버

**마이그레이션**:
- articles 테이블 재설계 (위 스키마 적용)
- favorites 테이블 신규 생성
- 기존 articles 데이터 전부 삭제

**collect_service.rs 수정**:
- 크롤링(`CrawlPort` 호출) 제거 — 수집 시 본문 추출 안 함
- 저장 항목: id, user_id, tag_id, title, url, snippet, source, published_at
- 중복 URL 처리 유지 (동일 URL은 1회만 저장)

**백그라운드 스케줄러 추가**:
- 서버 시작 시 별도 태스크로 주기적 수집 실행 (간격: 환경변수 `COLLECT_INTERVAL_SECS`)
- 기존 `POST /me/collect` 엔드포인트도 유지 (수동 트리거)

**GET /me/articles 응답 경량화**:
```json
[
  {
    "id": "...",
    "title": "...",
    "snippet": "...",
    "source": "...",
    "published_at": "...",
    "tag_id": "..."
  }
]
```

### iOS

- `FeedView`: 기사 카드 → 제목 + source + 날짜만 표시
- Pull-to-refresh: `POST /me/collect` 호출 후 목록 갱신
- 백그라운드 수집 결과는 앱 포그라운드 진입 시 자동 갱신

### 웹

- `/feed`: 기사 카드 → 제목 + source + 날짜만 표시
- Pull-to-refresh 또는 새로고침 버튼
- 백그라운드 수집 결과는 페이지 진입 시 자동 갱신

### 작업 단위

**서버** (순서대로)
1. DB 마이그레이션: articles 테이블 재설계 + favorites 테이블 생성
2. 기존 articles 데이터 삭제 스크립트
3. `collect_service.rs`: 크롤링 호출 제거, 메타데이터만 저장
4. 백그라운드 스케줄러 추가 (`tokio::time::interval`, 주기: `COLLECT_INTERVAL_SECS` 환경변수)
5. `GET /me/articles` 응답 경량화 (title, url, snippet, source, published_at, tag_id)
6. 서버 테스트 업데이트

**iOS + 웹 (서버 완료 후 병렬)**

iOS:
1. `FeedView` 카드 UI: 제목 + source + 날짜만 표시 (요약 제거)
2. Pull-to-refresh: `POST /me/collect` 호출 후 목록 갱신
3. 포그라운드 진입 시 피드 자동 갱신
4. iOS 테스트 업데이트

웹:
1. `/feed` 카드 UI: 제목 + source + 날짜만 표시 (요약 제거)
2. 새로고침 버튼 or pull-to-refresh
3. 웹 테스트 업데이트

- [ ] favorites 테이블 생성 완료
- [ ] 기존 기사 데이터 삭제 완료

- [ ] 백그라운드 스케줄러 동작 확인

## M2: 디테일 페이지 + 온디맨드 요약

### 목표
기사 클릭 시 디테일 페이지 진입. 요약/인사이트는 버튼 클릭 시 온디맨드 LLM 호출.

### 서버

**신규 엔드포인트**:

```
GET  /me/articles/:id          → 기사 상세 (snippet 포함)
POST /me/articles/:id/summarize → 온디맨드 요약 + 인사이트 생성
```

**POST /me/articles/:id/summarize 동작**:
1. 기사 URL 크롤링 (Firecrawl) → 본문 추출
2. LLM 호출: 요약 + 인사이트 한국어로 생성
3. 결과 반환 (DB 저장 안 함 — 클라이언트 세션 캐시)
4. 해당 article_id가 favorites에 있으면 → summary, insight 컬럼 업데이트

**응답**:
```json
{
  "summary": "...",
  "insight": "..."
}
```

**기존 POST /me/summarize 제거** (배치 요약 엔드포인트 삭제)

**타임아웃**: 30초 (MVP4 패턴 그대로)

### iOS

**ArticleDetailView 구성**:
```
[ 제목 ]
[ source | published_at ]
[ 원문 리드 (snippet, 3~5문장) ]
[ "원문 보기" 버튼 ] → SFSafariViewController로 열기
─────────────────────────────
[ 요약하기 버튼 ]
  → 로딩 중: 30초 타임아웃 + 재시도 버튼 (MVP4 패턴 재사용)
  → 완료 시:
    [ 요약 섹션 ]
    [ 인사이트 섹션 ]
─────────────────────────────
[ 즐겨찾기 버튼 ] (M3에서 구현, 여기서 레이아웃만 예약)
```

**세션 캐시**:
- `ArticleDetailFeature`에서 `[article_id: SummaryResult]` 딕셔너리로 관리
- 같은 기사 재진입 시 LLM 재호출 없이 캐시 반환
- 앱 종료 시 자동 소멸

### 웹

**`/feed/[id]` 페이지 구성**:
- 제목, source, 날짜
- snippet (원문 리드)
- "원문 보기" 링크 → 새 탭 또는 인앱 WebView
- "요약하기" 버튼 → 로딩 UX (spinner + 30초 타임아웃)
- 요약/인사이트 결과 표시 섹션

**세션 캐시**:
- Svelte store에 `Map<articleId, SummaryResult>` 보관
- 페이지 새로고침 시 소멸

### 작업 단위

**서버** (순서대로)
1. `GET /me/articles/:id` 엔드포인트 (snippet 포함 단건 조회)
2. `POST /me/articles/:id/summarize` 엔드포인트
   - Firecrawl로 URL 크롤링 (본문 추출)
   - LLM 호출: 요약 + 인사이트 한국어 생성
   - 해당 article이 favorites에 있으면 summary/insight 컬럼 업데이트
   - DB 저장 없이 결과만 반환 (즐겨찾기 없으면)
3. `POST /me/summarize` 배치 엔드포인트 제거
4. LLM 프롬프트 한국어 출력 확인
5. 서버 테스트 업데이트

**iOS + 웹 (서버 완료 후 병렬)**

iOS:
1. `ArticleDetailView` 전면 수정
   - snippet 표시 (리드 3~5문장)
   - "원문 보기" 버튼 → `SFSafariViewController`
   - "요약하기" 버튼 + 30초 타임아웃 + 재시도 (MVP4 M2 패턴 재사용)
   - 요약/인사이트 결과 섹션
   - 세션 캐시 (`ArticleDetailFeature`에 `[UUID: SummaryResult]` 딕셔너리)
2. iOS 테스트 업데이트

웹:
1. `/feed/[id]` 페이지 수정
   - snippet 표시
   - "원문 보기" 링크 → 새 탭
   - "요약하기" 버튼 + 30초 타임아웃 + 재시도 (MVP4 M3 패턴 재사용)
   - 요약/인사이트 결과 섹션
   - 세션 캐시 (Svelte store `Map<string, SummaryResult>`)
2. 웹 테스트 업데이트

## M3: 즐겨찾기 + 스크랩 목록

### 목표
즐겨찾기 시 현재 상태(메타 + 요약/인사이트 nullable)를 DB에 저장.  
스크랩 목록은 하단 탭바 별도 탭으로 접근.

### 서버

**신규 엔드포인트**:

```
POST   /me/favorites/:article_id  → 즐겨찾기 추가 (현재 요약/인사이트 포함)
DELETE /me/favorites/:article_id  → 즐겨찾기 해제
GET    /me/favorites              → 즐겨찾기 목록 (최신순)
```

**POST /me/favorites/:article_id 요청 바디**:
```json
{
  "summary": "...",   // nullable (요약 안 했으면 null)
  "insight": "..."    // nullable (인사이트 없으면 null)
}
```

**GET /me/favorites 응답**:
```json
[
  {
    "id": "...",           // favorites.id
    "article_id": "...",
    "title": "...",
    "url": "...",
    "source": "...",
    "published_at": "...",
    "summary": "...",      // nullable
    "insight": "...",      // nullable
    "created_at": "..."    // 즐겨찾기한 시각, 최신순 정렬 기준
  }
]
```

### iOS

**즐겨찾기 버튼** (ArticleDetailView):
- 하트/북마크 아이콘
- 누르면: 현재 세션의 summary/insight(있으면 포함) + article 메타데이터 → POST /me/favorites
- 이미 즐겨찾기된 상태면: 채워진 아이콘 표시 + 누르면 DELETE

**즐겨찾기 후 요약**:
- favorites에 이미 등록된 기사에서 요약하기 → POST /me/articles/:id/summarize
- 서버가 favorites 테이블 자동 업데이트

**스크랩 목록 탭**:
- 하단 탭바에 북마크 아이콘 탭 추가
- 즐겨찾기 목록 (제목 + source + 즐겨찾기 날짜, 최신순)
- 탭하면 ArticleDetailView로 진입 (요약 있으면 바로 표시)

### 웹

**즐겨찾기 버튼** (/feed/[id] 페이지):
- 하트/북마크 아이콘
- 동일 동작

**스크랩 목록 탭**:
- 네비게이션에 즐겨찾기 탭 추가
- `/favorites` 라우트
- 목록 (제목 + source + 즐겨찾기 날짜, 최신순)
- 클릭하면 /feed/[id]로 이동

### 작업 단위

**서버** (순서대로)
1. `POST /me/favorites/:article_id` — 즐겨찾기 추가 (summary/insight nullable 바디)
2. `DELETE /me/favorites/:article_id` — 즐겨찾기 해제
3. `GET /me/favorites` — 목록 조회 (최신순, article 메타 + summary/insight 포함)
4. 서버 테스트 업데이트

**iOS + 웹 (서버 완료 후 병렬)**

iOS:
1. `ArticleDetailView`에 즐겨찾기 버튼 추가 (하트/북마크 아이콘, 토글)
   - 누를 때: 현재 세션 캐시의 summary/insight 포함해서 `POST /me/favorites`
   - 이미 즐겨찾기 상태면: 채워진 아이콘 + 누르면 `DELETE`
2. 하단 탭바에 즐겨찾기 탭 추가
3. `FavoritesView` 신규 구현 (목록, 최신순, 탭하면 ArticleDetailView 진입)
4. iOS 테스트 업데이트

웹:
1. `/feed/[id]`에 즐겨찾기 버튼 추가 (토글)
2. `/favorites` 라우트 신규 구현 (목록, 최신순, 클릭하면 /feed/[id])
3. 네비게이션에 즐겨찾기 탭 추가
4. 웹 테스트 업데이트
