## 문제 요약

피드에 개별 뉴스 기사 대신 뉴스 카테고리·태그 **인덱스 페이지**가 노출된다.

- 원인: Tavily·Exa 모두 뉴스 전용 파라미터 없이 일반 웹 검색 모드로 동작
- 증상 예: `"Data science recent news | AI Business"`, `"data science News & Articles - IEEE Spectrum"` 같은 listing 페이지가 피드에 포함됨
- 폴백 체인: Tavily → Exa → Firecrawl (Firecrawl은 뉴스/날짜 필터 파라미터 자체 없음)

## ST-1: Exa `type:"news"` + `startPublishedDate` 추가

### 현재 상태 (`exa.rs:109-118`)

```rust
let body = serde_json::json!({
    "query": query,
    "numResults": max_results,
    "contents": {
        "highlights": { "numSentences": 3, "highlightsPerUrl": 1 }
    }
});
```

### 목표 상태

```rust
let now = chrono::Utc::now();
let week_ago = now - chrono::Duration::days(7);
let body = serde_json::json!({
    "query": query,
    "numResults": max_results,
    "type": "news",
    "startPublishedDate": week_ago.to_rfc3339(),
    "contents": {
        "highlights": { "numSentences": 3, "highlightsPerUrl": 1 }
    }
});
```

### 수정 대상 파일

- `server/src/infra/exa.rs` — `SearchPort::search()` body 수정
- `server/Cargo.toml` — `chrono` 의존성 확인 (이미 있을 수 있음)

### 추가할 테스트

- `exa_request_includes_type_news` — `body_partial_json({"type": "news"})` matcher로 파라미터 포함 검증
- `exa_request_includes_start_published_date` — `startPublishedDate` 키 존재 + ISO8601 형식 검증

### 성공 기준

- `body_partial_json` matcher 테스트 통과
- Exa 응답의 `publishedDate` 필드가 `published_at`에 매핑되어 반환 (기존 동작 유지)
- `cargo test` 전체 통과

### 리스크

- `type:"news"` 적용 시 결과 수 감소 가능 → `numResults` 증가로 대응
- chrono가 이미 Cargo.toml에 있는지 확인 필요

### 수정 대상 파일

- `server/src/infra/tavily.rs` — `SearchPort::search()` body 수정

### 추가할 테스트

- `tavily_request_includes_topic_news` — `body_partial_json({"topic": "news"})` matcher로 파라미터 포함 검증

### 주의 사항

- Tavily `topic:"news"` + `time_range:"week"` 조합 동작 여부: 현재 플랜 rate limit(432) 상태이므로 Mock 기반 단위 테스트로만 검증하고, 실제 API 응답은 rate limit 리셋 후 수동 확인
- `topic:"news"` + `time_range:"week"` 조합이 Tavily 공식에서 미지원이면 `time_range` 제거 후 `topic`만 적용하는 안으로 전환

### 성공 기준

- `body_partial_json` matcher 테스트 통과
- `cargo test` 전체 통과
- rate limit 리셋 후 실제 피드에서 listing 페이지 감소 수동 확인

## ST-3: URL 패턴 후처리 필터 추가 (listing 페이지 제거)

### 목적

ST-1·ST-2 파라미터 수정만으로 완전히 차단하지 못하는 listing 페이지를 URL 패턴 기반으로 제거한다. Firecrawl (3순위 폴백)은 뉴스 파라미터 자체가 없으므로 이 필터가 유일한 방어선이 된다.

### 필터링 대상 URL 패턴

```
/tag/, /tags/
/category/, /categories/
/topics/, /topic/
/section/
```

> 주의: `/news/`, `/articles/` 는 포함하지 않는다. 이 패턴은 정상 기사 URL (예: `/news/some-article`)을 오탐(false positive)할 수 있다.

### 위치 선택 — 미결 (ST-3 인터뷰에서 확정)

두 가지 후보가 있으며 각각 트레이드오프가 있다:

| 후보 | 파일 | 장점 | 단점 |
|------|------|------|------|
| **A안** | `feed.rs` (`is_homepage_url()` 옆에 추가) | 기존 URL 필터와 응집, 코드 위치 일관성 | feed 엔드포인트에서만 동작 (다른 소비자 없음 — 현재는 동등) |
| **B안** | `search_chain.rs` (폴백 체인 내 후처리) | 체인 소비자 모두에 자동 적용 | 검색 오케스트레이션에 필터 로직 혼입 |

현재 `SearchFallbackChain`의 소비자는 `feed.rs` 하나뿐이므로 기능 차이 없음. 설계 원칙(응집도 vs 단일 책임) 기준으로 ST-3 인터뷰에서 확정한다.

### 추가할 함수 (위치 미결)

```rust
/// listing URL 패턴 (태그·카테고리·섹션 인덱스 페이지) 판별
fn is_listing_url(url: &str) -> bool {
    let lower = url.to_lowercase();
    let listing_segments = [
        "/tag/", "/tags/",
        "/category/", "/categories/",
        "/topics/", "/topic/",
        "/section/",
    ];
    listing_segments.iter().any(|seg| lower.contains(seg))
}
```

### 추가할 테스트

- `listing_url_tag_path_filtered` — `/tag/ai` URL → 필터링 확인
- `listing_url_category_path_filtered` — `/category/tech` → 필터링 확인
- `listing_url_real_article_not_filtered` — `/news/some-article` → 통과 확인
- feed 통합 테스트: listing URL이 응답에서 0건 확인

### 성공 기준

- 단위 테스트 통과
- 수동 피드 3회 새로고침: `/tag/`, `/category/` 패턴 URL 0건
- `cargo test` 전체 통과
