iOS 퀴즈 — 어떤 기사를 열어도 Swift 문제만 나왔다
시뮬레이터에서 직접 앱을 써보다가 발견. ArticleDetailView의 quiz 파라미터 기본값이 MockQuizAdapter()였고, AppDependencies에 quiz 포트가 빠져 있어서 View 생성 시 항상 Mock이 주입되는 구조였다. Mock 픽스처가 전부 Swift 문제라 어떤 기사든 Swift 퀴즈가 노출됐다. AppDependencies 체인에 포트를 추가하고 기본값 Mock을 제거해 수정.
단위 테스트가 통과해도 실제 의존성 연결은 시뮬레이터 직접 사용에서만 잡힌다. 새 포트 추가 시 AppDependencies 체인 전체를 확인하는 습관이 필요하다.
스코프 결정
MVP8 M2/M3를 별도 마일스톤으로 분리한 이유
M2는 신규 API 호출 없이 기존 화면 수정만이라 M3와 코드 충돌 없이 독립 진행이 가능했다. 동시에 M3는 M1 엔드포인트를 소비하므로, M1 완료 후에야 시작할 수 있다. 의존성 방향이 자연스럽게 M1 → M2 → M3 순서를 만들었고, M2를 먼저 끝내 UI 기반(확인 버튼, 타이포)을 안정화한 뒤 M3 오답 노트를 올렸다.
마일스톤 분리 기준은 "작업량"이 아니라 "API 의존성 방향"이다. M2가 API 독립적이라는 사실이 병렬/직렬 설계의 근거가 됐다.
기술 결정
snippet 정제 — 외부 크롤링 API 대신 코드 필터링
Jina Reader 같은 외부 API는 레이턴시·비용·의존성 리스크를 동시에 올린다. MVP 단계에서는 "완벽한 정제"보다 빠른 개선 후 Known Issues 문서화가 더 실용적이라 판단했다. clean_snippet()으로 HTML 태그·URL·이메일 제거 + 단어 경계 절단을 구현하고, 한국 뉴스 사이트 네비게이션 텍스트 혼입 문제는 Known Issues로 명시했다.
Known Issues 문서화는 "나중에 고치겠다"는 미루기가 아니라, 현재 결정의 한계를 명시하는 기술 부채 관리다. 다음 MVP에서 재검토 기준이 된다.
UX 결정
퀴즈 확인 버튼 — "즉시 표시"에서 "확인 후 표시"로
시뮬레이터에서 실제로 퀴즈를 풀다가 발견. 스크롤 중 실수로 보기를 탭하면 즉시 정오 결과가 나와서 당황스러웠다. 보기 선택 → 확인 버튼 클릭 → 정오 + 해설 표시 순서로 변경했고, 확인 전까지 보기를 다시 선택할 수 있다.
"즉각 피드백"이 항상 좋은 UX가 아니다. 모바일 터치는 실수가 많으므로, 선택과 확정 사이의 의도적 간격이 필요하다. 이런 불편함은 기획 단계가 아니라 직접 사용에서만 발견된다.
제거 결정
연관 기사 섹션 전체 제거
현재 구현은 키워드 기반 검색 결과를 "연관 기사"로 노출했다. 이는 원래 의도인 취향 기반 추천과 다르다 — 사용자 취향이 전혀 반영되지 않는 유사 검색 결과였다. 잘못된 방향의 기능이 계속 화면에 있으면 사용자 기대를 어긋나게 만들고, 나중에 올바른 기능을 붙일 때 혼란을 더한다고 판단해 과감히 제거했다.
"있는 것보다 없는 게 나은 기능"이 존재한다. 제거도 의사결정이다. 재설계는 다음 MVP에서 사용자 취향 데이터를 활용하는 방식으로 처음부터 설계할 것.
